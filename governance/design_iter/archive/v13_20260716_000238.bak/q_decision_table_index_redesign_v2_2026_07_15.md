## 触发

用户原话：
> 给 `@governance/design_iter/INDEX.md` 做定义，**是版本迭代落地进度记录版本交接的总览表，迭代进度当我主动停止，完整迭代或者切换迭代版本时，触发进度落地和状态总览**。

## 用户原话拆解（已 Read）

| 关键词 | 含义 |
|---|---|
| "**版本迭代落地进度记录**" | INDEX.md = 进度看板（每个版本当前推进到哪、还差多少） |
| "**版本交接**" | INDEX.md = 交接总览（v{N} → v{N+1} 关键交付 + 截止） |
| "**总览表**" | 不是详情——详情在 plans/v{N}/PLAN.md |
| "**迭代进度当我主动停止**" | 触发时机 1：主动停止 → 触发进度落地 |
| "**完整迭代**" | 触发时机 2：完整迭代完成 → 触发进度落地 |
| "**切换迭代版本**" | 触发时机 3：切版本 → 触发状态总览 |
| "**触发进度落地和状态总览**" | 触发的产物 = 进度记录 + 状态总览都写到 INDEX.md |

## 新 INDEX.md 结构（4 段）

```markdown
# AIDocxWorkFlow 方案迭代总览

> **定义**：版本迭代落地进度 + 版本交接的总览表。
> **触发**：主动停止 / 完整迭代完成 / 切换版本时，触发进度落地和状态总览。
> **详情**：见 `plans/v{N}/PLAN.md` 各自维护 3 栏（解决 / 新增 / 遗留）。

## 1. 当前生效 + 进度

| 版本 | 状态 | 进度 | 下一动作 | 接手 |
|---|---|---|---|---|
| v14 | current | 100%（D-V14-001~005 + P1~P3 全部确认） | 启动 v15 或转入 archive | （待交接） |

## 2. 进度看板（按版本历史）

| 版本 | 启动日 | 闭环日 | 关键交付 | 状态 |
|---|---|---|---|---|
| v14 | ... | ... | ... | current |
| v13 | ... | ... | ... | 已闭环 |
| ... | | | | |

## 3. 交接承诺（v{N} → v{N+1}）

| 来源版本 | 目标版本 | 关键交付项 | 截止 | 状态 |
|---|---|---|---|---|
| v9 | v10 | ... | 2026-07-... | 已接 |
| v14 | v15 | Q-V15-001（plans 重生成） | 启动 v15 时 | 待接 |

## 4. 触发与维护

| 触发 | 操作 | 写入位置 |
|---|---|---|
| **主动停止**（用户明说"停了"） | 把当前 current 进度归 §2 + 状态改"已停止" | §2 + §1 |
| **完整迭代**（v{N} 闭环） | §2 新增 v{N} 行 + §3 新增 v{N+1} 承诺 | §2 + §3 |
| **切版本**（执行 `rollback` 或 `new`） | §1 改 current + §2 加 v{N-1} 历史行 | §1 + §2 |
```

## 触发实现候选（请拍 1）

### 候选 A（推荐）— 新建 index_landing_hook.py，加 3 子命令

- 加 `.cursor/hooks/index_landing_hook.py`：
 - 命令 `stop-iter`（主动停止）
 - 命令 `complete-iter`（完整迭代）
 - 命令 `switch-iter <vN>`（切版本）
- 或挂 `sessionEnd` 时检测"是否触发三时机"
- 影响：1 新 hook + 与 `scripts/design_iter.py` 的 `status` 命令集成
- 关键：触发落地必须**幂等**（多次触发 = 同样结果）

### 候选 B（最简）— 纯文档规则 + CLI 3 子命令（不动 hook）

- `scripts/design_iter.py` 加 3 子命令：`stop_iter` / `complete_iter` / `switch_iter <vN>`
- 写规则到 `governance/design_iter/README.md` §"触发与维护"
- 不挂任何 IDE hook——靠人主动调
- 影响：1 脚本加 3 命令 + README 加 1 段
- 优点：不动 IDE 钩子
- 缺点：仍靠人手动——**但有 3 时机的明确规则比没规则好**

### 候选 C（彻底）— A + pre-commit / GitHub Action

- A 基础上加 pre-commit：`git commit` 时校验 INDEX.md 是否更新
- 优点：彻底防止脱钩
- 缺点：项目无 .github/、无 pre-commit 配置；改动面大

## 推荐

**A + INDEX.md 重写**——你刚说"没有维护的钩子，没用"，候选 B 也仍靠人；候选 A 直接把"主动停止/完整迭代/切版本"三个时机变成可执行命令。
