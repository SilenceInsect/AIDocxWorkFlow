## 1. 背景与动机

### 1.1 问题陈述

S3 原型导出阶段当前**只有一种模式**（默认深度版），无论需求规模大小、UI 复杂度高低，一律生成详细 Mermaid 页面流图 + 控件树 + 状态机。

**实际痛点**（来自 v14 实战反馈）：
- **小型需求**（UI ratio < 0.20）：深度版产出 80% 内容被 QA 跳过，浪费时间
- **中型需求**（UI ratio 0.20-0.50）：深度版正合适，无需调整
- **大型需求**（UI ratio > 0.50）：深度版**不够细**，需补充用户旅程图

### 1.2 期望目标

按 UI ratio + 风险等级**自动判定走哪条路径**——LLM/Agent 不再主观选择，避免决策漂移。

---

## 2. 三档路径定义

### 2.1 轻量版（lightweight）

**适用场景**：
- UI ratio < 0.30（UI 类 Story < 30%）
- 风险等级 P2（低风险）
- Story 数 ≤ 5

**产出**：
- 仅关键页面截图清单（≤ 10 页）
- 关键交互流程图（≤ 3 个）
- 无控件树细节
- 无状态机

**预计耗时**：深度版的 **20-30%**

### 2.2 标准版（standard，默认）

**适用场景**：
- UI ratio 0.30-0.50
- 风险等级 P1
- Story 数 6-15

**产出**：
- 完整页面清单 + 关键页面控件树
- Mermaid 页面流图
- 主要状态机

**预计耗时**：当前深度版的 **70-90%**

### 2.3 深度版（depth）

**适用场景**：
- UI ratio > 0.50
- 风险等级 P0（强制）
- Story 数 > 15
- 含跨系统联动（LINK 模块占比 > 20%）

**产出**：
- 完整页面 + 控件树 + 状态机
- 完整 Mermaid 页面流图 + 时序图
- 用户旅程图 + 异常决策树
- 跨系统联动关系图

**预计耗时**：标准版的 **150-200%**

## 3. 触发决策树（D-V15-002 待拍板）

### 3.1 推荐触发条件（按 P0 优先原则）

```
开始 → 风险等级判断
  ├── P0 → 强制深度版（无论 UI ratio）
  ├── P1 → UI ratio > 0.30？ → 是：标准版 / 否：轻量版
  └── P2 → UI ratio > 0.50？ → 是：标准版 / 否：轻量版
```

### 3.2 强制规则边界

| 规则 | 触发条件 | 强制路径 | 备注 |
|---|---|---|---|
| **强制深度版** | 任一 P0 Story | 深度版 | 即便 UI ratio 低 |
| **强制轻量版** | Story 数 ≤ 3 且无 UI | 轻量版 | 配置/日志类小需求 |
| **强制标准版** | 跨 ≥ 3 模块 | 标准版 | 避免轻量版漏掉联动 |

### 3.3 三种触发方案对比

| 方案 | 触发依据 | 优点 | 缺点 |
|---|---|---|---|
| **方案 1：UI ratio 单维度** | 仅看 UI ratio | 简单 | 漏掉 P0 强制 |
| **方案 2：风险等级优先**（推荐） | 风险等级 > UI ratio > Story 数 | 全覆盖 P0 | 决策树稍复杂 |
| **方案 3：混合打分** | 风险 + UI + Story 加权 | 最精细 | 阈值拍板难 |

**推荐方案 2**——保证 P0 不漏，UI ratio 二级判断兼顾效率。

## 5. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| 阈值拍板难（UI ratio 0.30/0.50 边界） | 边界需求路径选择摇摆 | 试点数据决定 / 留人工 override |
| 强制规则漏 P0 | 测试覆盖不足 | 触发决策树 P0 优先 + S7 审查员 B 校验 |
| 标准版 / 深度版产出质量差异 | 中型需求漏掉关键细节 | 标准版保留 Mermaid 页面流图 + 关键状态机 |
