# A2：用例价值评分标准 + 试点方案

> **文档类型**：v15 §5 阶段 2 应做项（L2 用例价值评分）
> **状态**：草案 / 等 D-V15-003 拍板
> **生效日期**：2026-07-16
> **继承**：v14 §3 遗留 L2 + v12 模块种子 TP 库评估经验

---

## 1. 背景与动机

### 1.1 问题陈述

当前 S6 生成的 TC 数量受限于 S5 TP 数（默认 ≥ 6 TP/Story），但**没有"用例价值密度"指标**——即：

- **同样的 50 个 TC**，有的覆盖了核心支付链路，有的覆盖了装饰动画细节
- **覆盖率达标但价值密度低**——S7 审查只能定性判断"哪些 TC 价值低"，无量化指标
- **bypass 例外滥用风险**——P1/P2 不足时易走例外，但 bypass 不能区分"低价值用例 vs 漏掉的用例"

### 1.2 期望目标

建立 **3 维度评分标准**：
- 量化每个 TC 的"业务价值密度"
- 让 S7 审查员有客观依据判定哪些 TC 可砍
- 与 bypass_log 联动：低分 TC 走 bypass = 合理；高分 TC 走 bypass = 异常

## 2. 3 维度评分标准

### 2.1 effectiveness（业务覆盖有效性）

| 分值 | 定义 | 例子 |
|---|---|---|
| **5** | 核心业务主路径，必测 | "购买 → 扣款 → 发货 → 结算" |
| **4** | 重要分支路径 | "购买失败 → 退款 → 通知" |
| **3** | 通用边界场景 | "金额为 0 / 负数 / 上限" |
| **2** | 辅助功能 | "购买按钮 hover 效果" |
| **1** | 装饰性 / 锦上添花 | "loading 动画时长" |

### 2.2 independence（用例独立性）

| 分值 | 定义 | 例子 |
|---|---|---|
| **5** | 完全独立，无需前置状态 | "登录态下点击购买" |
| **4** | 需登录态，不依赖其他 TC 状态 | "已登录玩家购买" |
| **3** | 依赖前置 TC 数据 | "续 TC-A 状态后执行 TC-B" |
| **2** | 强依赖 + 共享数据 | "与 TC-A 共享背包状态" |
| **1** | 必须按特定顺序执行 | "先跑 TC-A → 再跑本 TC" |

### 2.3 executability（可执行性）

| 分值 | 定义 | 例子 |
|---|---|---|
| **5** | QA 5 分钟内可独立跑通，无需特殊环境 | "登录 → 购买 → 看结果" |
| **4** | 5-10 分钟 + 正常测试环境 | "充值 → 等回调 → 看发货" |
| **3** | 10-30 分钟 + 部分特殊配置 | "GM 后台造数据 → 客户端验证" |
| **2** | 需多角色协作 / 跨服环境 | "玩家 A 给玩家 B 转账" |
| **1** | 难以自动化 / 需人工长时间盯 | "断网 5 分钟后恢复 → 看补偿" |

## 3. 加权方案（D-V15-003 待拍板）

### 3.1 推荐加权

```
总评分 = effectiveness × 0.5 + independence × 0.2 + executability × 0.3
```

**理由**：
- **effectiveness（0.5）**：业务价值最重要——这是评分的主轴
- **executability（0.3）**：可执行性直接影响 QA 工作量
- **independence（0.2）**：独立性是次要考量——可串行跑的用例也有价值

### 3.2 分级阈值

| 总评分范围 | 分级 | S7 审查动作 |
|---|---|---|
| **4.0-5.0** | 🟢 高价值 | 必修（MUST），不可 bypass |
| **3.0-3.9** | 🟡 中价值 | 应修（SHOULD），bypass 需理由 |
| **2.0-2.9** | 🟠 低价值 | 可改（COULD），bypass 相对宽松 |
| **1.0-1.9** | 🔴 极低 | 可砍，bypass 无需理由 |

### 3.3 3 种加权方案对比

| 方案 | effectiveness 权重 | executability 权重 | independence 权重 | 适用 |
|---|---|---|---|---|
| **方案 A（推荐）** | 0.5 | 0.3 | 0.2 | 通用 |
| **方案 B（QA 友好）** | 0.4 | 0.4 | 0.2 | QA 资源紧张 |
| **方案 C（业务优先）** | 0.6 | 0.2 | 0.2 | 业务复杂、bug 风险高 |

## 4. 试点方案

### 4.1 试点范围

- **目标需求**：最近 1 个 `test_cases.json` 完整的需求（建议 ≥ 50 TC）
- **试点时长**：1 周
- **评估方式**：
  - QA 人工打分（50 TC × 3 维度 × 1 次 = 150 次评估）
  - Agent 自动打分（同一批 TC，输出对比）
  - 双向校验：差异 ≥ 1.0 分的 TC 人工复审

### 4.2 数据契约（`case_value_pilot.json`）

```json
{
  "meta": {
    "req_name": "...",
    "version": "v1.0",
    "scored_at": "2026-07-XX",
    "scorer": "QA张三 + Agent",
    "schema_version": "v1"
  },
  "cases": [
    {
      "tc_id": "BIZ-TC-001",
      "effectiveness": 5,
      "independence": 4,
      "executability": 5,
      "total_score": 4.7,
      "level": "高价值",
      "qa_comment": "核心支付路径，必测"
    }
  ],
  "summary": {
    "total_scored": 50,
    "high_value_count": 18,
    "medium_value_count": 22,
    "low_value_count": 8,
    "very_low_count": 2,
    "avg_score": 3.6
  }
}
```

### 4.3 验收指标

| 指标 | 目标值 | 评估方式 |
|---|---|---|
| QA-Agent 打分一致率（±0.5 分） | ≥ 70% | 试点后比对 |
| 高价值 TC 通过率 | ≥ 95% | S7 审查统计 |
| 低价值 TC bypass 占比 | ≥ 50% | bypass_log 统计 |
| 试点后 v15.1 评分自动化 | 可行 | 决定是否入 v16 |

## 5. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| QA 打分主观性 | 一致率低 | 双人独立打分 + 差异复审 |
| effectiveness 权重过高 | 忽略可执行性 | 方案 A 默认 + 试点后微调 |
| 评分标准理解不一致 | 同 TC 不同人打分差 | 培训 + 评分例子库（≥ 20 个示例）|
