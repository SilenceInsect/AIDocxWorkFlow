## 0. 术语与契约

| 术语 | 定义 |
|------|------|
| **OBJ** | Requirement Object，S2 拆解出的需求对象（如 `OBJ-001 道具 SKU`） |
| **FP** | Feature Point，OBJ 下的功能点（如 `FP-001-01 单价校验`） |
| **TP** | Test Point，测试点（"测什么 + 测什么边界"），由 S5 产出 |
| **TC** | Test Case，测试用例（"怎么操作 + 怎么判定"），由 S6 产出 |
| **scenario_test_points** | S4 异常树叶子 ID（如 `BIZ-001-X.3.2`），TP 引用形成覆盖率链路 |
| **s4_reference** | S4 风险点 ID（如 `R-006`），TP 引用形成溯源链路 |

> **本草案不创造新术语**——所有字段来自 `test_points.json` / `test_cases.json` 既有契约。

### 2.3 推理链（LLM 必须输出"为什么这条 TP 存在"）

```
对每个 Story：
  1. 读取 scope_modules → 决定主 module（多个时按 §4 优先级排序）
  2. 读取每个 FP → 决定 module.subclass（按 §4 关键词命中规则）
  3. 读取 FP.verification_method → 决定 test_point_type
     - "正确路径" → POSITIVE
     - "边界 / 数值 / 时间" → BOUNDARY
     - "反例 / 校验" → NEGATIVE
     - "异常处理 / 错误码" → EXCEPTION
  4. 命中 S4 风险点 → 填 s4_reference；未命中 → s4_self_inferred + is_assumed=true
  5. 命中 S4 异常叶子 → 填 scenario_test_points；未命中 → 强制该 TP 为 EXCEPTION 类型
  6. 写 OBJ.id + FP.id → covered_objects / covered_feature_points
  7. 输出 description（人话、≤ 80 字）
```

### 2.5 故事复杂度系数（SCC）

> **Round 2 新增**：替代"≤ 50 TP/Story"旧约束。
> **详细定义**：见 `governance/design_iter/plans/v31/v31_SCC.md`

**核心公式**：
```
SCC = |actors| × |states| × |timings| × |boundaries| × |exceptions|
理论 TP 数 = SCC × TP_TYPE_FACTOR
TP_TYPE_FACTOR = { POSITIVE: 1.5, BOUNDARY: 1.0, NEGATIVE: 1.0, EXCEPTION: 0.5 }
软下限 = 理论 TP 数 × 0.8
```

**维度说明**：

| 维度 | 来源 | 简单 | 复杂 |
|------|------|------|------|
| actors（行为主体）| Story 的参与者 | 1 | ≥ 3 |
| states（对象状态）| `requirement_objects.json` state_machine 字段 | 1~2 | ≥ 5 |
| timings（时序变体）| 正常/零点/跨日/并发时序 | 1 | ≥ 3 |
| boundaries（边界条件）| 数值/数量/字符边界 | 1~4 | ≥ 4 |
| exceptions（异常分支）| S4 exception_tree_leaves[] | 1 | ≥ 5 |

**使用规则**：
1. 对每个 Story，先估算 SCC
2. 计算理论 TP 数（按主型）
3. 若实际 TP 数 < 软下限 → `assumption_basis` 说明"缺少 {维度} 维度 TP"
4. 不设硬上限（复杂 Story 可以 SCC ≥ 100，对应 150+ 理论 TP）

**维度 S2 字段映射**（Round 3 DT-2 补全）：

| 维度 | S2 字段来源 | 计数方法 |
|------|-------------|---------|
| actors | `requirement_objects.json#objects[].scene` 中主体角色 | 取 scene 描述中的独立主体数 |
| states | `requirement_objects.json#objects[].feature_points[].normal_flow` 中的状态节点 | 抽取主谓宾状态节点 + S4 异常树深度 |
| timings | `backlog.json#stories[].acceptance_criteria[]` 步骤数 + `requirement_objects.json#objects[].feature_points[].normal_flow` 时序节点 | 找"零/活动/赛季/跨日/并发"关键词 + 步骤数 |
| boundaries | `requirement_objects.json#objects[].feature_points[].input` 中数值范围 + 显式边界 | 解析"min/max/长度/数量"取值 |
| exceptions | `requirement_objects.json#objects[].feature_points[].exception_flow[]` + S4 `business_flow.json#exception_tree_leaves[]` | 合并 + 去重 |

**timings 计数举例**：
- "登录 → 验证 → 跳转" = 3 timings（3 个步骤节点）
- "活动开始 → 活动期间 → 活动结束" = 3 timings（3 个时序变体）
- "正常 + 零点重置 + 跨日" = 3 timings（3 个时序变体）

**示例**（商城道具购买 Story）：
```
actors=2（玩家+系统）× states=5（未选→已选→购买中→成功→失败）
× timings=3（正常+活动期间+零点重置）× boundaries=4（数量+价格）
× exceptions=5（网络+余额+售罄+并发+支付）
= SCC=600 → 理论 TP（POSITIVE）=600×1.5=900 → 软下限=720
```

> **v31 修订注**：STAGE_S6_TEST_CASES.mdc §字段语义规范中规定 `用例描述` = `S2 backlog.epics[].title`、`功能描述` = `S2 backlog.stories[].title`（均 100% 严格相等）。本表 schema 与 STAGE 一致。`is_exploratory / exploratory_reason / verified_against_s2_path / module_boundary_check` 是 v31 草案 §3.4 + §3.2 增强字段，不与 STAGE 字段冲突，作为可选字段。

### 3.3 推理链（LLM 必须输出"为什么这条 TC 存在"）

```
对每个 TP：
  1. 取 TP.test_point_type → 决定 scenario
  2. 取 TP.test_input → 拆为 steps[].action + steps[].data
  3. 取 TP.expected_result → 拆为 steps[].expected（逐步判定）
  4. 取 TP.precondition → 拆为 preconditions[] 步骤列表
  5. 取 TP.covered_objects → obj_id = 该列表第 1 个（多 OBJ 时选主 OBJ）
  6. 取 TP.covered_feature_points → feature_point_ref[]（完整继承）
  7. 取 TP.scenario_test_points → scenario_test_points[]（完整继承）
  8. 取 TP.priority → priority（完整继承）
  9. 生成 case_id = {TP.id}-TC-NN（每个 TP 至少 1 个 TC；BOUNDARY 类可能生成 3 个）
```

### 3.4 TC 字段语义收紧规则

> **Round 2 新增**：解决"模型喜欢往常态经验靠拢"问题——TC 字段必须严格对应 S2 真实 OBJ/FP 条目。

#### 3.4.1 三条铁律

| 铁律 | 规则 | 违规处理 |
|------|------|---------|
| **铁律 A** | `scenario`（用例描述）必须 == `S2 backlog.epics[].title` 或 `stories[].title`（严格相等，非包含） | 违规 → 强制改写 |
| **铁律 B** | `description`（功能描述）中的每个业务关键词必须能在 `S2 requirement_objects.json` OBJ.name 或 FP.name 中找到匹配（允许探索性测试建议新 OBJ/FP，但需标 `is_exploratory: true`） | 违规 → 标 `is_exploratory: true` + 在 `assumption_basis` 中引用"建议补充的 OBJ/FP 路径" |
| **铁律 C** | `obj_id` 和 `feature_point_ref` 必须从 `TP.covered_objects` 和 `TP.covered_feature_points` 直接取值，不允许 LLM 自补 | 违规 → 强制回退到 TP 的字段值 |

#### 3.4.2 判断方法（LLM 自检执行）

```
步骤 1：从 S2 backlog.json 提取 epic_title_set = {epic.title for epic in epics}
步骤 2：从 S2 backlog.json 提取 story_title_set = {story.title for story in stories}
步骤 3：从 S2 requirement_objects.json 提取 obj_name_set = {obj.name for obj in objects}
步骤 4：从 S2 requirement_objects.json 提取 fp_name_set = {fp.name for fp in feature_points}

步骤 5：对每个 TC：
  - scenario ∈ epic_title_set ∪ story_title_set？  否 → 强制改写
  - 提取 description 关键词集合 keyword_set
  - keyword_set ⊆ obj_name_set ∪ fp_name_set？ 否 → 对差集：
      - 若差集关键词是探索性建议 → is_exploratory=true + assumption_basis 引用建议路径
      - 若差集关键词是 LLM 自补 → 强制删除，替换为 TP.covered_objects[0].name
```

#### 3.4.3 is_exploratory 字段

当 `is_exploratory=true` 时，新增字段：

| 字段 | 值 |

| `is_exploratory` | `true` |

| `suggested_fp` | 建议补充的 FP.name（选填）|

#### 3.4.4 自检清单

```
Self-check（每条 TC）：
□ scenario ∈ {epic.title} ∪ {story.title}（严格相等）
□ description 关键词 ⊆ {OBJ.name} ∪ {FP.name}
□ 若存在差集 → is_exploratory=true + assumption_basis 非空
□ obj_id ∈ TP.covered_objects
```

### 5.1 推导原则

## 6. 覆盖率计算公式

> **本节是 §4.3 SSOT 的具体化**——给出可执行的计算脚本 + 字段溯源报告。

### 6.1 OBJ 覆盖率

```
OBJ 覆盖率 = TP 集合引用的 OBJ 总数 / S2 OBJ 总数
（分母 = requirement_objects.json#objects 长度；
 分子 = ∪(test_points.json#covered_objects) 长度）
```

### 6.2 FP 覆盖率

```
FP 覆盖率 = TP 集合引用的 FP 总数 / S2 FP 总数
（分母 = ∑(objects[].feature_points.length)；
 分子 = ∪(test_points.json#covered_feature_points) 长度）
```

### 6.3 异常路径覆盖率（S4 叶子）

```
异常路径覆盖率 = TP 引用的异常叶子数 / S4 异常叶子总数
（分母 = business_flow.json#exception_tree_leaves 长度 = 66；
 分子 = ∪(test_points.json#scenario_test_points) 长度）
```

> **SSOT §4.3 `S4_ANOMALY_COVERAGE = 1.0`** —— 100% 为硬阈值；未覆盖必须显式标注 `skip_reason` ∈ {`deprecated` / `out_of_scope` / `low_risk` / `manual_test` / `unknown`}。

### 6.4 风险点覆盖率

```
风险点覆盖率 = TP 引用的 S4 风险点数 / S4 风险点总数
（分母 = business_flow.json#risks 长度 = 25；
 分子 = ∪(test_points.json#s4_reference) 长度）
```

### 6.5 字段溯源报告

每条 TP / TC 必须能在字段溯源报告 §2.5 / §3.5 表格中找到"已读且匹配"行；否则视为**字段悬挂**（Q1.3 失败）。

## 8. S8 回灌 TP 库契约

> **Round 2 新增**：基于 `governance/design_iter/plans/v31/s8_knowledge_backflow_diagnosis.md` 根因诊断。
> **适用范围**：S8 自迭代 → 知识沉淀 → TP 库入库

### 8.1 根因诊断结论

**TP 库为空的根因不是"S8 没干活"，而是"S8 的双段归档链路从未被触发"**：

| 链路段 | 预期行为 | 实际行为 | 根因 |
|--------|---------|---------|------|
| 第 1 段（本地队列）| S8 识别 must_fix/should_fix → 写入 `.review_queue/` | v3.01 无 S7/S8 产物 → **链路从未触发** | v3.01 未执行 S7 → 无 S8 输入 → 无归档触发 |
| 第 2 段（公共库）| 人工审核 `.review_queue/` → 回写 `test_point_library/` | **从未执行** | 人工程序缺失 |
| 整体 | 跨项目知识积累 | 零积累 | 整个 v3.01 流程未走完 S5→S6→S7→S8 |

### 8.2 归档链路（两段制）

```
S8 识别 must_fix/should_fix 根因
    ↓
[第 1 段] 写入 knowledge/project_local/.review_queue/<Module>/<Subclass>__<defect_id>__<date>.md
    ↓
[人工审核] 触发通知 → 人工阅读候选 → 通过/拒绝
    ↓
[第 2 段] 通过 → 回写 knowledge/public/test_point_library/<MODULE>/<Subclass>.md
    ↓
TP 库累计条目 → S5 生成时优先复用 → tpl_id + usage_count++
```

### 8.3 候选文件命名规范

```
knowledge/project_local/.review_queue/<Module>/<Subclass>__<defect_id>__<date>.md
```

**示例**：
```
knowledge/project_local/.review_queue/BIZ/A_biz_logic__ITER-012__2026-07-21.md
knowledge/project_local/.review_queue/SPECIAL/C_weak_net__ITER-015__2026-07-21.md
```

### 8.4 候选文件格式

```markdown
# 经验补充候选

> 来源：S8 iteration.json #ITER-012
> 根因：S5_MODULE（模块归属错误）
> 缺陷：弱网场景应归 SPECIAL 但错标为 BIZ
> 建议回写：`knowledge/public/module_templates/BIZ/O_boundary.md` §3 误判案例

## 补充内容

> **补充类型**：新增误判案例
> **目标文件**：`knowledge/public/module_templates/BIZ/O_boundary.md`
> **目标段落**：§3 常见误判案例
> **建议补充内容**：
- 弱网超时 → 归 SPECIAL（弱网容错），不归 BIZ（网络层）

## 审核意见（人工填写）

- [ ] 通过 → 回写目标文件
- [ ] 拒绝 → 附原因
- [ ] 需修改 → 附修改意见

审核人：_______ 日期：_______
```

### 8.5 入库标准（SSOT：`test_point_library/README.md` §入库流程）

| 标准 | 值 |
|------|---|
| 复用门槛 | 被 ≥ 2 个需求复用 |
| 质量门槛 | 经 S7 审查 PASS |
| 格式要求 | 剥离需求特定信息，保留"测试思路" |
| 归档层级 | OBJ.id / FP.id 替换为模板变量 |

### 8.6 TP 库激活阈值

当 `test_point_library/<MODULE>/` 累计 ≥ 10 个有效条目时，激活"自动复用"机制：

```
S5 生成 TP 前：
1. 在 test_point_library/ 中匹配相同 module + subclass + test_point_type 的条目
2. 匹配成功 → 复用模板 + 填入当前需求特定信息 + tpl_id + usage_count++
3. 匹配失败 → 从 module_templates 生成新 TP
```

### 8.7 与 recurring_failures.json 的区别

| 维度 | `recurring_failures.json` | `test_point_library/` |
|------|------------------------|---------------------|
| 写入者 | `recurring_failures.py`（S8 聚合）| 人工审核后回写 |
| 位置 | `workflow_assets/_governance/` | `knowledge/public/` |
| 性质 | 项目内本地失败模式记录 | 公共可复用 TP 模板 |
| 用途 | 防止同类错误重复出现 | 供后续需求 S5 生成时复用 |

### 8.8 人工审核触发契约（DT-3 Round 3 新增）

> **问题**：§8.1 描述"人工审核 → 回写公共库"，但缺少"如何通知人工有新候选"机制。
> **解法**：在 `iteration.json` 顶层加 `pending_candidates` 字段，触发项目流程约定的人工审核。

**字段定义**（写入 `workflow_assets/<req_name>/<version>/「S8 自迭代」/iteration.json`）：

```json
{
  "iteration_number": 1,
  "verdict": "PASS",
  "review_queue_triggered": true,
  "pending_candidates": [
    {
      "path": "knowledge/project_local/.review_queue/BIZ/A_biz_logic__ITER-012__2026-07-21.md",
      "module": "BIZ",
      "subclass": "A_biz_logic",
      "defect_id": "ITER-012",
      "root_cause_source": "S5_MODULE",
      "created_at": "2026-07-21T00:00:00+08:00",
      "requires_human_review": true,
      "target_public_file": "knowledge/public/module_templates/BIZ/O_boundary.md"
    }
  ],
  "next_iteration_focus": []
}
```

**触发流程**：

1. S8 写入 `.review_queue/` 候选文件
2. S8 iteration.json 自动填 `review_queue_triggered: true` + `pending_candidates: [...]`
3. 项目流程约定（人工）定期检查 `iteration.json#pending_candidates`
4. 审核通过 → 回写 `test_point_library/` + `tpl_id` 登记
5. 审核拒绝 → 标记 `rejected`，由 S8 下一轮迭代处理

**STAGE_S8 建议修改**：在 `STAGE_S8_SELF_ITERATION.mdc` §4.1 归档条件中加：
> S8 执行完成后，iteration.json 必须填 `pending_candidates` 数组（含 path / module / subclass / defect_id / target_public_file 五个字段），项目流程据此触发人工审核。
