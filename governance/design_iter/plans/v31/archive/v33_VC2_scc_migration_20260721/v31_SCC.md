# v31 故事复杂度系数（Story Complexity Coefficient，SCC）

> **文档性质**：v31 方法论的子规范，Round 2 新增
> **适用范围**：S5 测试点数量估算
> **替代旧约束**："≤ 50 TP/Story" 硬上限（已废除）
> **Round 2 新增**：Round 1 audit 发现 "≤ 50 TP/Story" 不合理 → Round 2 采纳 SCC 公式替代

---

## 1. 核心概念

**Story Complexity Coefficient（SCC）**：量化单个 Story 的测试点生成复杂度，作为 S5 TP 数量的软下限依据。

**公式**：
```
SCC = |actors| × |states| × |timings| × |boundaries| × |exceptions|
```

其中：
- `|actors|` = 行为主体数量（玩家、角色、系统账号等）
- `|states|` = 对象状态数（初始/正常/临界/异常）
- `|timings|` = 时序变体数（前/中/后、零点、跨日等）
- `|boundaries|` = 边界条件数（最小值/临界值/最大值/超界）
- `|exceptions|` = 异常分支数（网络/权限/并发/配置等）

**理论 TP 数**：
```
理论 TP 数 = SCC × TP_TYPE_FACTOR
TP_TYPE_FACTOR = { POSITIVE: 1.5, BOUNDARY: 1.0, NEGATIVE: 1.0, EXCEPTION: 0.5 }
```

**加权说明**：
- POSITIVE × 1.5：主路径覆盖更全面（正例是业务核心）
- BOUNDARY × 1.0：边界值与主路径 1:1
- NEGATIVE × 1.0：反例与主路径 1:1
- EXCEPTION × 0.5：异常路径是主路径的子集（异常往往共享主路径结构）

**软下限**：
```
TP 数量 ≥ 理论 TP 数 × 0.8
```

低于此值 → LLM 必须在 `assumption_basis` 中说明"少在哪类维度"

## 2. 维度说明

### 2.1 actors（行为主体）

| 情况 | 示例 | 计数 |
|------|------|------|
| 单主体 | 玩家操作 | 1 |
| 双主体 | 玩家 + 系统 | 2 |
| 多主体 | 玩家 + 系统 + 运营后台 | 3 |
| 跨端主体 | 客户端玩家 + 服务端 GM | 2 |

### 2.2 states（对象状态）

| 情况 | 示例 | 计数 |
|------|------|------|
| 无状态对象 | 静态展示 | 1 |
| 简单状态 | 开启/关闭 | 2 |
| 状态机 | 未激活/激活/进行中/完成/取消 | 5 |
| 复杂状态机 | 活动状态 × 角色状态交叉 | ≥ 5 |

**状态来源**：
- `backlog.json` story 的 acceptance_criteria 中的状态描述
- `requirement_objects.json` 中 OBJ 的 state_machine 字段
- S3 prototype.json 中 states 数组

### 2.3 timings（时序变体）

| 情况 | 示例 | 计数 |
|------|------|------|
| 无时序 | 静态展示 | 1 |
| 单一时刻 | 固定时间点 | 1 |
| 多个时刻 | 开服/活动期间/活动结束 | 3 |
| 跨时区 | UTC+8 / UTC+0 | 2 |
| 零点重置 | 0点前/0点后 | 2 |
| 赛季周期 | 赛季内/休赛期 | 2 |
| 并发时序 | 正常/超时/乱序 | 3 |

### 2.4 boundaries（边界条件）

| 情况 | 示例 | 计数 |
|------|------|------|
| 无边界 | 纯展示 | 1 |
| 数值边界 | 最小值/临界值/最大值/超界 | 4 |
| 数量边界 | 0/1/最大/N+1 | 3 |
| 时间边界 | 活动开始/活动结束/结束前1秒/结束后1秒 | 4 |
| 字符边界 | 空/1字符/最大字符/超长 | 4 |

**边界值来源**：
- `requirement_objects.json` 中 FP 的 input 字段
- 显式边界（"不超过100"→ 100, 99, 101）
- 隐式边界（数据库字段长度、API 限流阈值）

### 2.5 exceptions（异常分支）

| 情况 | 示例 | 计数 |
|------|------|------|
| 无异常 | 纯正向流 | 1 |
| 单一异常 | 网络超时 | 2 |
| 多种异常 | 网络超时 + 权限不足 + 并发冲突 | 4 |
| 异常组合 | 异常 × 降级 | 2+ |

**异常来源**：
- `requirement_objects.json` 中 FP 的 exception_flow 数组
- S4 business_flow.json 的 exception_tree_leaves[]
- SPECIAL 模块的子类枚举（A-I）

## 3. SCC 计算示例（商城样本）

### 示例 Story：BIZ-001 道具购买

**Story 信息**（来自 `backlog.json`）：
```
Story: 玩家在商城购买道具
主路径：玩家选择道具 → 确认购买 → 扣款 → 发货
```

**维度计数**：

| 维度 | 分析 | 计数 |
|------|------|------|
| actors | 玩家 + 系统（支付回调）| 2 |
| states | 未选择 → 已选 → 购买中 → 购买成功 → 购买失败 | 5 |
| timings | 正常时刻 + 活动期间（折扣）+ 零点重置后 | 3 |
| boundaries | 道具数量（0/1/最大/N+1）+ 价格（0/正数/超限）| 4 |
| exceptions | 网络超时 + 余额不足 + 道具售罄 + 并发超卖 + 支付失败 | 5 |

**SCC 计算**：
```
SCC = 2 × 5 × 3 × 4 × 5 = 600
```

**理论 TP 数**（按主型）：
```
理论 TP 数 = 600 × TP_TYPE_FACTOR
如果主型 = POSITIVE（主路径）：600 × 1.5 = 900
如果主型 = BOUNDARY（边界为主）：600 × 1.0 = 600
```

**实际 TP 期望**：
```
TP ≥ 理论 TP 数 × 0.8 = 480 ~ 900
```

> **注**：此 Story 的 SCC 600 远超"≤ 50"旧上限。说明旧上限在复杂业务场景下严重低估了真实需求。

## 4. SCC 简化版（快速估算）

当无法精确计数时，使用简化版 SCC：

| Story 复杂度 | SCC 估算 | 理论 TP 数（POSITIVE 主型）|
|-------------|---------|--------------------------|
| 简单（单路径、无状态、无异常）| 1~5 | 1~8 |
| 一般（2-3 个路径、有状态、少量异常）| 5~20 | 8~30 |
| 中等（多路径、状态机、有异常）| 20~100 | 30~150 |
| 复杂（跨系统、状态机、跨时序、大量异常）| 100~500 | 150~750 |
| 极复杂（多主体、跨服、并发、高风险）| ≥ 500 | ≥ 750 |

## 5. SCC 与旧约束的对比

| 维度 | 旧约束（≤ 50 TP/Story） | SCC 公式 |
|------|-------------------------|--------|
| 适用性 | 任意 Story | 任意 Story |
| 复杂度量化 | ❌ 无量化 | ✅ 5 维度乘积 |
| 上限 | 硬上限 50 | 软下限 0.8 × 理论 TP 数 |
| 超出时的处理 | 无处理 | 显式说明"少在哪类维度" |
| 简单 Story | 可能过多 | SCC ≈ 1~5 → 理论 1~8 |
| 复杂 Story | 强制截断（不合理）| 按公式计算，无截断 |

## 6. SCC 在 S5 LLM Prompt 中的使用方式

```
## SCC 估算（每个 Story 前执行）
1. 识别 actors / states / timings / boundaries / exceptions
2. 计算 SCC = |actors| × |states| × |timings| × |boundaries| × |exceptions|
3. 计算理论 TP 数 = SCC × TP_TYPE_FACTOR
4. 估算 TP 数量 ≥ 理论 TP 数 × 0.8
5. 若实际 TP < 理论 × 0.8 → 在 assumption_basis 中说明"缺少 {维度} 维度"

## Self-check
□ 每个 Story 的 TP 数 ≥ 理论 TP 数 × 0.8
□ 若低于软下限，assumption_basis 非空
```

## 7. 与 STAGE_S5_TEST_POINTS.mdc 的关系

- **旧约束**：STAGE_S5 中有"≤ 50 TP/Story"硬上限 → **废除**（本 SCC 替代）
- **新约束**：STAGE_S5 中的"每 Story ≥ 6 TP 指导值"保留（作为简单 Story 的参考基准）
- **SCC vs 6 TP 指导值**：6 TP 是"简单 Story"的下限；SCC 是"复杂度匹配"的软下限；两者互补
