## 现状评估（§10.4 基于已有实现推理）

### 文件碎片化问题

| 文件 | 内容 | 问题 |
|---|---|---|
| `MODULES.md` §3.5 | 交叉场景快速归类表 | 是总表，不是决策流程 |
| `_decision_tree.md` | 二叉决策树（97 行） | 独立文件，S5 Agent 不一定先读 |
| `S5 SKILL.md` §1.2 | 模块 × 类型双维度强制规则 | 是约束，不是归类方法 |
| `O_boundary.md` × 6 | 各模块判定三问 + 边界规则 | **分散在 6 个子目录下** |

实际：S5 Agent 归类时读 `MODULES.md` + `_decision_tree.md`，不一定进子目录读 `O_boundary.md`。

### O_boundary 反例库——不是数据库，是 3~5 个典型案例

我提出"反例库"但没说清楚规模 → §10.5 无根据结论。

**实际需求**：
- O_boundary = **边界规则 + 正例 + 反例**（3~5 个案例，说清楚"这个归 X 不是 Y"）
- **不是 SQLite / JSON 库**——是叙述型文档
- **够用标准**：以 S5 历史误标案例驱动补充，没有误标就不补

## 解决方案

### 方案：单一入口文件

将分散在 4 处（MODULES.md §3.5 + `_decision_tree.md` + 6 个 O_boundary + SKILL §1.2）的内容合并为**单一文件**：

**`knowledge/public/module_templates/MODULARITY_GUIDE.md`**

结构：
1. 顶层二叉决策树（Mermaid，5 个问题）
2. 8 模块总表（含 §3.5 交叉场景对照表）
3. 高频混淆边界（5~8 个典型案例，正反对比，不拆子目录）
4. S5/TP 模块标注规范（从 SKILL.md §1.2 提取）

不新增数据库 / 不新建子目录 / 不改 O_boundary 现状——**合并不是替代**。
