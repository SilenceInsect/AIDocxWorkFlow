# LINK 模块：关联 + 回归 全链路图

> **本文档** = LINK 模块双职责的工程化落档——基于 2026-07-21 用户拍板修正版 3
>
> ⚠️ **链路图 B 经历了 3 轮校准**——每一次都是 Agent 臆造后被用户纠正：
> - **Round 1（被否认）**："关联 = 测试可追溯 4 层链 / 回归 = 系统级拓扑" — 用户 9:23 否
> - **Round 2（被否认）**："回归 = 子功能模块静态范围扩展" — 用户 9:27 否
> - **Round 3（采纳）**："回归 = **变更视角**：新逻辑（含新增/修改）波及到已有通用能力时，**对该通用能力的原有流程做一次回归验证**" — 用户 9:27 实例确认
>
> **正确的双职责口径（修正版 3）**：
>
> - **关联** = **功能 ↔ 功能** 之间的结构关系，4 种典型形态：
>   - **功能跳转**（点击 A → 跳转到 B）
>   - **从属关系**（A 包含 B / B 从属于 A）
>   - **数据传递**（A → 传出字段 → B 接收）
>   - **触发链**（A 完成 → 触发 B / B 完成 → 触发 C）
>
> - **回归** = **变更视角触发**：
>   - **触发器**：新功能上线 / 已有逻辑被修改（含新增/修改）
>   - **波及识别**：S2 拆解时识别哪些"通用能力 / 通用流程 / 共享底层 / 上游下游"会被这次改动碰到
>   - **回归执行**：对该波及到的"通用能力原有流程"做一轮回归测试
>   - **目的**：确认"改动后通用流程还是正常的"——**不是测新逻辑本身**（新逻辑归 BIZ/UI），**是测通用流程没有被新逻辑破坏**
>   - **回归 ≠ 测试回归集** + **回归 ≠ 新逻辑功能测试** + **回归 ≠ 静态全量范围**
>
> **链路图 A（关联）** = 功能跳转图 / 从属树 / 数据流图（**功能之间**的结构）
> **链路图 B（回归）** = **变更触发 → 波及识别 → 通用能力回归验证清单**（**变更视角下的回归触发链**）

## 2. 链路图 A · 功能 ↔ 功能 关联（关联职责）

### 2.1 4 种典型形态

#### (1) 功能跳转
```
[商城首页] ──navigate_to──▶ [道具详情页] ──navigate_to──▶ [购买确认页] ──navigate_to──▶ [支付页]
```

#### (2) 从属关系
```
[商城首页]
   ├── belongs_to: [道具分类导航]
   ├── belongs_to: [热门道具列表]
   │      └── belongs_to: [道具卡片 N]（含名称/价格/缩略图）
   ├── belongs_to: [搜索框]
   └── belongs_to: [分页器]
```

#### (3) 数据传递
```
[道具详情页]
   passes_data ──▶ {道具ID, 道具名称, 单价, 限购数量}
                                          │
                                          ▼
                                     [购买确认页]（接收 + 展示 + 计算总价）
                                          │
                                          passes_data ──▶ {订单ID, 道具ID, 数量, 支付方式, 总价}
                                                                              │
                                                                              ▼
                                                                          [支付页]
```

#### (4) 触发链
```
[点击购买] ──triggers──▶ [订单创建] ──triggers──▶ [支付执行] ──triggers──▶ [道具到账] ──triggers──▶ [邮件通知]
                                              │                                │
                                              └── on_fail: [退款补偿] ◀────────┘
```

## 3. 链路图 B · 变更视角回归（回归职责）

### 3.0 核心结构（**新视角，替代 v34 Round 2 的"系统级拓扑"**）

回归不是"识别系统间依赖"，而是**4 步触发链**：

```
┌──────────────────────────────────────────────────────────────────────────┐
│  [1] 触发器（Trigger）                                                     │
│      新功能 / 新逻辑上线 / 已有逻辑被修改（含新增/修改）                          │
│      例：新做"多层副本"功能 → 含新阶段 + 新重连逻辑 + 发奖逻辑                     │
├──────────────────────────────────────────────────────────────────────────┤
│  [2] 波及识别（Affected Capability Identification）                            │
│      S2 拆解时识别：这次改动碰到的"通用能力 / 通用流程 / 共享底层 / 上游下游"          │
│      例：多层副本新增 → 碰到的通用能力 = {通用副本阶段、通用副本流程、通用发奖流程}       │
├──────────────────────────────────────────────────────────────────────────┤
│  [3] 回归执行（Regression Execution）                                        │
│      对每个被波及的"通用能力"做一次原有流程的回归测试 —— **测通用流程原有路径，不是测新逻辑**│
│      例：跑一遍"原通用副本流程"，确认改动后仍正常                                   │
├──────────────────────────────────────────────────────────────────────────┤
│  [4] 目的（Goal）                                                           │
│      **确认改动后通用流程还是正常的** —— 不是测新功能本身，是测"通用流程没被新功能破坏"     │
└──────────────────────────────────────────────────────────────────────────┘
```

### 3.1 实例：多层副本功能（用户原话）

**触发器**：
```
新做"多层副本"功能
   ├── 新增：副本阶段 1-N（多阶段机制，区别于通用副本的单一阶段）
   ├── 新增：新重连逻辑（断线后从原阶段继续，而不是重置到第 1 阶段）
   └── 新增：发奖逻辑（按多阶段累计发奖）
```

**波及识别**（用户原话推断）：
```
"通用副本逻辑"
   ├── 通用副本阶段
   ├── 通用副本流程
   └── 通用发奖流程
```

**回归执行**：
```
对"通用副本逻辑"原有流程跑一遍回归测试
   ├── 跑一遍"单一阶段副本流程"（没碰新功能前怎么走，现在怎么走）
   ├── 跑一遍"通用副本重连逻辑"（断线后默认重置到第 1 阶段，没碰新功能前怎么走，现在怎么走）
   └── 跑一遍"通用发奖流程"（按单一阶段发奖，没碰新功能前怎么走，现在怎么走）
```

**目的**：
```
✅ 确认改动后，通用副本的流程还是正常的
❌ 不是测"多层副本新功能本身"（那是 BIZ/UI 模块的工作）
```

**核心学习（避免下次再臆造）**：
1. **关联** = 用户明确给过 4 种形态（跳转/从属/数据传递/触发链）—— **这是用户原话，不是推断**
2. **回归** = 用户给过 2 个层次的原话：
   - 第 1 原话："回归是系统功能关联功能回归，比如活动系统要回归任务系统..." → 提示"系统级"，但不是"拓扑"
   - 第 2 原话（关键）："你拆解完需求，应该就能知道有哪些功能模块，你的完整的功能是，各个子功能系统上新增或者修改上构建出来的，所以，除了要测试需求本身，也要回归各个关联的子功能" → **完整功能 = 子功能叠加构建**，**关联的子功能都要回归**
   - 第 3 原话（关键）："比如的副本是新做的多层副本，副本阶段个通用副本阶段不一样，你有新的重连逻辑，发奖逻辑，那么通用副本逻辑就要做回归，确认改动后，通用副本的流程还是正常的" → **变更触发 → 波及 → 通用能力原有流程回归验证**
