## 2. 11 项行为契约（每项附用户原话 + 实现机制）

> **用户原话 #7**："你需要思考怎么进行角色创造，行为创造，行为约束，行为推进，确定行为计划，行为执行规范，方法论，行为结果审计，反思，迭代，经验沉淀，模板维护"
>
> **11 项 = 用户原话拆解**（按出现顺序）：

### 2.1 角色创造（Role Creation）

- **用户原话**：#6 "生成对应的模块专项资深的专家"
- **机制**：8 角色卡 §1 已固化（CONFIG/UI/BIZ/UTIL/LINK/SPECIAL/LOG/HINT）
- **核心约束**：**8 角色是 SSOT**（`.cursor/MODULES.md` §4.X 已定义）—— 不允许 Agent 在 8 角色基础上"再细分 / 再合并 / 再创造新角色"
- **待用户拍板（Q-001）**：每个角色"专家身份"的具体深度（如 BIZ 角色 = 服务端业务逻辑专家 vs 业务架构师 vs DBA）| 状态：**PENDING**

### 2.2 行为创造（Behavior Creation）

- **用户原话**：#2 "每个角色按自己的模块，对文档进行深度阅读和思考"
- **机制**：每个角色的"过程行为"= 读模板（学习来源）→ 读需求（输入）→ 思考（用户原话 #2："深度阅读和思考"）→ 输出（产物）
- **核心约束**：行为过程必须留下**思考产物**（不是黑盒），如 S1 角色输出"模块覆盖矩阵"，S2 角色输出"template_refs"
- **与 v33 goal-loop 关系**：行为过程可视化为 goal-loop 5 段式（Plan → Act → Observe → Reflect → Refine）
- **待用户拍板**：8 角色"思考"的产出形态（思考清单 / 决策树 / 反例 / 缺漏清单）| 状态：**PENDING**

### 2.3 行为约束（Behavior Constraints）

- **用户原话**：#6 "他们负责从模块的模板里学习和思考" + #7 "行为约束"
- **机制**：每角色**只能从自己模块的 `<MODULE>/<X>_*.md` 学习**——不允许跨模块读模板（避免"越界出主意"）
- **跨模块场景的处理**：用户原话 #4 "补齐策划需求没有的，找到矛盾" → **跨模块的矛盾由冲突矩阵判定，不允许一个角色"自决"**
- **核心约束**：每角色的输入/输出边界必须显式定义（见 §1.2）
- **待用户拍板**：角色"跨界"边界（如 BIZ 角色发现涉及 LINK 模块，是上报冲突矩阵还是自动触发 LINK 角色）| 状态：**PENDING**

### 2.4 行为推进（Behavior Propulsion）

- **用户原话**：#7 "行为推进"
- **机制**：8 角色在阶段内的推进顺序（**并行 vs 串行**）
  - **并行**：8 角色同时启动 → `goal_parallel_executor.py` 调度（v33 §10 已规划但未实现）
  - **串行**：CONFIG → UI → BIZ → UTIL → LINK → SPECIAL → LOG → HINT（按依赖关系）
- **核心约束**：行为推进**必须有可审计产物**（不能"默默跑完"）
- **待用户拍板（Q-002）**：并行执行是 goal_parallel_executor.py 真并行，还是 LLM 上下文分块思考 | 状态：**PENDING**

### 2.5 确定行为计划（Behavior Plan）

- **用户原话**：#7 "确定行为计划"
- **机制**：每角色在阶段开始前**明确**：
  - 任务范围（输入哪些文件、产出哪些文件）
  - 时间预算（角色思考时长 / 产物复杂度）
  - 依赖关系（依赖哪些其他角色的输出）
- **核心约束**：行为计划落到 `v34_<stage>_plan_<role>.md`，可审计
- **待用户拍板**：行为计划颗粒度（粗：阶段级 / 中：OBJ 级 / 细：FP 级）| 状态：**PENDING**

### 2.6 行为执行规范（Behavior Execution Standards）

- **用户原话**：#7 "行为执行规范"
- **机制**：
  - 每角色输出必须遵守 8 模块子系统规范（如 S2 OBJ 输出必须含 `template_refs[]`）
  - 每角色输出必须有"策划需求 vs 通用经验对比"段（用户原话 #4）
  - 每角色产物必须可机检（不能是"叙述型 markdown 自由文本"）
- **核心约束**：参考 `DESIGN_AND_EXECUTION_STANDARDS.mdc` §2.1 通用执行流程（输入审查 → 材料提取 → 质量门禁 → 产出生成 → 失败报告）
- **待用户拍板**：行为执行规范的"机检严格度"（强校验 vs 弱建议）| 状态：**PENDING**

### 2.7 方法论（Methodology）

- **用户原话**：#7 "方法论"
- **机制**：每角色使用的方法论 = **学习来源模板的方法论**（即 `<MODULE>/<X>_*.md` 第 1-5 段："子类定义 / 典型场景 / 种子测试点 / 边界陷阱 / 常见误判"）
- **核心约束**：方法论**不外推**——不引入"通用测试方法论"覆盖模板方法论
- **与 v33 关系**：v33 §3 v31 方法论已迁移（`v31_方法论_草案.md`）—— 8 角色方法论是 v31 方法论的"模块化版本"
- **待用户拍板**：方法论是否每个角色不同（深度专业化）vs 8 角色共用一套方法论框架 | 状态：**PENDING**

### 2.8 行为结果审计（Behavior Result Audit）

- **用户原话**：#7 "行为结果审计"
- **机制**：
  - 每角色产物由"**审计员**"复核（双角色制：A 角色产出 + B 角色审计）
  - 审计员 = 跨模块视角的"全局角色"（如 `_common_structure.md` 通用结构专家）
  - 审计产物：`v34_<stage>_audit_<role>.md`
- **核心约束**：审计必须基于产物文件 + 学习来源对比（不凭印象）
- **与 v33 goal-loop 关系**：审计 = v33 goal-loop 第 4 段（Observe）+ 第 5 段（Reflect）的合并
- **待用户拍板**：审计员角色（固定全局角色 vs 邻接角色 vs 全部 8 角色轮值）| 状态：**PENDING**

### 2.9 反思（Reflection）

- **用户原话**：#7 "反思"
- **机制**：每角色在阶段结束后**显式回答**：
  - 我产出了什么？
  - 我漏了什么？（用户原话 #4："补齐策划需求没有的"）
  - 我和哪个角色冲突了？（用户原话 #4："找到策划需求和通用经验矛盾的"）
  - 我的学习来源（模板）有哪些不够？（为模板维护提供输入）
- **核心约束**：反思产物必须落到 `v34_<stage>_reflection_<role>.md`
- **与 v33 goal-loop 关系**：反思 = v33 goal-loop 第 5 段（Reflect）的单角色版
- **待用户拍板**：反思周期（每阶段一次 vs 每 Story 一次 vs 每 OBJ 一次）| 状态：**PENDING**

### 2.10 迭代（Iteration）

- **用户原话**：#7 "迭代"
- **机制**：基于反思（§2.9）→ 下一轮角色行为调整
  - **输入**：反思产物 + 审计产物
  - **输出**：行为调整清单（哪些模板该改、哪些规范该改、哪些产物该改）
- **核心约束**：迭代 ≠ 重做——是基于反思的**增量改进**
- **与 v33 goal-loop 关系**：迭代 = v33 goal-loop 第 5 段（Refine）
- **待用户拍板**：迭代触发条件（强制 vs 审计发现问题时）| 状态：**PENDING**

### 2.11 经验沉淀 + 模板维护（Experience Distillation + Template Maintenance）

- **用户原话**：#6 "也负责补充模块沉淀知识和经验"
- **用户原话**：#7 "经验沉淀，模板维护"
- **机制**：
  - **经验沉淀**：每角色每阶段产物中的"通用经验"提炼到 `module_templates/<MODULE>/P_game_specific.md`（或 `_experience.md`）
  - **模板维护**：每角色发现模板问题 → 写候选 → 人工审核入库（**候选制**，Agent 不直接入库）
  - **输入**：反思产物（§2.9）+ 审计产物（§2.8）
  - **输出**：模板维护候选清单
- **核心约束**：
  - **Agent 不得直接入库模板**（DESIGN_AND_EXECUTION_STANDARDS.mdc §0.1 铁律："Agent 不得直接写正式内容；只能写入候选区或候选文件"）
  - 经验沉淀必须**可追溯到具体产物文件 + 具体场景**
- **待用户拍板（Q-003、Q-005）**：
  - 经验沉淀落点：`P_game_specific.md` vs 新增 `_experience.md` vs 单独目录
  - 模板维护权限：候选制 vs 提案制
  - 状态：**PENDING**
