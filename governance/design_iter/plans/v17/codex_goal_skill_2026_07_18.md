# Codex Goal Skill 创建闭环记录

## 目标契约

- 目标：创建可通过 `/goal-loop` 显式调用的项目级 Skill，并用创建 Skill 的任务验证目标闭环。
- 范围：新增 `.cursor/skills/goal-loop/SKILL.md`；同步根入口的 Skill 数量；运行结构和语义验证。
- 非目标：不新增运行时代码，不修改 AIDocxWorkFlow 阶段契约，不创建传统 `.cursor/commands/` 兼容入口。
- 约束：遵守项目 Skill 规范、Git 分类规则、三文件改动上限；缺少验收标准或正确范例时推理补全继续。

## 验收标准

1. `goal-loop` 目录名与 frontmatter `name` 一致，可被显式调用且禁止环境自动触发。
2. Skill 固化“目标锁定→产出→自检论证→问题复盘→迭代修复→收敛判定”闭环。
3. 输入缺少 plan、验收标准或正确范例时，Skill 要求推理补全并继续。
4. Skill 能识别表面完成反模式，暂停主任务，创建、执行和关闭决策任务。
5. 只有全部验收标准有证据且无开放问题时才能判定收敛。
6. 项目 Skill 验证脚本通过，根入口 Skill 数量与实际一致，最终差异不超范围。

## 正确范例

用“创建 `/goal-loop` Skill”作为首个目标任务：创建后不能只凭格式验证宣布完成；必须触发一次表面完成反模式检查，执行语义挑战和差异审查，发现问题则复盘修复，证据闭合后才收敛。

## 反例

只创建 `SKILL.md`，未运行验证或仅以单个验证脚本通过为完成依据。

## 决策任务

### DT-001 防止测试通过即完成

- 触发反模式：只因测试通过就宣布目标完成。
- 触发证据：结构验证只能证明 frontmatter 和目录格式，不能证明六阶段语义、反模式打断、决策任务和收敛条件完整。
- 当前断点：结构验证完成后的收敛判定。
- 根因假设：把工具格式合规误当成目标行为合格。
- 候选行动：A. 结构验证通过后直接收敛；B. 增加语义断言和最终差异审查；C. 新增运行时代码模拟器。
- 选择与依据：选择 B。B 能验证行为契约且不引入超出用户目标的运行时代码；A 证据不足；C 过度实现。
- 执行结果：已运行全量 Skill 合规验证、15 项语义断言、错误范例拒绝挑战和 Skill 数量一致性检查；两次验证设计问题均进入复盘后修正。
- 验证证据：验证器扫描 13 个 Skill，结果为 `0 errors, 0 warnings`；语义断言输出 `PASS (15 clauses, anti-example rejected, 13 skills, 181 lines)`。
- 恢复点：决策任务已关闭，返回最终差异审查与收敛判定。

## 问题复盘

### R-001 验证器调用路径错误

- 预期：项目 Skill 验证器检查 `goal-loop/SKILL.md` 并返回合规结果。
- 实际：命令返回退出码 2，报告“在 `.cursor/skills/goal-loop` 下未发现任何 `*/SKILL.md`”。
- 症状：新 Skill 没有进入验证器扫描范围。
- 直接原因：把单个 Skill 目录传给要求“Skill 根目录”的验证器。
- 根因：未按 `.cursor/rules/SKILL_STANDARDS.md` §6 的调用方式使用 `.cursor/skills`。
- 影响范围：仅首轮验证命令；Skill 内容、目标契约和项目文件不受影响。
- 修复：改用 `python3 ai_workflow/validate_skills.py .cursor/skills`，随后重跑语义断言、反例挑战和差异检查。
- 回归风险：全量验证可能暴露既有 Skill 问题；若失败，必须区分新增回归与既有问题，不得隐藏。

### R-002 语义断言过度绑定文案

- 预期：语义断言证明错误范例能被识别并拒绝收敛。
- 实际：Skill 合规验证通过，但断言要求 Skill 逐字包含治理记录中的反例句，返回 `AssertionError: anti-example cannot be recognized`。
- 症状：语义等价的错误范例被误判为缺失。
- 直接原因：断言匹配了治理记录的完整句子，而 Skill 使用“只报告文件已创建”的具体场景表达同一反模式。
- 根因：验证条件与文案耦合，没有验证结构和行为语义。
- 影响范围：自定义语义挑战，不影响 Skill 合规结果。
- 修复：同时断言 `## 错误范例`、表面完成行为、缺失证据和“必须判定为未收敛”，不要求跨文件逐字相等。
- 回归风险：关键词断言不能替代运行时行为测试；本任务只创建指令型 Skill，因此再用正确范例的真实执行记录和最终差异审查补足证据。

### R-003 既有尾随空格干扰范围审查

- 预期：`git diff --check` 验证本轮三个目标文件没有空白错误。
- 实际：检查报告 `AGENTS.md:8: trailing whitespace`。
- 症状：全文件检查被不在本轮修改行的既有空格阻断。
- 直接原因：`git diff` 基于工作树当前全部未提交差异，而 `AGENTS.md` 在会话开始前已有用户修改。
- 根因：验证范围没有区分本轮新增行与用户既有未提交差异。
- 影响范围：`AGENTS.md` 第 8 行；与第 36 行 Skill 数量同步无关。
- 决策：不清理用户既有修改。验证两个本轮新增 Markdown 文件无行尾空格，并精确检查 `AGENTS.md` 第 36 行；同时用状态输出确认目标范围只有三份文件。
- 回归风险：无法声称整个 `AGENTS.md` 无空白问题，只能证明本轮修改行和新增文件符合验收范围。

### R-004 治理 current 符号链接漂移

- 预期：闭环记录写入当前治理方案目录并由 Git 显示为新文件。
- 实际：`governance/design_iter/current` 是 Git 跟踪的符号链接，物理目标仍是 `plans/v15`；Git 拒绝穿过符号链接的 pathspec。
- 症状：记录被写入已闭环的 v15，且逻辑路径不能直接用于 Git 归属检查。
- 直接原因：先依据逻辑路径创建记录，没有先解析符号链接物理目标。
- 根因：仓库 current 链接与正在工作的治理目录存在既有漂移。
- 影响范围：闭环记录的归属路径；Skill 和 `AGENTS.md` 不受影响。
- 修复：把记录迁移到 `governance/design_iter/plans/v17/`，删除 v15 误落文件；不在本任务中改动 current 链接或索引。
- 回归风险：current 链接漂移仍是独立治理问题，后续通过 current 写文件前必须先解析目标。

- 反向挑战：错误范例“只报告文件已创建”已被语义断言判定为未收敛。
