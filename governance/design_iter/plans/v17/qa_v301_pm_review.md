## 1. 关键业务流程覆盖判定

### P0 [P0 BLOCKER] 库存并发场景严重欠测

- **现状**：游戏道具商城核心场景是"道具库存"——并发超卖 / 库存扣减 / 库存告警 / 库存同步
- **当前覆盖**：仅 TC-025（"并发超卖时的回滚"）、TC-085（"数据库异常时的回滚"）共 2-4 TC
- **业务盲区**：
  1. 高并发场景下库存预占 vs 真实扣减的时序竞争（无 TC）
  2. 库存为 0 时购买按钮是否立即禁用（仅 TC-017 有 UI 验证）
  3. 库存告警阈值（如库存 < 10 触发警告）—— **未覆盖**
  4. 道具限购（如"每个账号限购 5 个"）—— **未覆盖**
  5. 跨 SKU 库存共享（如同一道具不同支付方式共用库存）—— **未覆盖**
- **业务影响**：库存并发是 P0 资损风险点，2-4 TC 不足以验证
- **修复方向**：S5 补 8-12 个 TP（库存预占 / 库存告警 / 限购 / 跨 SKU 共享）；S6 补 30-50 TC

### P1 [P0 BLOCKER] 风控 / 反作弊场景欠测

- **现状**：游戏道具商城是高价值虚拟资产交易——风控核心场景
- **当前覆盖**：仅 TC-082（"高频购买风控拦截"）、TC-083（"同 IP 多账号购买风控"）共 2 TC（SPECIAL 模块）
- **业务盲区**：
  1. 退款欺诈（"先退款再使用道具"）—— 仅 TC-080 部分覆盖
  2. 账号盗用 + 大额支付（异地登录后立即大额充值）—— **完全未覆盖**
  3. 批量注册账号 + 套利（自动化脚本批量购买 + 退款）—— **完全未覆盖**
  4. 支付渠道切换规避（同一订单切多个支付渠道逃单）—— **完全未覆盖**
- **业务影响**：游戏道具商城是黑产重灾区，2 TC 风控不足以应对实际风险
- **修复方向**：S5 补 5-8 个风控 TP；S6 补 20-30 TC（建议与安全工程师协作）

### P2 [P0 BLOCKER] 账号安全场景完全缺失

- **现状**：游戏道具商城核心是"玩家虚拟资产"——账号被盗 = 直接资损
- **当前覆盖**：**0 TC**
- **业务盲区**：
  1. 异地登录检测（IP 突变 + 设备指纹变化）
  2. 大额支付二次验证（短信 / 邮箱验证码）
  3. 实名认证（防未成年大额充值）
  4. 找回密码流程 + 安全问题
  5. 账号封禁 / 解封流程
  6. 登录失败锁定（密码错误 5 次锁定）
- **业务影响**：账号安全是 P0 资损风险，0 TC 是**结构性缺陷**
- **修复方向**：业务侧列账号安全清单（10-15 个场景）；S5 补 TP；S6 补 TC

### P3 [P1] 退款 30 天边界场景过度细分 + 数据冗余

- **现状**：30 天退款相关 TC 12 条，但实际只有 3 种业务场景（31 天拒绝 / 30 天内允许 / 刚好 30 天允许）
- **资深测试已识别 Q-025 重复**：12 TC 中 8 TC 重复（同一 test_scenario）
- **业务盲区**：
  1. 30 天退款的人工客服介入场景（VIP 用户可延长退款期）—— **未覆盖**
  2. 退款到账时效（支付宝实时 vs 银行卡 1-3 工作日）—— **未覆盖**
  3. 部分退款（购买 5 个道具退 3 个）—— **未覆盖**
- **业务影响**：
  - 重复 TC 浪费 QA 工时（资深测试已识别）
  - 部分退款是真实高频场景，缺测 = 真实业务漏洞
- **修复方向**：去重到 4 条 TC（31/60/90/边界 30 天），新增"部分退款""VIP 延长退款期""退款到账时效"3 个场景

## 2. 业务优先级排序判定

### P4 [P1] 网络异常 P1 优先级与业务风险不符

- **现状**：支付超时、网络异常等场景全部 P1（资深测试已识别 Q-015）
- **业务判定**：
  - 玩家支付超时（已扣款但未到账）= **资损 P0**（用户投诉主因）
  - 道具列表加载超时 = P1（体验问题）
- **当前错误**：所有网络异常统一 P1，未区分"已扣款超时"（P0）和"页面加载超时"（P1）
- **修复方向**：normalizer 加"网络异常 → 按场景细分优先级"业务规则

### P5 [P1] VIP 订单筛选 P2 优先级与业务频次不符

- **现状**：TC-060/061/062（订单筛选）全部 P2（资深测试已识别 Q-016）
- **业务判定**：VIP 用户是付费核心人群，订单筛选是高频操作（每周 3-5 次）
- **修复方向**：VIP 用户的订单筛选应升 P1（业务频次决定优先级，非风险等级）

### P6 [P2] 缺 P3 分级导致"体验性"用例挤占 P0/P1

- **现状**：331 TC 全部分到 P0/P1/P2（资深测试已识别 Q-014）
- **业务判定**：游戏道具商城有以下"体验性"用例应归 P3：
  - 商城首页轮播图
  - 道具图片加载占位符
  - 页面字体 / 颜色一致性
  - 移动端适配
- **影响**：当前 185 条 P1 中估计有 30-50 条可降 P3（让 QA 资源聚焦核心）
- **修复方向**：业务侧确认"体验性"用例清单，加 P3 分级

## 3. 业务边界 / 异常场景覆盖判定

### P7 [P0 BLOCKER] 国际化 / 多语言场景完全缺失

- **现状**：游戏道具商城通常面向多地区玩家（中/英/日/韩/繁中等）
- **当前覆盖**：**0 TC**
- **业务盲区**：
  1. 多语言文案长度差异（德语长 30% 导致 UI 截断）
  2. 货币符号 / 小数点（欧美用 . 亚洲用 ，）
  3. 时区差异（活动结束时间按 UTC 还是本地时区）
  4. 支付方式地域差异（国内微信/支付宝，海外信用卡/PayPal）
- **业务影响**：若 v3.01 面向海外（业务侧确认），则 0 国际化 TC 是 P0 缺陷
- **修复方向**：业务侧确认目标地域；若是国内版本，可在 follow-up 处理；若是海外版本，P0 补 20-30 TC

### P8 [P0 BLOCKER] 性能 / 压力场景完全缺失

- **现状**：游戏道具商城大促期间（双 11 / 周年庆）瞬时高并发
- **当前覆盖**：仅 TC-024（"道具 1 秒内到账"）隐含性能，但无明确性能 TC
- **业务盲区**：
  1. 双 11 峰值并发（10 万 QPS）下系统稳定性
  2. 库存扣减数据库行锁竞争
  3. 支付回调高并发堆积
  4. 大促期间退款高峰
- **业务影响**：游戏道具商城大促是核心营收场景，无性能 TC 是 P0 缺陷
- **修复方向**：与 SRE / DBA 协作，S5 补 5-8 个性能 TP；S6 补 TC（性能 TC 通常在 S9 性能阶段独立做）

### P9 [P1] 退款到账时效场景未覆盖

- **现状**：游戏道具退款是高频场景，但退款到账时效是用户投诉主因
- **业务盲区**：
  1. 支付宝退款实时到账
  2. 银行卡退款 1-3 工作日到账
  3. 退款失败重试机制
  4. 退款到账短信通知
- **修复方向**：业务侧确认退款渠道清单；S5 补 3-5 个 TP

### P10 [P1] 道具使用 / 消耗场景未覆盖

- **现状**：S2 backlog 中 Story 主要是"购买 / 退款 / 库存管理"，但"购买后的道具使用"未覆盖
- **业务盲区**：
  1. 道具使用后是否影响退款（部分退款逻辑）
  2. 限时道具过期处理
  3. 道具叠加规则
- **影响**：影响"已使用的道具退款"测试（TC-080 涉及但不完整）
- **修复方向**：业务侧列"道具使用"场景清单

### 6.2 资深产品判定结论

- **核心业务（购买/支付/退款）覆盖合格**：52+32+36 = 120 TC，关键路径 P0 优先级合理
- **三类关键业务完全缺失**：账号安全（0 TC）/ 国际化（0 TC）/ 性能（0 TC）——结构性缺陷
- **数据冗余与盲区并存**：30 天退款重复 8 TC 同时缺部分退款；VIP 过度细分同时缺等级升降级
- **优先级分类与业务频次不匹配**：网络异常全 P1 / VIP 订单筛选 P2 / 体验性用例挤占 P0/P1
- **总评**：v3.01 在"已识别业务"上覆盖合格（3.5/5），但"未识别业务"（账号安全 / 国际化 / 性能）是 P0 结构性缺失——**3.0/5**

### 6.3 资深产品移交清单（业务侧补全）

1. **账号安全场景清单**：业务侧列 10-15 个场景（资深产品 + 安全工程师协作）

4. **性能场景**：业务侧确认是否有大促 / 高峰时段；若是，与 SRE 协作补性能 TC
5. **库存告警规则清单**：业务侧列告警阈值 + 通知规则
6. **VIP 等级升降级规则**：业务侧列完整规则矩阵
7. **限购规则清单**：业务侧列道具限购维度（账号/IP/设备）
8. **优先级业务规则**：业务侧确认"支付超时 P0 / VIP 订单筛选 P1 / 体验性用例 P3"分类
