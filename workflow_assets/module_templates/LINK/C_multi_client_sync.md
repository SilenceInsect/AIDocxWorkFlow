# C. 多端一致性关联

> **子类代码**：`MULTI_CLIENT_SYNC`
> **归属模块**：`LINK`
> **来源**：用户细化定义 §三-3「多端一致性关联」（原"多端一致性"扩展 + `MULTI_TENANT_SYNC` 改名）
>
> **测什么**：同一玩家账号在**多个客户端**（PC/移动端/模拟器/Web）登录时，**端到端**的数据同步、登录冲突约束、客户端本地缓存与服务端真实数据的差值刷新与兜底校正。
> **不测什么**：单端内部的本地存储（归 AUX）、跨服务端的数据同步（归 B）、跨端 UI 适配（归 UI）。
> **与其他子类的差异**：C 关注"**多端之间**"的对齐——A 关注"**业务系统之间**"的联动、B 关注"**服务之间**"的同步、D 关注"**外部平台之间**"的对接。

> **命名说明**：v1.1 旧称 `MULTI_TENANT_SYNC`（多租户）改为 v1.2 `MULTI_CLIENT_SYNC`（多端）——语义从 B2B 多租户纠正为 B2C 游戏多端。

---

## 1. 典型场景

### 场景 1：PC 端操作移动端实时同步
- 业务背景：玩家在 PC 端购买道具，移动端同时在线
- 涉及端：PC 客户端、移动客户端、服务端
- 触发动作：PC 端点击"购买"
- 验证点：移动端在 1s 内看到新道具、货币余额同步减少、UI 飘字同步

### 场景 2：移动端操作 PC 端实时同步
- 业务背景：玩家在移动端完成任务，PC 端同时在线
- 涉及端：移动客户端、PC 客户端、服务端
- 触发动作：移动端完成任务 X
- 验证点：PC 端任务列表立即更新为"已完成"、奖励已发放、跨端状态一致

### 场景 3：模拟器与移动端实时同步
- 业务背景：玩家在 PC 模拟器和真实手机上同时登录
- 涉及端：模拟器客户端、移动客户端、服务端
- 触发动作：任一端操作
- 验证点：操作实时同步到另一端、跨端操作响应延迟 < 1s

### 场景 4：多端同时登录冲突
- 业务背景：玩家 A 端登录后未退出，B 端也尝试登录
- 涉及端：A 端、B 端、服务端
- 触发动作：B 端登录
- 验证点：A 端被顶号提示、B 端进入游戏、顶号后数据落地一致性

### 场景 5：顶号后数据落地一致性
- 业务背景：A 端操作到一半被 B 端顶号
- 涉及端：A 端、B 端、服务端
- 触发动作：A 端在"购买中"被顶号
- 验证点：A 端的"购买中"状态是否提交到服务端（如未提交则回滚）、B 端登录后看到正确数据

### 场景 6：多端操作互斥锁
- 业务背景：玩家在 A 端进入副本，B 端也想进副本
- 涉及端：A 端、B 端、服务端
- 触发动作：A 端进副本后 B 端进副本
- 验证点：B 端提示"该账号已在副本中"、互斥锁生效

### 场景 7：本地缓存与服务端真实数据不一致
- 业务背景：玩家在 PC 端断网 5 分钟后重连
- 涉及端：PC 端、服务端
- 触发动作：PC 端断网期间移动端有操作、PC 重连
- 验证点：PC 端本地缓存与服务端真实数据差值刷新、PC 显示最新数据

### 场景 8：差值刷新
- 业务背景：服务端推送"差值更新"给客户端
- 涉及端：客户端、服务端
- 触发动作：服务端推送"玩家 X 获得道具 A x 10"
- **expected**：客户端只更新该差值字段、不重新拉全量
- **notes**：注意"差值 vs 全量"——差值性能更好但需保证数据一致性

### 场景 9：兜底校正逻辑
- 业务背景：差值推送丢失后客户端数据错乱
- 涉及端：客户端、服务端
- 触发动作：客户端检测到本地数据可疑
- 验证点：客户端主动拉取服务端"全量数据"进行兜底校正

### 场景 10：活动状态多端同步
- 业务背景：玩家 A 端进入限时活动后 B 端登录
- 涉及端：A 端、B 端、服务端
- 触发动作：B 端登录
- 验证点：B 端显示玩家已在该活动中、活动进度与 A 端一致

### 场景 11：邮件多端同步
- 业务背景：A 端收到新邮件，B 端在线
- 涉及端：A 端、B 端、服务端
- 触发动作：A 端收到新邮件
- 验证点：B 端邮件列表立即出现该邮件、未读/已读状态跨端同步

### 场景 12：充值记录多端同步
- 业务背景：A 端充值 100 元，B 端在线
- 涉及端：A 端、B 端、服务端
- 触发动作：A 端完成充值
- 验证点：B 端充值记录立即出现、累充进度同步更新、奖励邮件跨端可见

### 场景 13：离线操作同步
- 业务背景：A 端离线操作、B 端在线
- 涉及端：A 端、B 端、服务端
- 触发动作：A 端断网期间操作、A 重连
- 验证点：断网期间操作排队、A 重连后批量上报、B 端在 A 上报后看到更新

## 2. 种子测试点（TP 模板）

### TP-001（MULTI_CLIENT_SYNC）：PC 端购买道具移动端同步
- **scenario**：场景 1
- **module**：`MULTI_CLIENT_SYNC`
- **precondition**：玩家 PC 端登录 + 移动端登录（同一账号）、货币 1000
- **test_data**：PC 端购买道具（价格 100）
- **expected**：移动端 1s 内看到新道具、货币变为 900、UI 飘字同步
- **notes**：注意"实时同步延迟"——网络抖动时可接受 1-3s

### TP-002（MULTI_CLIENT_SYNC）：移动端完成任务 PC 端同步

- **precondition**：玩家双端在线、任务"日常 X"未完成
- **test_data**：移动端完成任务 X
- **expected**：PC 端任务列表立即更新为"已完成"、奖励已发放、跨端状态一致
- **notes**：注意"任务完成"是 BIZ，但"跨端同步"是 C

### TP-003（MULTI_CLIENT_SYNC）：模拟器与移动端同步

- **expected**：移动端 1s 内看到模拟器玩家移动、跨端操作响应延迟 < 1s

## 3. 边界陷阱

### 边界 1：vs B 跨服务
- **混淆点**："全服排行榜"听起来像 C（多端），但"全服聚合"是 B（跨服务）
- **判定规则**：测"多端客户端之间的同步" → C；测"服务端跨服数据" → B
- **实例**：排行榜在 PC 和移动端显示一致 → C；聚合服从各逻辑服拉数据 → B

### 边界 2：vs A 内部业务
- **混淆点**："充值"听起来像 A，但"充值跨端同步"是 C
- **判定规则**：测"业务联动规则" → A；测"多端同步" → C
- **实例**：充值触发累充发奖 → A；A 端充值后 B 端看到充值记录 → C

### 边界 3：vs AUX 底层
- **混淆点**："本地缓存"听起来像 C，但缓存实现是 AUX
- **判定规则**：测"业务互通规则（差值刷新、兜底校正）" → C；测"底层能力（缓存实现）" → AUX
- **实例**：本地缓存与服务端差值刷新业务规则 → C；本地缓存的 LRU 淘汰算法 → AUX

### 边界 4：vs UI 控件基础
- **混淆点**："多端 UI 显示一致"听起来像 C，但 UI 控件适配是 UI 模块
- **判定规则**：测"多端数据一致性" → C；测"多端 UI 控件适配（不同分辨率/不同主题）" → UI
- **实例**：多端邮件红点状态同步 → C；移动端分辨率下的邮件红点大小适配 → UI

## 4. 验证证据

### 视觉证据
- PC 端购买后移动端同步显示截图
- 多端同时在线时操作同步截图
- 顶号提示截图
- 断网重连后本地缓存与服务端数据对齐截图

### 日志证据
- `<LOG_MULTI_CLIENT_SYNC>`：多端同步事件，source_endpoint, target_endpoint, latency
- `<LOG_TOP_ACCOUNT>`：顶号事件
- `<LOG_DIFF_REFRESH>`：差值刷新
- `<LOG_FALLBACK_PULL>`：兜底拉取
- `<LOG_OFFLINE_QUEUE>`：离线操作排队

### 数据证据
- 数据库 `player_endpoint_state` 表：每端玩家状态（最后在线、操作时间）
- 数据库 `sync_diff` 表：差值推送记录
- 数据库 `top_account_log` 表：顶号日志
- 数据库 `offline_op_queue` 表：离线操作队列

### 性能证据
- 多端同步延迟 < 1s（同区）、< 3s（跨区）
- 顶号响应延迟 < 500ms
- 差值刷新带宽占用 < 10KB/次
- 离线操作批量上报 < 5s（100 个操作）
