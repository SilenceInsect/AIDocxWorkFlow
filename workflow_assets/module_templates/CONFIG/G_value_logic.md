# G. 数值逻辑校验

> **子类代码**：`VALUE_LOGIC`
> **归属模块**：`CONFIG`
> **来源**：用户细化定义 §5(3)「数值逻辑校验（游戏配置核心）」
>
> **测什么**：概率配置总和、计算公式、上限约束、货币产出消耗平衡。
> **不测什么**：单字段合法性（归 A）、跨表 ID 引用（归 C）、热更（归 D）、导出（归 H）。
> **与其他子类的差异**：G 关注"多字段间的数学/逻辑关系"——A 关注"单字段"，B 关注"同表/全表一致"。

---

## 1. 典型场景

### 场景 1：概率总和
- 业务背景：抽卡权重
- 涉及字段：5 个道具的抽中概率
- 触发动作：5 个概率分别为 30% + 30% + 20% + 20% + 10% = 110%
- 验证点：概率总和 ≤ 100%

### 场景 2：抽卡权重分配
- 业务背景：卡池配置
- 涉及字段：SSR/SR/R/N 权重
- 触发动作：SSR 50%, SR 30%, R 15%, N 5%
- 验证点：权重无逻辑错误（高等级 ≥ 低等级）

### 场景 3：保底配置
- 业务背景：抽卡保底
- 涉及字段：保底次数 90
- 触发动作：玩家抽 90 次未中 SSR
- 验证点：第 90 次必中 SSR

### 场景 4：成长公式
- 业务背景：等级属性
- 涉及公式：HP = base_hp × (1 + level × growth_rate)
- 触发动作：录入基础值和成长率
- 验证点：按公式计算结果正确

### 场景 5：伤害系数
- 业务背景：技能伤害
- 涉及公式：damage = atk × skill_rate × crit_rate
- 触发动作：录入攻击、技能系数、暴击
- 验证点：伤害按公式计算

### 场景 6：背包容量上限
- 业务背景：背包
- 涉及字段：背包最大容量 100
- 触发动作：玩家获得 101 个道具
- 验证点：超出上限不堆叠、提示"背包已满"

### 场景 7：堆叠上限
- 业务背景：道具
- 涉及字段：单道具堆叠上限 99
- 触发动作：玩家获得 100 个同一道具
- 验证点：自动开新格位、不超过 99/格

### 场景 8：每日获取上限
- 业务背景：日常副本
- 涉及字段：每日最多 100 金币
- 触发动作：玩家今日已获 100 金币，继续刷副本
- 验证点：不再获得金币

### 场景 9：任务产出平衡
- 业务背景：日常任务
- 涉及字段：每个任务奖励
- 触发动作：完成 10 个日常任务
- 验证点：10 个任务奖励 = 每日总收入

### 场景 10：商城售价
- 业务背景：商城
- 涉及字段：商品价格
- 触发动作：玩家购买
- 验证点：扣款正确、发货正确

## 3. 边界陷阱

### 边界 1：vs A. 字段合法性
- **混淆点**：「概率字段 0-100"」——A 测范围、G 测总和
- **判定规则**：测"单字段范围" → A；测"多字段总和/公式" → G
- **实例**：单个 drop_rate = 150 → A-006；所有 drop_rate 总和 = 120% → G-001

### 边界 2：vs B. 同表一致性
- **混淆点**：「公式"一致"」——B 测同表数值一致、G 测公式逻辑
- **判定规则**：测"同表多字段值统一" → B；测"字段间公式/概率关系" → G
- **实例**：5 处基础倍率值相同 → B-005；5 个抽卡概率总和 110% → G-001

### 边界 3：vs K. 游戏专项
- **混淆点**：「礼包"限购"」——G 测数值逻辑、K 测游戏专项
- **判定规则**：测"概率/公式/上限的数学关系" → G；测"礼包限购次数/活动过期" → K
- **实例**：抽卡保底 90 次 → G-003；礼包限购 1 次 → K-005

### 边界 4：vs BIZ
- **混淆点**：「货币"逻辑"」——G 测配置层、BIZ 测业务逻辑
- **判定规则**：测"配置层的数值公式" → G；测"业务层的货币扣减/发放" → BIZ
- **实例**：商城定价 = 100 钻石 → G-007；购买时扣 100 钻石 → BIZ
