## OBJ-001: 购买流程

**所属Story**: BIZ-PURCHASE
**主模块**: BIZ
**辅助模块**: UI, AUX, LINK

### 对象状态拆解

| 状态名 | 触发条件 | 结果 |
|--------|----------|------|
| 待确认 | 玩家点击购买，进入确认弹窗 | 展示道具名称+数量+总价 |
| 余额不足 | 玩家余额 < 所需金额 | 购买按钮禁用，显示提示 |
| 数量超限 | 购买数量 > 99 | 前端拦截，提示数量超限 |
| 支付中 | 玩家点击确认，扣款发起 | 显示支付中状态 |
| 支付成功 | 扣款完成，道具到账 | 道具立即到账，邮件发送 |
| 支付失败 | 扣款失败（网络/余额/风控） | 余额回滚，显示失败原因 |
| 重复提交防护 | 支付中状态下再次点击 | 前端禁用按钮，防重复 |

### 交互形态

- **界面操作**: 玩家在道具详情页点击购买按钮 → 弹出购买确认弹窗 → 选择/输入数量 → 点击确认支付
- **API调用**: `POST /api/purchase/confirm` → `POST /api/payment/deduct` → `POST /api/item/deliver`
- **系统事件**: 支付成功事件触发道具到账、邮件发送、订单创建

### 业务逻辑

**输入**: 玩家ID，道具ID，购买数量，支付方式
**处理规则**:
1. 校验玩家登录状态
2. 校验道具可购买状态（已上架、未被禁用）
3. 校验玩家余额（余额充足才可继续）
4. 锁定汇率（以订单创建时刻汇率为准）
5. 计算最终价格（含折扣/满减）
6. 发起扣款（余额冻结 → 实际扣减）
7. 道具到账（立即写入玩家背包）
8. 创建订单记录
9. 异步发送邮件通知

**输出**: 购买结果（成功/失败），订单号，道具到账确认

### 异常业务边界

| 边界场景 | 处理方式 |
|---------|---------|
| 道具已下架 | 拒绝购买，提示「道具已下架」 |
| 玩家余额刚好等于所需金额 | 允许购买，扣款后余额归零 |
| 并发购买同一限量道具 | 超卖防护，最后购买者成功，其余退款 |
| 购买数量为0或负数 | 前端拦截，后端兜底拒绝 |
| 游戏币扣款后到账失败 | 回滚扣款，订单标记失败 |

### 异常交互

| 异常类型 | 处理方式 |
|---------|---------|
| 支付超时（>500ms） | 返回超时错误，提示重试 |
| 网络断开 | 前端检测断网，显示网络异常，禁用支付按钮 |
| 支付回调丢失 | 后端轮询订单状态，5分钟内未确认则退款 |
| 渠道支付失败 | 显示渠道返回的错误信息，引导重试 |

- **测试类型**: 正向 + 边界
- **前置条件**: 玩家已登录，在道具详情页
- **输入**: 道具ID

- **分类边界**: 余额=0/余额<价格/余额=价格/余额>价格
- **测试类型**: 边界
- **前置条件**: 玩家余额不足
- **输入**: 道具价格，玩家余额
- **预期输出**: 购买按钮禁用，显示「余额不足」提示
- **优先级**: P0

#### OBJ-001-FP-3: 数量范围校验（1-99）
- **分类维度**: 数量校验
- **分类边界**: 数量=0/数量=1/数量=99/数量=100/数量=-1/数量=999
- **测试类型**: 边界 + 异常
- **前置条件**: 玩家在数量选择器
- **输入**: 选择的数量
- **预期输出**: 1-99通过，超出前端拦截或后端拒绝
- **优先级**: P1

#### OBJ-001-FP-4: 道具到账时效
- **分类维度**: 到账时效
- **分类边界**: 到账时间<1秒/到账时间<3秒/到账时间>3秒（超时）
- **测试类型**: 正向 + 异常
- **前置条件**: 支付成功
- **输入**: 订单号
- **预期输出**: 3秒内道具到账，背包数据刷新
- **优先级**: P0

#### OBJ-001-FP-5: 超时与重试
- **分类维度**: 异常处理
- **分类边界**: 超时时间=500ms/超时时间>500ms/重试1次/重试3次/幂等
- **测试类型**: 异常
- **前置条件**: 支付接口调用超时
- **输入**: 订单号，重试次数
- **预期输出**: 超时返回错误，重试成功则完成，重试耗尽则退款
- **优先级**: P1

#### OBJ-001-FP-6: 支付幂等性
- **分类维度**: 幂等校验
- **分类边界**: 同一订单号重复支付1次/重复支付3次/不同订单号相同金额
- **测试类型**: 异常
- **前置条件**: 订单已支付成功
- **输入**: 已支付订单号
- **预期输出**: 重复支付被拒绝，返回「订单已支付」
- **优先级**: P0

#### OBJ-001-FP-7: 并发超卖防护
- **分类维度**: 并发控制
- **分类边界**: 单用户顺序购买/多用户并发购买限量道具/超卖临界点
- **测试类型**: 异常
- **前置条件**: 限量道具剩余1件，10个用户同时抢购
- **输入**: 10个并发请求
- **预期输出**: 仅1个成功，其余退款或提示已售罄
- **优先级**: P1

#### OBJ-001-FP-8: 邮件通知字段完整性
- **分类维度**: 邮件通知
- **分类边界**: 邮件所有字段完整/邮件某字段缺失/邮件发送失败
- **测试类型**: 正向 + 异常
- **前置条件**: 购买成功
- **输入**: 订单信息
- **预期输出**: 邮件包含：玩家名称、道具名称、购买数量、金额（含币种）、购买时间、订单号
- **优先级**: P1

## OBJ-002: 道具数据

**所属Story**: UI-SHOP
**主模块**: AUX
**辅助模块**: UI, CONFIG

### 对象状态拆解

| 状态名 | 触发条件 | 结果 |
|--------|----------|------|
| 正常展示 | 道具已上架，未过期 | 正常展示名称、图标、价格 |
| 折扣中 | 限时折扣生效中 | 展示折扣标签+划线原价+折后价 |
| 已下架 | 道具被管理员下架 | 不在列表中展示 |
| 缓存中 | 数据来自5分钟缓存 | 无网络请求，直接返回 |
| 缓存失效 | 缓存超过5分钟或配置变更 | 重新从配置表加载 |
| VIP专属 | 道具标记为VIP专属 | 仅对VIP用户展示 |

### 交互形态

- **界面操作**: 玩家浏览商城首页 → 点击分类/搜索 → 查看道具列表 → 点击道具进入详情页
- **API调用**: `GET /api/items`（道具列表）、`GET /api/items/{id}`（道具详情）
- **系统事件**: 配置变更事件触发缓存失效，hot标签变更触发首页刷新

### 业务逻辑

**输入**: 玩家ID，分类ID，页码，搜索关键词
**处理规则**:
1. 检查道具缓存是否存在且未过期
2. 缓存命中则直接返回，缓存未命中则查询配置表
3. 按分类/搜索条件筛选
4. 应用VIP专属道具可见性过滤
5. 应用折扣标签（限时折扣+VIP折扣计算后价格）
6. 分页返回（每页20个）
7. 计算热门道具（hot标签+优先级排序）

**输出**: 道具列表（名称、图标、价格含折扣、折扣标签、分页信息）

### 异常业务边界

| 边界场景 | 处理方式 |
|---------|---------|
| 道具数量<20（最后一页） | 正常返回，不足20个 |
| 搜索关键词=1个字符 | 前端拦截，返回「至少输入2个字符」 |
| 搜索结果为空 | 返回空列表，展示空状态 |
| 热门道具=0个 | 推荐区隐藏或不展示 |
| 热门道具>10个 | 截取前10个 |

### 异常交互

| 异常类型 | 处理方式 |
|---------|---------|
| 道具数据缓存失效时大量请求 | 限流/排队加载，防止击穿 |
| 配置表读取超时 | 返回缓存旧数据（若缓存存在），否则返回错误 |
| hot标签变更后前端未刷新 | 强制刷新或展示过期提示 |

### 关联模块推理

| 模块 | 角色 |
|------|------|
| UI（主） | 道具列表渲染、折扣标签展示 |
| AUX（主） | 道具数据缓存、配置表读取 |
| CONFIG | hot标签配置、VIP折扣比例 |

### 日志与埋点

| 事件名 | 触发时机 | 字段列表 |
|--------|----------|---------|
| `item.viewed` | 玩家查看道具详情 | player_id, item_id, timestamp |
| `item.search` | 玩家搜索道具 | player_id, keyword, result_count |
| `cache.hit` | 道具缓存命中 | item_id, cache_age_seconds |
| `cache.miss` | 道具缓存未命中 | item_id, reload_reason |
| `hot_tag.changed` | hot标签变更 | item_id, old_tag, new_tag, operator |

### 关联协议

| 协议名 | 调用方 | 接收方 | 关键字段 |
|--------|--------|--------|---------|

| `ItemService.GetDetail` | 前端 | 后端道具服务 | item_id, player_id |
| `ConfigService.GetItemConfig` | 道具服务 | 配置服务 | item_id |
| `CacheService.Get` | 道具服务 | 缓存服务 | cache_key |

## OBJ-003: VIP配置

**所属Story**: CONFIG-VIP
**主模块**: CONFIG
**辅助模块**: BIZ

### 对象状态拆解

| 状态名 | 触发条件 | 结果 |
|--------|----------|------|
| 永久有效 | VIP等级无时效配置 | 持续有效，不自动降级 |
| 定时考核 | VIP等级有时效配置，到期后判定 | 按配置决定是否降级/保留 |
| 变更中 | 策划修改配置表 | 新配置实时生效 |
| 等级不足 | 玩家等级 < 道具要求等级 | 专属道具不可见/不可购买 |

### 交互形态

- **界面操作**: 策划登录配置后台 → 编辑VIP等级规则 → 保存配置
- **API调用**: `GET /api/vip/config`（读取）、`POST /api/vip/config`（更新，策划）
- **系统事件**: VIP配置变更事件 → 下发至全服玩家 → 玩家登录时拉取最新配置

### 业务逻辑

**输入**: VIP等级配置数据（等级数、折扣比例、解锁条件、时效配置）
**处理规则**:
1. 策划通过配置后台编辑VIP规则
2. 配置保存后触发全服生效事件
3. 玩家登录时拉取最新VIP配置
4. 购买时根据玩家当前VIP等级计算折扣
5. VIP等级变更时（升级/降级），下次购买即时生效

**输出**: VIP配置生效，玩家享受对应折扣

### 异常业务边界

| 边界场景 | 处理方式 |
|---------|---------|
| VIP等级配置缺失（未配置） | 使用默认折扣（不打折） |
| 折扣比例超出0-100%范围 | 前端拦截，后端拒绝保存 |
| 解锁条件配置为空 | 该等级不可解锁 |
| 玩家同时满足多个解锁条件 | 以第一个匹配的条件为准 |

### 异常交互

| 异常类型 | 处理方式 |
|---------|---------|
| 配置保存失败 | 提示保存失败，配置不更新 |
| 配置变更时大量玩家在线 | 异步下发，不阻塞主流程 |
| 玩家在购买中途VIP升级 | 本次订单以购买时等级为准 |

### 关联模块推理

| 模块 | 角色 |
|------|------|
| CONFIG（主） | VIP规则配置管理 |
| BIZ（主） | VIP折扣计算逻辑 |
| UI | 配置后台编辑界面 |

### 日志与埋点

| 事件名 | 触发时机 | 字段列表 |
|--------|----------|---------|
| `vip.config.updated` | 策划更新VIP配置 | operator, vip_level, old_config, new_config, timestamp |
| `vip.player.upgraded` | 玩家VIP等级升级 | player_id, old_level, new_level, upgrade_reason |
| `vip.player.downgraded` | 玩家VIP等级降级 | player_id, old_level, new_level, downgrade_reason |
| `vip.discount.applied` | VIP折扣应用 | player_id, vip_level, item_id, original_price, discounted_price |

### 关联协议

## OBJ-006: 折扣计算

**所属Story**: CONFIG-DISCOUNT, BIZ-PURCHASE
**主模块**: BIZ
**辅助模块**: CONFIG

### 对象状态拆解

| 状态名 | 触发条件 | 结果 |
|--------|----------|------|
| 原价 | 无任何折扣 | 显示原价 |
| 限时折扣中 | 限时促销生效 | 折扣后价格 |
| VIP折扣中 | 玩家为VIP | VIP折扣后价格 |
| 多重折扣 | 限时+VIP同时生效 | 取最优价 |

### 交互形态

- **API调用**: 购买服务调用折扣计算服务
- **系统事件**: 折扣计算完成后返回结果，用于确认弹窗和结算

### 业务逻辑

**输入**: 原价，道具ID，玩家ID，当前时间
**处理规则**:
1. 查询是否有生效的限时折扣
2. 查询玩家VIP等级及VIP折扣比例
3. 计算限时折扣后价格 = 原价 × 限时折扣比例
4. 计算VIP折扣后价格 = 原价 × VIP折扣比例
5. 取最优价 = min(限时折后价, VIP折后价) [待确认]
6. 满减在折扣后执行
7. 返回最终价格

**输出**: 最终结算价格，各折扣明细

### 异常业务边界

| 边界场景 | 处理方式 |
|---------|---------|
| 折扣比例=0（免费） | 允许，结果为0 |
| 折扣后价格<0 | 最低为0，不出现负数 |
| 角分四舍五入后不一致 | 以分为最小单位，四舍五入 |
