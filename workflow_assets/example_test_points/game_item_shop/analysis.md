# 拆解思路分析 — 游戏道具商城系统 v1.0

> **本文件**：从"为什么这么拆"的角度分析游戏道具商城系统的 75 个 TP 产出
> **目的**：帮助 S5 LLM 学习"拆解思路"——遇到新需求时如何思考、如何归类、如何避免误标

---

## 1. 拆解哲学

### 1.1 核心原则（去硬数字 + 按业务自然划分）

| 原则 | 含义 | 反例 |
|------|------|------|
| **按业务自然划分** | 每个 Story 的 TP 数 = 该 Story 的业务复杂度 | ❌ 强制 1:1:1:1 凑 4 个 |
| **跨模块多 TP** | 同 Story 命中多模块 → 每个模块单独生成 TP | ❌ "取主"只标 1 个 |
| **EXCEPTION 必引 S4** | EXCEPTION TP 的 s4_reference 字段必填 | ❌ 凭空生成异常场景 |
| **类型数量 = 业务需要** | 业务流多的 Story TP 多，业务流少的 Story TP 少 | ❌ 强制每 Story 6-8 个 |

### 1.2 数量分布（按业务自然产出的结果）

| Story 类型 | TP 数 | 典型场景 |
|-----------|-------|---------|
| **简单 Story** | 6-7 | 单一操作（如购买确认、道具详情）|
| **中 Story** | 7-8 | 多流程组合（如游戏币支付含下单+扣款+到账+通知）|
| **复杂 Story** | 7-8 | 涉及配置/异常（如 VIP 等级配置含保存+校验+热更+异常）|
| **极简 Story** | 1-2 | 微型修改（v2.0 才有 1-2 个）|

> 75 TP / 26 Story ≈ **平均 2.9 个 TP/Story**（不是 1:1:1:1）

## 2. 模块 × 类型双维度判定实战

### 2.1 案例 1：`BIZ-PURCHASE-002`（游戏币支付）— 8 个 TP

**Story 描述**：玩家使用游戏币支付购买道具

**8 模块判定**：
- ✅ **BIZ**：核心业务（扣款+到账+订单状态机）
- ✅ **HINT**：道具到账提示（飘字+红点+系统消息）
- ✅ **LOG**：资产变更埋点 + 异常日志
- ✅ **UI**：购买按钮 + 弹窗
- ❌ CONFIG / LINK / SPECIAL：未涉及

**4 类型分布**（按业务需要）：
- POSITIVE × 2（TP-001 支付成功 / TP-002 到账提示）
- BOUNDARY × 2（TP-003 余额刚好 / TP-004 余额差 1）
- NEGATIVE × 3（TP-005 扣款失败 / TP-007 余额差 1 重写 / TP-008 扣款金额为负）
- EXCEPTION × 1（TP-006 扣款成功但到账失败）

**总 TP = 8**（**不是** 4 类型的 1:1:1:1 = 4 个）

### 2.2 案例 2：`UI-SHOP-003`（搜索功能）— 8 个 TP

**Story 描述**：商城搜索道具

**8 模块判定**：
- ✅ **UI**：搜索框 + 结果列表 + 搜索触发
- ✅ **BIZ**：搜索接口 + SQL 防护
- ✅ **HINT**：无搜索结果提示（`SYS_MSG` → v1.7 升级为 `MODAL_DIALOG`）
- ❌ CONFIG / LINK / SPECIAL / LOG：未涉及

**4 类型分布**：
- POSITIVE × 2（TP-001 搜索触发 / TP-002 结果展示）
- BOUNDARY × 2（TP-003 2 字符 / TP-004 1 字符不触发）
- NEGATIVE × 3（TP-005 无结果 / TP-007 Unicode 字符 / TP-008 超 1000 条截断）
- EXCEPTION × 1（TP-006 SQL 注入防护）

**总 TP = 8**

### 2.3 案例 3：`SPECIAL-VIP-CHANGE-001`（VIP 等级变更）— 7 个 TP

**Story 描述**：VIP 玩家升降级时折扣生效

**8 模块判定**：
- ✅ **CONFIG**：配置表 + 折扣规则
- ✅ **BIZ**：等级变更服务 + 业务处理
- ✅ **SPECIAL**：状态机异常 + 批量处理
- ❌ UI / HINT / LINK / LOG / AUX：未涉及

**4 类型分布**：
- POSITIVE × 2（TP-001 升级生效 / TP-002 购买中等升级）
- BOUNDARY × 2（TP-003 配置规则变更 / TP-004 批量 1 万人升级）
- NEGATIVE × 2（TP-005 通知失败 / TP-007 购买中等变更）
- EXCEPTION × 1（TP-006 降级后仍享旧折扣）

**总 TP = 7**

## 3. HINT vs UI 二次判定实战（**S5 误标高发区**）

### 3.1 案例：`BIZ-PURCHASE-002`（游戏币支付）的 HINT TP

| TP ID | title | module | **正确判定** |
|-------|-------|--------|-------------|
| TP-002 | 道具到账提示 — 背包红点+飘字 | HINT | ✅ 临时弹出 = HINT |
| TP-001 | 游戏币支付成功 — 扣款+道具到账 | BIZ | ✅ 业务流 = BIZ |

> **错误判定**（实际未发生）："道具到账提示" 标 UI（看到红点 + 飘字就标 UI）→ 误标
> **正确判定**：HINT（红点 + 飘字是**事件触发**临时反馈）

### 3.2 案例：`UI-DETAIL-001`（道具详情）的 HINT 误标风险

| 元素 | 归 HINT | 归 UI |
|------|---------|-------|
| 折扣角标（**事件触发显示**）| ✅ | ❌ |
| 折扣角标（**页面常驻**）| ❌ | ✅ |
| 限时倒计时浮窗（**事件触发弹出**）| ✅ | ❌ |
| 详情页倒计时（**页面常驻**）| ❌ | ✅ |
| 错误提示 Toast | ✅ | ❌ |
| 错误码显示 | ❌ | ✅（如果常驻）|

> **核心三问**：
> 1. 事件触发弹出？→ HINT
> 2. 自动消失？→ HINT
> 3. 测"内容/数字/文案"？→ HINT

## 4. EXCEPTION 类型 TP 的 s4_reference 实战

### 4.1 引用 S4 风险点格式

| 风险类型 | s4_reference | 实际 TP 示例 |
|----------|--------------|-------------|
| 竞态条件 | `R-01` | `BIZ-PURCHASE-001-TP-006` 余额不足时按钮禁用 |
| 时间依赖 | `R-07` | `CONFIG-DISCOUNT-002-TP-003` 倒计时最后一秒 |
| 状态损坏 | `R-09` | `UI-SHOP-001-TP-004` 缓存刷新延迟 |
| 支付幂等 | `R-02` | `LINK-PAYMENT-001-TP-006` 重复回调幂等 |
| 数据一致性 | `R-04` | `BIZ-PURCHASE-002-TP-006` 扣款成功到账失败 |
| 资源/容量 | `R-08` | `CONFIG-DISCOUNT-004-TP-006` 多设备并发 |
| 安全/合规 | `R-10` | `UI-SHOP-003-TP-006` SQL 注入防护 |

### 4.2 异常树叶子节点引用

| 异常树 ID | 场景 | 实际 TP 示例 |
|----------|------|-------------|
| `S4-1.3` | 余额/数量边界 | `BIZ-PURCHASE-001-TP-007` 数量为 0 |
| `S4-2.3` | VIP 配置冲突 | `CONFIG-VIP-001-TP-005` 解锁条件冲突 |
| `S4-3.3` | 折扣叠加冲突 | `CONFIG-DISCOUNT-001-TP-005` 时间重叠 |
| `S4-7.3` | 缓存异常 | `AUX-CACHE-001-TP-006` Redis 降级 |
| `S4-8.3` | 汇率配置异常 | `AUX-EXRATE-001-TP-003` 汇率≤0 |
| `S4-10.3` | 日志异常 | `LOG-PAYMENT-001-TP-005` 磁盘满 |

## 5. 8 模块覆盖率（v1.0 S7 审查指标）

| 模块 | 75 TP 覆盖率 | S7 100% 覆盖率 | 改进方向 |
|------|--------------|----------------|----------|
| **CONFIG** | 19 / 75 = 25.3% | 17 子模板覆盖 9 | 覆盖较好 |
| **UI** | 14 / 75 = 18.7% | 10 子模板覆盖 8 | 覆盖较好 |
| **BIZ** | 19 / 75 = 25.3% | 11 子模板覆盖 9 | 覆盖较好 |
| **AUX** | 1 / 75 = 1.3% | 16 子模板覆盖 1 | ⚠️ 偏少（缓存 Story 多 AUX 机会）|
| **LINK** | 3 / 75 = 4.0% | 8 子模板覆盖 3 | 偏少 |
| **SPECIAL** | 9 / 75 = 12.0% | 11 子模板覆盖 4 | 偏少 |
| **LOG** | 5 / 75 = 6.7% | 15 子模板覆盖 5 | 偏少 |
| **HINT** | 5 / 75 = 6.7% | 16 子模板覆盖 4 | 偏少 |

> **结论**：v1.0 旧产物的 8 模块覆盖**不均衡**（BIZ/CONFIG/UI 多，HINT/LOG/LINK/SPECIAL/AUX 少）——v2.0+ 需补充 HINT（按"13 大类"）、LOG（按"13 大类"）、SPECIAL（按"9 大类"）等子模块覆盖。

## 6. 4 类型覆盖率

| 类型 | 75 TP 覆盖率 | 平均/Story | 改进方向 |
|------|--------------|-----------|----------|
| **POSITIVE** | 35 / 75 = 46.7% | 1.35 | ✅ 充足 |
| **BOUNDARY** | 19 / 75 = 25.3% | 0.73 | ✅ 接近 1.0 |
| **NEGATIVE** | 12 / 75 = 16.0% | 0.46 | ⚠️ 略少（建议补足到 1+）|
| **EXCEPTION** | 9 / 75 = 12.0% | 0.35 | ⚠️ 偏少（建议补足到 1+，重点引 S4）|

> **结论**：v1.0 POSITIVE + BOUNDARY 充足，NEGATIVE + EXCEPTION 偏少——v2.0+ S5 阶段需重点补足。

## 7. 关键 takeaway（**S5 LLM 总结**）

1. **按业务自然划分 TP 数量**——不要"1:1:1:1 凑 4 个"硬性约束
2. **跨模块多 TP**——同 Story 命中多模块 → 每个模块单独生成 TP
3. **HINT vs UI 严格按"临时 vs 常驻"判定**——临时弹出 = HINT；常驻 = UI
4. **EXCEPTION TP 必引 S4 风险点**——`s4_reference` 字段必填
5. **避免旧枚举名**——`SYS_MSG` / `RED_DOT` / `ENTITY_CACHE` / `ASSET_CHANGE` 等 v1.6.1 前枚举按迁移表升级
6. **8 模块分布应均衡**——HINT / LOG / LINK / SPECIAL / AUX 不应 < 5%
7. **测试 ID 严格 `{StoryID}-TP-NNN`**——`BIZ-PURCHASE-001-TP-001`
