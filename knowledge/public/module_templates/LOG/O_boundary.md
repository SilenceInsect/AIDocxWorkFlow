# O. LOG 模块边界区分（判定规则）

> **非测试类型**——本文件是 S5 生成测试点时的**判定参考**，
> 避免误标 LOG 标签（实际是 UTIL / BIZ / SPECIAL / LINK / UI / HINT / CONFIG）。
>
> **v1.9 核心原则**：**LOG = 日志业务规范 + 审计 + 埋点触发 + 合规**；**UTIL = 日志底层采集 SDK + 文件读写 + 网络上报框架**——两者**严格隔离、无职责重叠**。

---

## 1. 判定原则

> **LOG 测什么**：测"日志/埋点/审计的**业务规范与合规性**"——业务应该埋什么点、日志应该怎么记、审计对账怎么做、字段如何合规、链路如何串联、崩溃如何上报。
>
> **UTIL 测什么**：测"日志底层**技术 SDK 与框架**"——SDK 怎么采集、文件怎么读写、网络怎么上报、压缩怎么分片、加密怎么落盘。
>
> **一句话判定**：
> - 测"业务侧应该写什么日志、写什么字段、字段怎么脱敏、链路怎么串联" → **LOG**
> - 测"S SDK 怎么采集、文件怎么写、网络怎么传" → **UTIL**

### 判定三问

1. **是"日志内容/审计/合规"还是"日志底层 SDK/框架"？** 是后者 → 不归 LOG（归 UTIL）
2. **是"日志业务触发规则"还是"日志技术实现"？** 是技术实现 → 不归 LOG（归 UTIL）
3. **是"日志规范/审计对账"还是"通用网络/缓存/加密"？** 是通用技术 → 不归 LOG（归 UTIL）

## 2. 边界对照表（重点：vs UTIL 严格隔离）

### vs UTIL（核心边界 — 严格隔离）

| 场景 | 归 LOG | 归 UTIL |
|------|--------|--------|
| 日志采集 | 业务埋点触发规则（应埋什么、什么时候埋）| 日志 SDK 采集机制（怎么采、断网缓存）|
| 日志打印 | 日志级别规范（FATAL/ERROR/WARN/INFO/DEBUG）| 通用打印工具（log4j/slf4j 封装）|
| 崩溃堆栈 | 崩溃日志内容（堆栈 + 上下文 + 设备）| 崩溃底层捕获（Native/JVM 异常处理）|
| 本地存储 | 日志业务分层（冷热/分级/生命周期）| 通用 localStorage 工具（玩家设置存档）|
| 网络上报 | 日志上报容错（断网/分片/重试）| 通用网络层（断线重连/HTTP 客户端）|
| 压缩分片 | 日志分片上传规则（多大分片、怎么合并）| 通用压缩工具（gzip 封装）|
| 加密 | 日志脱敏规则（手机号/实名/设备 ID 脱敏）| 加密算法本身（AES/SHA 实现）|
| 缓存 | 日志查询性能指标（按 player_id 查 < 1s）| Redis 缓存机制（缓存击穿/雪崩）|
| 异常兜底 | 业务异常日志（购买失败/网络失败）| 全局崩溃捕获（try-catch 框架）|
| 性能监控 | 业务性能埋点（接口 P99/CCU/转化率）| 通用性能组件（FPS 监控/内存监控）|
| GM 工具 | GM 操作审计日志（封号/发奖/重置）| GM 指令 SDK（`gm_add_item` 工具）|
| 运营工具 | 运营操作日志（白名单/灰度）| 运营后台 SDK（批量发奖工具）|

### vs BIZ

| 场景 | 归 LOG | 归 BIZ |
|------|--------|--------|
| 业务触发 | 业务操作日志留痕（购买/合成/领奖）| 业务流程本身（购买扣款/发货业务）|
| 业务结果 | 业务结果日志（购买成功 +100 钻）| 业务结果计算（扣款/发货/状态机）|
| 业务异常 | 业务异常日志（购买失败/参数非法）| 业务异常处理（购买失败补偿/重试）|
| 业务审计 | 业务侧落点规范（购买日志格式正确）| 业务审计对账（充值-发奖-扣资源三角账）|
| 业务状态 | 状态变更日志（活动开启/Buff 过期）| 状态机本身（活动状态流转规则）|
| 并发 | 并发操作日志（100 玩家抢礼包）| 并发业务结果（限量/先到先得）|
| 定时 | 定时任务日志（0 点重置/赛季结算）| 定时业务（重置逻辑/赛季奖励）|
| 付费 | 付费业务日志（充值/退款/对账）| 付费业务流程（订单/回调/对账业务）|

## 3. 常见误判案例

### 误判 1：崩溃日志 → 错标 LOG
- **错误**：看到"崩溃"，就标 LOG
- **正确**：
  - LOG-E 测"崩溃日志内容"（堆栈 + 上下文 + 设备）→ 归 LOG
  - UTIL-N 测"崩溃底层捕获"（Native 异常处理）→ 归 UTIL
  - BIZ 测"崩溃后业务恢复"（自动重连/补偿）→ 归 BIZ
  - 同一 Story 应有 3 个 TP

### 误判 2：玩家点击 → 错标 LOG
- **错误**：看到"玩家点击"，就标 LOG 一个
- **正确**：
  - LOG-A 测"点击埋点触发"（什么时候埋、含什么字段）→ 归 LOG
  - UI 测"按钮 UI 渲染"（按钮样式/状态）→ 归 UI
  - BIZ 测"点击触发业务"（购买扣款）→ 归 BIZ
  - 同一 Story 应有 3 个 TP

### 误判 3：商城购买 → 错标 LOG
- **错误**：看到"购买日志"，就标 LOG
- **正确**：
  - LOG-A 测"购买埋点"（业务触发规则）→ 归 LOG
  - LOG-B 测"购买资产对账"（扣款/发货流水）→ 归 LOG
  - LOG-C 测"购买操作日志"（操作留痕）→ 归 LOG
  - LOG-H 测"购买字段脱敏"（合规）→ 归 LOG
  - BIZ-A 测"购买业务"（扣款/发货）→ 归 BIZ
  - BIZ-H 测"支付订单"（订单流程）→ 归 BIZ
  - 同一 Story 应有 6 个 TP

### 误判 4：日志写入 → 错标 LOG
- **错误**：看到"日志写入"，就标 LOG
- **正确**：
  - LOG-F 测"日志分级/生命周期/冷热分离"（业务侧）→ 归 LOG
  - LOG-M 测"日志上报容错/重试/分片"（业务侧）→ 归 LOG
  - UTIL-J 测"通用 localStorage 工具"（底层）→ 归 UTIL
  - UTIL-B 测"通用网络层"（底层）→ 归 UTIL
  - 同一 Story 应有 4 个 TP

### 误判 5：断线重传 → 错标 LOG
- **错误**：看到"断线"，就标 LOG 一个
- **正确**：
  - LOG-G 测"日志完整性补传"（断网缓存+补传）→ 归 LOG
  - LOG-M 测"日志上报容错"（重试/分片）→ 归 LOG
  - UTIL-B 测"断线重连"（通用网络层）→ 归 UTIL
  - BIZ-B 测"业务断线处理"（业务数据时序）→ 归 BIZ
  - SPECIAL 测"弱网降级"（行为异常）→ 归 SPECIAL
  - 同一 Story 应有 5 个 TP

### 误判 6：充值 → 错标 LOG
- **错误**：看到"充值"，就标 LOG
- **正确**：
  - LOG-A 测"充值埋点"（业务触发）→ 归 LOG
  - LOG-B 测"充值流水对账"（资产审计）→ 归 LOG
  - LOG-C 测"充值操作日志"（操作留痕）→ 归 LOG
  - LOG-K 测"支付回调日志"（第三方）→ 归 LOG
  - BIZ-H 测"充值业务"（订单/回调/对账）→ 归 BIZ
  - LINK 测"支付 SDK 集成"（支付宝/微信）→ 归 LINK
  - 同一 Story 应有 6 个 TP

### 误判 7：行为埋点 → 错标 UTIL
- **错误**：看到"埋点"，就标 UTIL
- **正确**：
  - LOG-A 测"埋点业务触发规则"（应埋什么）→ 归 LOG
  - LOG-H 测"埋点字段合规"（脱敏/必填）→ 归 LOG
  - LOG-M 测"埋点上报容错"（断网/分片）→ 归 LOG
  - UTIL 测"埋点 SDK"（采集框架）→ 归 UTIL
  - 注意：UTIL v1.6.1 已收窄到"底层 SDK"，不再含"业务埋点"

### 误判 8：审计 → 错标 LOG
- **错误**：看到"审计"，就标 LOG
- **正确**：
  - LOG-B 测"资产审计流水"（财务对账）→ 归 LOG
  - LOG-C 测"操作审计日志"（GM/玩家操作留痕）→ 归 LOG
  - LOG-I 测"线上问题溯源"（trace/上下文）→ 归 LOG
  - BIZ-I 测"业务侧落点规范"（业务是否写日志）→ 归 BIZ
  - 同一 Story 应有 4 个 TP

### 误判 9：跨服 → 错标 LOG
- **错误**：看到"跨服日志"，就标 LOG
- **正确**：
  - LOG-K 测"跨服同步日志"（发送/接收留痕）→ 归 LOG
  - LOG-L 测"跨服日志隔离"（不可互读）→ 归 LOG
  - LOG-I 测"跨服 trace"（链路串联）→ 归 LOG
  - LINK 测"跨服同步业务"（数据一致性）→ 归 LINK
  - BIZ-F 测"跨服并发业务"（拍卖/组队）→ 归 BIZ
  - 同一 Story 应有 5 个 TP

### 误判 10：合规 → 错标 LOG
- **错误**：看到"合规"，就标 LOG
- **正确**：
  - LOG-H 测"埋点字段合规"（必填/脱敏/隐私）→ 归 LOG
  - LOG-J 测"安全违规日志"（风控溯源）→ 归 LOG
  - UTIL-M 测"加密算法"（AES 实现）→ 归 UTIL
  - 业务合规（法务/数据安全）→ 项目外
  - 同一 Story 应有 2 个 TP

## 4. LOG vs UTIL 严格隔离速查表

> **核心原则**（项目内反复强调）：

| 维度 | LOG（业务规范） | UTIL（底层 SDK）|
|------|---------------|--------------|
| 关注什么 | 业务侧应该写什么日志 | SDK 怎么采集、怎么传输 |
| 测什么 | 埋点触发规则、字段合规、链路串联 | 采集性能、断网缓存、压缩分片 |
| 谁负责 | S5/S6 测试用例生成 | 通用工具/框架组件 |
| 例子：崩溃 | 测"崩溃日志含堆栈+上下文+设备" | 测"crash SDK 捕获 Native 异常" |
| 例子：埋点 | 测"购买应该埋 click_buy 事件" | 测"埋点 SDK 断网缓存 1MB" |
| 例子：上报 | 测"上报失败重试 3 次后丢弃" | 测"HTTP 客户端 5s 超时" |
| 例子：存储 | 测"FATAL 保留 5 年、DEBUG 自动清" | 测"localStorage 读写 100ms" |
| 例子：脱敏 | 测"手机号脱敏为 138****5678" | 测"AES 加密算法" |

> **v1.6.1 → v1.9 演进**：
> - v1.6 时代 UTIL 同时含"日志/埋点/崩溃"等
> - v1.6.1 UTIL 收窄到"底层基础工具"，日志/埋点/崩溃业务侧迁出
> - v1.9 LOG 完整承接，**LOG 管业务规范、UTIL 管底层 SDK**——**严格隔离**
