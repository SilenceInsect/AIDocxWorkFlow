# O. SPECIAL 模块边界区分（判定规则）

> **非测试类型**——本文件是 S5 生成测试点时的**判定参考**，
> 避免误标 SPECIAL 标签（实际是 BIZ / UTIL / LINK / UI / LOG / HINT / CONFIG）。
>
> **v1.11 核心**：SPECIAL = **异常 + 高危 + 对抗 + 极限 + 合规 + 资源耗尽** 类业务风控与容错逻辑；
> 底层网络重连、崩溃捕获、缓存工具等基础设施归 **UTIL**；正常正向业务流程归 **BIZ**；
> 第三方 / 跨服互通异常归 **LINK**。
> **判不出来的回归到 3 步判定法**：（1）涉及底层能力/工具/SDK？→ UTIL；（2）涉及单系统独立业务？→ BIZ；（3）涉及多系统/多端/外部互通？→ LINK；（4）涉及异常/高危/对抗/极限/合规/资源耗尽？→ SPECIAL

---

## 1. 判定原则

> **SPECIAL = 异常 + 高危 + 对抗 + 极限 + 合规 + 资源耗尽 业务风控与容错**
>
> **不归 SPECIAL**：底层网络/工具/SDK → UTIL；正常业务流转 → BIZ；第三方/跨服 → LINK；UI 控件 → UI；提示内容 → HINT；通用日志 → LOG；配置字段 → CONFIG

### 判定五问

1. **是"异常环境/极限场景"吗？**（弱网/宕机/资源耗尽/合规）→ 是 → 归 SPECIAL
2. **是"对抗行为/高危操作"吗？**（反作弊/批量操作/风控拦截）→ 是 → 归 SPECIAL
3. **是"业务边界/异常分支"吗？**（CD/超时/权限不足/版本不兼容）→ 是 → 归 SPECIAL
4. **是"正常运行/业务流转"吗？** → 是 → 归 BIZ
5. **是"底层工具/SDK"吗？** → 是 → 归 UTIL

### 何时归 SPECIAL（核心信号）

- ✅ 涉及**反作弊/数据篡改/参数校验**（"检测到 XX 异常"）
- ✅ 涉及**弱网/断网/网络抖动的业务容错**（"网络异常"）
- ✅ 涉及**前后台切换/进程生命周期的状态恢复**（"App 切回前台"）
- ✅ 涉及**服务宕机/回档/并发极限/高危操作**（"Failover" / "GM 高危"）
- ✅ 涉及**版本兼容/资源热更缺失**（"客户端版本过低"）
- ✅ 涉及**环境/渠道/灰度/测试服隔离**（"测试服数据泄漏"）
- ✅ 涉及**防沉迷/未成年/敏感词/地区合规**（"未成年游戏时长"）
- ✅ 涉及**资源耗尽/服务器压力**（"背包满" / "DB 连接耗尽"）
- ✅ 涉及**业务异常分支/边界值**（"活动未开启" / "CD 中" / "权限不足"）
- ✅ 涉及**限流/防刷/雪崩保护**（"操作太频繁"）

## 3. 常见误判案例

### 误判 1：网络断开 → 错标 SPECIAL
- **错误**：看到"断网"，就标 SPECIAL
- **正确**：
  - UTIL 测"底层重连"（TCP/HTTP 重试）→ 归 UTIL B
  - SPECIAL C 测"业务容错"（重发/回滚/不重复扣款）→ 归 SPECIAL
  - BIZ 测"断线时业务处理"（订单恢复）→ 归 BIZ
  - 同一 Story 应有 3 个 TP

### 误判 2：金币为负数 → 错标 BIZ

## 6. v1.11 SPECIAL 核心定位（防止再混淆）

> **SPECIAL = "非正常、高危、极端、对抗性场景"**
> 1. **底层能力归 UTIL**（网络 SDK/缓存工具/崩溃框架等）——SPECIAL 校验**这套能力之上**的业务容错
> 2. **正常业务归 BIZ**（正常流程/常规分支）——SPECIAL 只处理**异常**场景
> 3. **第三方/跨服正常互通归 LINK**——SPECIAL 只处理**第三方接口异常 + 跨服数据作弊拦截**
> 4. **界面加载失败归 UI**（纯视觉）——SPECIAL 处理**界面异常导致的业务错乱**
> 5. **SPECIAL 专精**：异常/高危/对抗/极限/合规/资源耗尽——9 大类（A-I）各管一段，无重叠
