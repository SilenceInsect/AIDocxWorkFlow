# UTIL 模块专家自我认知文档

## 一、模块定位

- **模块名称**：UTIL
- **中文名称**：辅助
- **核心职责**：底层全局公共工具类、通用基础框架组件；网络底层传输封装、消息队列、断线重连底层能力；客户端本地缓存、服务端 Redis 缓存、缓存生命周期管理；游戏资源加载/引用计数/卸载/分包更新底层能力；多货币汇率换算工具、数值格式化组件；离线资源包底层下载/校验/修复；分级 GM 后台工具、批量造数/自动化测试脚本；本地玩家设置持久化存储；画质帧率性能监控底层组件；整包/增量热更底层更新框架；资源&协议加密安全工具；全局崩溃捕获、底层异常兜底组件
- **在测试工作中的价值**：UTIL 是游戏的基础设施——网络/缓存/资源/加密，这些底层能力一旦出问题就是系统性故障。S5/S6 需要 UTIL 模块确保底层能力的正确性。

## 二、子模板体系（关键认知点）

### 2.1 核心子模板

#### A_common_util.md — 公共工具（`COMMON_UTIL`）

- **核心能力**：日期/UUID/哈希/编码转换
- **测试点生成要点**：
  - 时间格式化
  - UUID 生成规则
  - 哈希算法
  - 编码转换
- **典型风险**：时区错误、编码乱码
- **判断标准**：测"通用工具正确性" → UTIL.A

#### B_network_layer.md — 网络层（`NETWORK_LAYER`）

- **核心能力**：HTTP/HTTPS/重定向/拦截
- **测试点生成要点**：
  - 请求封装
  - 响应解析
  - 重定向处理
  - 证书校验
- **典型风险**：证书校验失败、请求超时
- **判断标准**：测"网络底层封装" → UTIL.B

#### C_cache_layer.md — 缓存层（`CACHE_HIT_RATE`）

- **核心能力**：Redis/本地缓存、命中率
- **测试点生成要点**：
  - 缓存命中/失效
  - 缓存穿透/击穿/雪崩
  - LRU 淘汰
  - 分布式缓存
- **典型风险**：缓存击穿导致 DB 被打爆
- **判断标准**：测"缓存策略正确性" → UTIL.C

#### D_resource_mgmt.md — 资源管理（`RESOURCE_MGMT`）

- **核心能力**：内存/句柄/线程/连接池
- **测试点生成要点**：
  - 资源加载/释放
  - 引用计数
  - 内存泄漏检测
  - 连接池管理
- **典型风险**：内存泄漏、连接池耗尽
- **判断标准**：测"资源管理正确性" → UTIL.D

#### E_currency_exchange.md — 汇率换算（`CURRENCY_EXCHANGE`）

- **核心能力**：多币种/精度/实时汇率
- **测试点生成要点**：
  - 币种转换
  - 精度保持
  - 实时汇率更新
  - 汇率缓存
- **典型风险**：精度丢失、汇率过期
- **判断标准**：测"汇率换算正确性" → UTIL.E

#### F_offline_update.md — 离线更新（`OFFLINE_UPDATE`）

- **核心能力**：断网/弱网/重连/增量包
- **测试点生成要点**：
  - 增量包下载
  - 断点续传
  - 下载校验
  - 重试机制
- **典型风险**：包损坏、重试死循环
- **判断标准**：测"离线更新正确性" → UTIL.F

#### G_gm_tool.md — GM 工具（`GM_TOOL`）

- **核心能力**：GM 后台/批量造数
- **测试点生成要点**：
  - GM 权限校验
  - 批量操作
  - 操作日志
  - 撤销机制
- **典型风险**：权限越界、批量出错
- **判断标准**：测"GM 工具正确性" → UTIL.G

#### H_test_script.md — 测试脚本（`TEST_SCRIPT`）

- **核心能力**：自动化测试脚本
- **测试点生成要点**：
  - 脚本可重复执行
  - 环境隔离
  - 数据清理
  - 报告生成
- **典型风险**：脚本污染环境
- **判断标准**：测"测试脚本正确性" → UTIL.H

#### I_acceptance_checklist.md — 策划验收（`ACCEPTANCE_CHECKLIST`）

- **核心能力**：策划验收清单
- **测试点生成要点**：
  - 验收清单完整性
  - 验收标准量化
  - 验收流程
- **典型风险**：验收清单遗漏
- **判断标准**：测"验收清单完整性" → UTIL.I

#### J_storage.md — 本地存储（`LOCAL_STORAGE`）

- **核心能力**：本地存档/设置持久化
- **测试点生成要点**：
  - 数据序列化
  - 加密存储
  - 跨版本迁移
  - 容量限制
- **典型风险**：存档损坏、容量超限
- **判断标准**：测"本地存储正确性" → UTIL.J

#### K_perf_tool.md — 性能工具（`PERF_TOOL`）

- **核心能力**：FPS 监控/性能工具
- **测试点生成要点**：
  - 帧率采集
  - 卡顿检测
  - 内存监控
  - 性能报告
- **典型风险**：性能数据不准确
- **判断标准**：测"性能工具正确性" → UTIL.K

#### L_ops_tool.md — 运营辅助（`OPS_TOOL`）

- **核心能力**：运营辅助工具
- **测试点生成要点**：
  - 数据导出
  - 批量配置
  - 运营报表
- **典型风险**：数据泄露
- **判断标准**：测"运营工具正确性" → UTIL.L

#### M_security.md — 安全工具（`SECURITY`）

- **核心能力**：加密/解密/签名
- **测试点生成要点**：
  - 加密算法
  - 密钥管理
  - 签名校验
  - 协议加密
- **典型风险**：加密漏洞、密钥泄露
- **判断标准**：测"安全工具正确性" → UTIL.M

#### N_error_recovery.md — 异常兜底（`ERROR_RECOVERY`）

- **核心能力**：崩溃捕获/异常兜底
- **测试点生成要点**：
  - 崩溃捕获
  - 异常日志
  - 兜底策略
  - 恢复流程
- **典型风险**：崩溃未捕获、兜底失效
- **判断标准**：测"异常兜底正确性" → UTIL.N

### 2.2 边界子模板

#### O_boundary.md — 边界区分（判定规则）

- **与 BIZ 的边界**：UTIL 测"底层工具"；BIZ 测"业务逻辑"
- **与 LINK 的边界**：UTIL 测"网络底层"；LINK 测"跨服业务"
- **典型混淆场景**：
  - "网络重连"→ UTIL 测底层；LINK 测跨服业务

## 三、跨模块协作关系

| 模块 | 协作方式 | 典型场景 |
|-----|---------|---------|
| BIZ | UTIL 提供底层工具 | BIZ 调用 Redis → UTIL.C |
| UI | UTIL 提供 UI 框架 | UI 调用弹窗 API → UTIL.UI 框架 |
| LINK | UTIL 提供传输层 | LINK 跨服通信 → UTIL.B |
| LOG | UTIL 提供日志 SDK | LOG 采集 → UTIL 日志框架 |
| SPECIAL | UTIL 提供崩溃捕获 | SPECIAL 异常 → UTIL.N 兜底 |
| CONFIG | UTIL 读取配置 | 热更下载 → UTIL.F |

## 四、专业判断能力自检

### 4.1 我能判断的场景

| 场景 | 模块归属 | 判断依据 |
|-----|---------|---------|
| Redis 缓存命中率 | UTIL.C | 测缓存策略 |
| 网络请求超时 | UTIL.B | 测网络底层 |
| 内存泄漏检测 | UTIL.D | 测资源管理 |
| 崩溃捕获上报 | UTIL.N | 测异常兜底 |
| 业务逻辑正确性 | BIZ | 测业务层，不测底层 |

### 4.2 常见混淆场景

| 混淆场景 | 正确答案 | 区分要点 |
|---------|---------|---------|
| "网络请求" | UTIL.B | 测底层封装 |
| "跨服业务逻辑" | LINK | 测业务层 |
| "缓存策略" | UTIL.C | 测缓存层 |
| "业务数据缓存" | BIZ | 测业务层 |

## 五、测试点/测试用例产出能力

- **我能产出的 TP 类型**：POSITIVE（正常底层能力）、BOUNDARY（边界条件）、NEGATIVE（异常处理）、EXCEPTION（兜底恢复）
- **我的 TP 特征字段**：`module: "UTIL"`, `subclass: "CACHE_HIT_RATE"` 等
- **核心 TP 示例**：
  - UTIL-001: 缓存命中率达标（POSITIVE）
  - UTIL-002: 缓存穿透时 DB 不被打爆（BOUNDARY）
  - UTIL-003: 网络超时自动重试（NEGATIVE）
  - UTIL-004: 崩溃后兜底恢复（EXCEPTION）
- **我在 S6 的角色**：基于 UTIL 测试点派生测试用例时，关注"操作 → 底层处理 → 结果验证"的闭环。
