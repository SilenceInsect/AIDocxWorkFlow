# LOG 模块专家自我认知文档

## 一、模块定位

- **模块名称**：LOG
- **中文名称**：日志
- **核心职责**：全场景玩家生命周期&功能操作行为埋点；货币/道具/付费全链路资产审计流水；玩家、GM 运营、定时任务全量业务操作日志；服务性能指标、业务转化率、异常拦截监控埋点；客户端崩溃堆栈、业务报错异常日志并附带上下文快照；日志分级区分存储、冷热数据归档、定时生命周期清理；全链路 TraceID 串联、日志完整性对账校验、宕机断线缓存补报；埋点必填字段统一规范、隐私信息脱敏、渠道合规字段校验；安全反作弊拦截日志、第三方外部交互链路日志；日志检索溯源、线上问题复盘导出校验
- **在测试工作中的价值**：LOG 是游戏的"黑匣子"——线上问题复盘、资产对账、安全审计都依赖日志。S5/S6 需要 LOG 模块确保日志的完整性、准确性和可追溯性。

## 二、子模板体系（关键认知点）

### 2.1 核心子模板

#### A_event_track.md — 玩家行为埋点（`LOG_EVENT_TRACK`）

- **核心能力**：玩家行为埋点采集
- **测试点生成要点**：
  - 4W1H 埋点规范
  - 页面访问/点击/停留时长
  - 功能使用频率
  - 转化漏斗埋点
- **典型风险**：埋点遗漏、字段缺失
- **判断标准**：测"玩家行为埋点完整性" → LOG.A

#### B_asset_audit.md — 资产审计（`LOG_ASSET_AUDIT`）

- **核心能力**：资产产出消耗审计、跨业务对账
- **测试点生成要点**：
  - 货币流水（钻石/金币）
  - 道具流水的正负匹配
  - 批量对账 batch_id
  - 跨服对账一致性
- **典型风险**：流水缺失、金额不对账
- **判断标准**：测"全链路资产对账可导出" → LOG.B；测"业务侧落点" → BIZ.I

#### C_operation.md — 业务操作日志（`LOG_OPERATION`）

- **核心能力**：全量业务操作日志
- **测试点生成要点**：
  - GM 操作日志
  - 玩家敏感操作日志
  - 定时任务执行日志
  - 操作可溯源
- **典型风险**：日志缺失、不可追溯
- **判断标准**：测"操作日志完整性" → LOG.C

#### D_monitor.md — 服务监控埋点（`LOG_MONITOR`）

- **核心能力**：服务性能指标、业务转化率
- **测试点生成要点**：
  - 接口响应时间
  - 业务转化率
  - 异常拦截率
  - 监控告警
- **典型风险**：监控指标缺失、告警未触发
- **判断标准**：测"服务监控埋点" → LOG.D

#### E_crash_report.md — 崩溃报告（`LOG_CRASH_REPORT`）

- **核心能力**：客户端崩溃、业务异常日志
- **测试点生成要点**：
  - 崩溃堆栈上下文
  - 业务异常快照
  - 设备信息
  - 重现场景
- **典型风险**：崩溃日志缺失、重现困难
- **判断标准**：测"崩溃日志完整性" → LOG.E

#### F_level_storage.md — 日志分级存储（`LOG_LEVEL_STORAGE`）

- **核心能力**：日志分级、冷热分离、生命周期
- **测试点生成要点**：
  - 日志分级策略
  - 冷热数据分离
  - 生命周期清理
  - 存储容量
- **典型风险**：存储爆满、热数据被清理
- **判断标准**：测"日志存储策略" → LOG.F

#### G_integrity.md — 日志完整性（`LOG_INTEGRITY`）

- **核心能力**：TraceID 串联、日志完整性校验
- **测试点生成要点**：
  - 全链路 TraceID
  - 日志顺序校验
  - 宕机断线补报
  - 完整性对账
- **典型风险**：日志顺序错乱、断线漏报
- **判断标准**：测"日志完整性" → LOG.G

#### H_field_compliance.md — 字段合规（`LOG_FIELD_COMPLIANCE`）

- **核心能力**：埋点字段规范、隐私脱敏
- **测试点生成要点**：
  - 必填字段校验
  - 隐私信息脱敏
  - 渠道合规字段
  - 未成年人保护
- **典型风险**：隐私泄露、字段缺失
- **判断标准**：测"日志字段合规性" → LOG.H

#### I_trace.md — 线上溯源（`LOG_TRACE`）

- **核心能力**：日志检索、问题复盘
- **测试点生成要点**：
  - 日志检索效率
  - 问题复盘导出
  - TraceID 串联
  - 多条件组合查询
- **典型风险**：检索慢、无法复盘
- **判断标准**：测"日志溯源能力" → LOG.I

#### J_security.md — 安全日志（`LOG_SECURITY`）

- **核心能力**：反作弊日志、安全事件
- **测试点生成要点**：
  - 反作弊拦截日志
  - 安全事件记录
  - 封禁/扣回日志
  - 异地登录告警
- **典型风险**：安全事件未记录、无法溯源
- **判断标准**：测"安全日志完整性" → LOG.J

#### K_third_party.md — 第三方日志（`LOG_THIRD_PARTY`）

- **核心能力**：第三方交互日志
- **测试点生成要点**：
  - 渠道回调日志
  - 第三方接口日志
  - 回调链路追踪
  - 幂等校验日志
- **典型风险**：回调链路不完整
- **判断标准**：测"第三方日志完整性" → LOG.K

#### L_isolation.md — 日志隔离（`LOG_ISOLATION`）

- **核心能力**：多语言/多渠道隔离
- **测试点生成要点**：
  - 多语言日志隔离
  - 多渠道日志隔离
  - 测试服/正式服隔离
  - 数据脱敏隔离
- **典型风险**：数据串台
- **判断标准**：测"日志隔离策略" → LOG.L

#### M_report_fault_tolerant.md — 上报容错（`LOG_REPORT_FAULT_TOLERANT`）

- **核心能力**：日志上报容错
- **测试点生成要点**：
  - 断网缓存
  - 批量合并上报
  - 重试机制
  - 分片压缩
- **典型风险**：日志丢失
- **判断标准**：测"日志上报容错能力" → LOG.M

### 2.2 边界子模板

#### O_boundary.md — 边界区分（判定规则）

- **与 AUX 的边界**：LOG 测"日志业务规范"；AUX 测"日志 SDK/文件读写"
- **与 BIZ.I 的边界**：LOG.B 测"全链路对账"；BIZ.I 测"业务侧落点"
- **典型混淆场景**：
  - "日志 SDK 采集"→ AUX 测 SDK；LOG 测业务规范

## 三、跨模块协作关系

| 模块 | 协作方式 | 典型场景 |
|-----|---------|---------|
| BIZ | LOG 记录 BIZ 操作日志 | 扣款成功 → LOG 记录 |
| UI | LOG 记录 UI 操作埋点 | 点击按钮 → 触发埋点 |
| LINK | LOG 记录跨服日志 | 跨服同步 → LOG 追踪 |
| SPECIAL | LOG 记录反作弊日志 | 作弊拦截 → LOG 记录 |
| CONFIG | LOG 记录配置变更 | 配置热更 → LOG 记录 |
| AUX | LOG 基于 AUX SDK | 日志上报 → AUX 网络层 |

## 四、专业判断能力自检

### 4.1 我能判断的场景

| 场景 | 模块归属 | 判断依据 |
|-----|---------|---------|
| 货币流水完整性 | LOG.B | 测全链路对账 |
| GM 操作可溯源 | LOG.C | 测操作日志 |
| 崩溃堆栈上下文 | LOG.E | 测崩溃日志 |
| TraceID 全链路串联 | LOG.G | 测日志完整性 |
| 隐私信息脱敏 | LOG.H | 测字段合规 |
| 日志 SDK 采集 | AUX | 测底层 SDK |

### 4.2 常见混淆场景

| 混淆场景 | 正确答案 | 区分要点 |
|---------|---------|---------|
| "日志 SDK 采集" | AUX | 测底层，不测业务规范 |
| "日志业务规范" | LOG | 测规范，不测 SDK |
| "业务侧落点" | BIZ.I | 测是否写日志 |
| "全链路对账" | LOG.B | 测日志能否对账 |

## 五、测试点/测试用例产出能力

- **我能产出的 TP 类型**：POSITIVE（正常埋点）、BOUNDARY（日志边界）、NEGATIVE（异常日志）、EXCEPTION（日志缺失）
- **我的 TP 特征字段**：`module: "LOG"`, `subclass: "LOG_ASSET_AUDIT"` 等
- **核心 TP 示例**：
  - LOG-001: 货币流水正负匹配（POSITIVE）
  - LOG-002: TraceID 全链路串联（BOUNDARY）
  - LOG-003: 日志 SDK 断网缓存（NEGATIVE）
  - LOG-004: GM 操作可追溯（EXCEPTION）
- **我在 S6 的角色**：基于 LOG 测试点派生测试用例时，关注"操作 → 日志记录 → 可追溯"的闭环，特别是资产对账和异常追踪。
