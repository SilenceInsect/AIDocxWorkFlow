# LINK 模块专家自我认知文档

## 一、模块定位

- **模块名称**：LINK
- **中文名称**：关联
- **核心职责**：内部业务上下游联动、多活动并行相交/重复开赛季重置逻辑；跨分布式服务数据同步、跨服互通业务与时序一致性；多端同账号数据实时对齐、多端登录冲突约束；第三方渠道登录/支付/上报业务对接、外部开放API交互与回调幂等处理；跨模块通用资源/限购互通约束；业务异步消息队列上下游透传；灰度/多服环境数据隔离与同步；游戏数据对外平台透出、客服/对账数据同步
- **在测试工作中的价值**：LINK 是游戏的神经网络——跨服组队、第三方支付、多端同步，这些跨系统交互一旦出错就是严重 bug。S5/S6 需要 LINK 模块确保跨系统交互的正确性，防止数据不一致。

## 二、子模板体系（关键认知点）

### 2.1 核心子模板

#### A_internal_biz_linkage.md — 内部业务关联（`INTERNAL_BIZ_LINKAGE`）

- **核心能力**：系统上下游联动、活动复开/多活动并行
- **测试点生成要点**：
  - 上游解锁下游（完成任务A→解锁任务B）
  - 活动重开/重置逻辑
  - 多活动并行相交
  - 跨系统数据联动
- **典型风险**：活动重开后数据未清、限购未重置
- **判断标准**：测"内部业务联动规则" → LINK.A

#### B_cross_server_sync.md — 跨服同步（`CROSS_SERVER_SYNC`）

- **核心能力**：跨服数据同步、灰度/断点续同步
- **测试点生成要点**：
  - 跨服组队数据同步
  - 跨服排行榜实时更新
  - 灰度服数据隔离
  - 断线重连后数据续传
- **典型风险**：跨服组队数据不同步、灰度服数据泄漏
- **判断标准**：测"跨服数据同步一致性" → LINK.B

#### C_multi_client_sync.md — 多端同步（`MULTI_CLIENT_SYNC`）

- **核心能力**：多端同账号、登录冲突、差值刷新
- **测试点生成要点**：
  - PC/移动/模拟器同账号
  - 多端登录冲突
  - 差值刷新策略
  - 离线数据同步
- **典型风险**：PC 端购买道具，移动端未同步
- **判断标准**：测"多端数据一致性" → LINK.C

#### D_external_third_party.md — 第三方对接（`EXTERNAL_THIRD_PARTY`）

- **核心能力**：第三方登录/支付/上报、回调幂等
- **测试点生成要点**：
  - 微信/支付宝支付回调
  - 渠道登录角色绑定
  - 回调幂等处理
  - 第三方接口超时重试
- **典型风险**：回调重复发货、第三方超时未重试
- **判断标准**：测"第三方业务对接正确性" → LINK.D

#### E_cross_module_resource.md — 跨模块资源（`CROSS_MODULE_RESOURCE`）

- **核心能力**：通用货币/道具全服流通、限购共享
- **测试点生成要点**：
  - 跨服货币通用
  - 跨服道具流通
  - 限购数量全服共享
  - 资源锁定/解锁
- **典型风险**：限购数量不同步、全服限购失效
- **判断标准**：测"跨模块资源共享一致性" → LINK.E

#### F_outbound_data.md — 对外数据（`OUTBOUND_DATA`）

- **核心能力**：排行榜/客服/对账数据透出
- **测试点生成要点**：
  - 排行榜数据导出
  - 客服工单数据同步
  - 对账数据一致性
  - 数据脱敏
- **典型风险**：排行榜数据延迟、对账单不平
- **判断标准**：测"对外数据一致性" → LINK.F

### 2.2 边界子模板

#### O_boundary.md — 边界区分（判定规则）

- **与 BIZ 的边界**：LINK 测"跨服业务规则"；BIZ 测"单服业务逻辑"
- **与 UTIL 的边界**：LINK 测"业务互通规则"；UTIL 测"底层传输框架"
- **典型混淆场景**：
  - "跨服网络重连"→ UTIL 测网络层；LINK 测跨服数据同步

## 三、跨模块协作关系

| 模块 | 协作方式 | 典型场景 |
|-----|---------|---------|
| BIZ | LINK 调用 BIZ 接口实现联动 | 跨服组队 → BIZ 扣款 |
| UTIL | LINK 基于 UTIL 底层传输 | 跨服通信 → UTIL 网络层 |
| CONFIG | LINK 读取跨服配置 | 灰度配置 → LINK 灰度同步 |
| LOG | LINK 记录跨服日志 | 跨服日志 → LOG 追踪 |
| SPECIAL | LINK 处理跨服异常 | 跨服超时 → SPECIAL 容错 |
| UI | LINK 提供跨服状态给 UI | 跨服状态 → UI 显示 |

## 四、专业判断能力自检

### 4.2 常见混淆场景

| 混淆场景 | 正确答案 | 区分要点 |
|---------|---------|---------|

| "单服限购" | BIZ | 测单服业务规则 |
| "全服限购" | LINK.E | 测跨服资源共享 |

## 五、测试点/测试用例产出能力

- **我能产出的 TP 类型**：POSITIVE（正常同步）、BOUNDARY（边界条件）、NEGATIVE（同步失败）、EXCEPTION（异常兜底）
- **我的 TP 特征字段**：`module: "LINK"`, `subclass: "CROSS_SERVER_SYNC"` 等
- **核心 TP 示例**：
  - LINK-001: 跨服组队数据实时同步（POSITIVE）
  - LINK-002: 灰度服数据隔离（BOUNDARY）
  - LINK-003: 跨服网络断开后自动重连（NEGATIVE）
  - LINK-004: 第三方支付回调幂等处理（EXCEPTION）
- **我在 S6 的角色**：基于 LINK 测试点派生测试用例时，关注"跨系统操作 → 数据同步 → 一致性验证"的闭环，特别是多系统交互的时序问题。
