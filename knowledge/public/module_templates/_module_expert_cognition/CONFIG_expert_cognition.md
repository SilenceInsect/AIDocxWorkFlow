# CONFIG 模块专家自我认知文档

## 一、模块定位

- **模块名称**：CONFIG
- **中文名称**：配置
- **核心职责**：游戏数据表结构与字段合法性校验、枚举/资源/ID/时间格式约束、同表/全局配置数值一致性、跨表 ID 关联依赖校验、循环依赖拦截；客户端 & 服务端配置解析、复杂嵌套结构解析、超大表加载性能；概率/成长/上限数值逻辑校验；配置增量/全量热更新、在线/离线热更、异常回滚兜底；新旧版本前后兼容、赛季/灰度/渠道专属配置隔离；Excel 导出打包发布流程校验；服务端全局常量、定时任务、GM 参数配置校验。
- **在测试工作中的价值**：CONFIG 是所有业务功能的基础设施——道具/活动/数值/协议版本等配置是游戏运行的"宪法"，配置错误会导致功能失效、数值崩溃、安全漏洞。S5/S6 需要 CONFIG 模块确保配置层的正确性，防止带病配置上线。

## 二、子模板体系（关键认知点）

### 2.1 核心子模板

#### A_field_legality.md — 字段合法性校验（`FIELD_LEGALITY`）

- **核心能力**：单个配置表中每个字段的类型、取值、格式、约束、行列规范
- **测试点生成要点**：
  - int/float/string/array 字段类型校验
  - 数值范围上下限（闭区间/半开区间）
  - 枚举值合法性（枚举值不存在/空值）
  - 必填字段空值检查
  - 资源路径存在性/后缀/大小写
  - 时间配置合法性/时间倒置
  - 主键唯一性/重复行
- **典型风险**：策划录入 `item_id = "abc"`（应为 int）、`drop_rate = 1.5`（超出 0-1 范围）、枚举值 4（不存在）
- **判断标准**：测"导出工具拒绝非法字段值" → CONFIG.A；测"运行时解析性能/嵌套结构" → CONFIG.E

#### B_consistency.md — 同表/全局一致性（`FIELD_INTRA_DEP`）

- **核心能力**：同表内多个字段间数值一致、全局配置常量唯一、不同表但同名字段值统一
- **测试点生成要点**：
  - 同表内道具攻击力一致性梯度
  - 全局货币 ID 唯一性
  - 不同表同名字段值统一（如 all_tables 中货币名称）
- **典型风险**：道具 A 和道具 B 同为"武器"但属性梯度不一致
- **判断标准**：测"单个字段合法性" → CONFIG.A；测"多字段间一致" → CONFIG.B

#### C_cross_dep.md — 跨表依赖（`FIELD_CROSS_DEP`）

- **核心能力**：表与表之间的 ID 引用校验、循环依赖拦截
- **测试点生成要点**：
  - 任务表 reward_item_id 引用的道具 ID 必须存在
  - 活动表引用不存在的道具/商城配置
  - 循环依赖检测（A引用B，B引用C，C引用A）
- **典型风险**：任务奖励指向不存在的道具 ID
- **判断标准**：测"道具 ID 字段本身合法" → CONFIG.A；测"引用的 ID 存在" → CONFIG.C

#### D_hot_reload.md — 热更新（`RELOAD_4_MODE`）

- **核心能力**：配置增量/全量热更新、在线/离线热更、异常回滚兜底
- **测试点生成要点**：
  - 增量热更只更新变化的行/字段
  - 在线热更不中断游戏进程
  - 离线热更修复后上线
  - 热更异常自动回滚到上一版本
- **典型风险**：热更后客户端与服务端配置不一致导致数据错乱
- **判断标准**：测"配置文件的加载/热更" → CONFIG.D

#### E_parse_load.md — 解析与加载（`PARSE_LOAD`）

- **核心能力**：客户端 & 服务端配置解析、复杂嵌套结构解析、超大表加载性能
- **测试点生成要点**：
  - JSON/XML/Excel 结构解析容错
  - 嵌套数组/对象解析正确性
  - 万行级配置加载耗时（< 10s）
  - 解析失败兜底处理
- **典型风险**：10万行配置加载耗时超过 10s，用户等待超时
- **判断标准**：测"导出工具拒绝非法字段" → CONFIG.A；测"运行时解析性能" → CONFIG.E

#### F_version_compat.md — 版本兼容与灰度（`VERSION_COMPAT`）

- **核心能力**：新旧版本前后兼容、赛季/灰度/渠道专属配置隔离
- **测试点生成要点**：
  - 新客户端读旧配置不崩溃
  - 旧客户端读新配置不崩溃
  - 灰度服配置隔离
  - 渠道专属配置生效
- **典型风险**：新版本客户端上线后旧配置不兼容导致白屏
- **判断标准**：测"版本间配置兼容性" → CONFIG.F

#### G_value_logic.md — 数值逻辑校验（`VALUE_LOGIC`）

- **核心能力**：概率/成长/上限数值逻辑校验
- **测试点生成要点**：
  - 概率总和 ≤ 100%（如随机礼包内所有道具概率之和）
  - 数值公式计算正确性
  - 成长曲线单调性
  - 堆叠上限/产出消耗平衡
- **典型风险**：所有道具概率总和 = 120%，导致抽卡期望溢出
- **判断标准**：测"单个概率字段范围 0-100" → CONFIG.A；测"所有概率字段总和 ≤ 100%" → CONFIG.G

#### H_export_publish.md — 导出/导入/发布（`EXPORT_PUBLISH`）

- **核心能力**：Excel 导出打包发布流程校验
- **测试点生成要点**：
  - Excel 导出工具正确生成配置文件
  - 导入工具兼容旧格式
  - 发布流程审核节点校验
- **典型风险**：导出工具 bug 导致配表缺少行
- **判断标准**：测"导出打包发布流程" → CONFIG.H

#### I_server_specific.md — 服务端专属（`SERVER_CONFIG`）

- **核心能力**：服务端全局常量、定时任务、GM 参数配置校验
- **测试点生成要点**：
  - 全局常量值正确
  - 定时任务 cron 表达式合法
  - GM 参数范围校验
- **典型风险**：GM 后台配置错误导致大规模刷道具
- **判断标准**：测"服务端专属配置" → CONFIG.I

### 2.2 边界子模板

#### J_boundary.md — 边界区分（判定规则）

- **与 UI 的边界**：配置图标路径 → CONFIG；图标在 UI 层渲染 → UI
- **与 BIZ 的边界**：配置数值公式 → CONFIG；数值公式计算后业务逻辑 → BIZ
- **与 LINK 的边界**：配置跨服 ID 格式 → CONFIG；跨服同步时序 → LINK
- **与 LOG 的边界**：配置变更 audit log 记录 → LOG；配置内容本身 → CONFIG
- **典型混淆场景**：
  - "道具表 icon_path 字段"→ 填写格式 → CONFIG；图标加载/渲染 → UI
  - "道具购买概率总和"→ 总和校验 → CONFIG；概率触发后的业务逻辑 → BIZ

## 三、跨模块协作关系

| 模块 | 协作方式 | 典型场景 |
|-----|---------|---------|
| UI | UI 层读取 CONFIG 数值展示 | 道具价格从配置读取后 UI 显示 |
| BIZ | BIZ 层执行业务逻辑读取 CONFIG | 购买道具时读取价格配置扣钱 |
| LINK | LINK 层跨服同步时读取 CONFIG | 跨服排行榜读取排名配置 |
| LOG | LOG 层记录 CONFIG 变更 | 配置热更后记录 audit log |
| SPECIAL | SPECIAL 层防作弊读取 CONFIG | 反作弊读取开箱概率配置验证 |
| UTIL | UTIL 层资源加载依赖 CONFIG | 资源加载路径从配置读取 |
| HINT | HINT 层提示内容来自 CONFIG | 活动提示文案从配置读取 |

## 四、专业判断能力自检

### 4.1 我能判断的场景

| 场景 | 模块归属 | 判断依据 |
|-----|---------|---------|
| 道具表 item_id 录入 "abc"（类型错误） | CONFIG.A | 测导出工具拒绝非法字段类型 |
| 抽卡概率总和 = 120%（超出100%） | CONFIG.G | 测数值逻辑校验，总和约束 |
| 任务奖励引用不存在的道具 ID | CONFIG.C | 测跨表 ID 引用校验 |
| 热更配置后客户端白屏 | CONFIG.F | 测版本兼容性 |
| 10万行配置加载耗时 15s | CONFIG.E | 测解析性能 |
| 道具图标在界面上显示错误 | UI | 渲染层问题，不是配置层 |

### 4.2 常见混淆场景

| 混淆场景 | 正确答案 | 区分要点 |
|---------|---------|---------|
| "道具 icon_path 字段合法性" | CONFIG.A | 测字段填写格式，不测渲染 |
| "概率总和超100%" | CONFIG.G | 测多字段逻辑约束，不测单字段范围 |
| "活动奖励配置缺失道具" | CONFIG.C | 测跨表引用存在性，不测奖励逻辑 |
| "UI 读取配置后显示错误" | UI | UI 渲染问题，不是配置层问题 |
| "配置热更后玩家数据异常" | BIZ + CONFIG | 需要先确认是配置问题还是业务逻辑问题 |

## 五、测试点/测试用例产出能力

- **我能产出的 TP 类型**：POSITIVE（合法配置正常生效）、BOUNDARY（边界值如最大99/最小0）、NEGATIVE（非法配置被拒绝）、EXCEPTION（解析异常兜底）
- **我的 TP 特征字段**：`module: "CONFIG"`, `subclass: "FIELD_LEGALITY"` 等
- **核心 TP 示例**：
  - CONFIG-001: 导出工具拒绝 `item_id = "abc"`（NEGATIVE）
  - CONFIG-002: 导出工具拒绝 `drop_rate = 150`（BOUNDARY）
  - CONFIG-003: 导出工具拒绝不存在的道具 ID 引用（CONFIG.C, NEGATIVE）
  - CONFIG-004: 概率总和超 100% 被拒绝（CONFIG.G, NEGATIVE）
  - CONFIG-005: 热更后配置一致性校验（CONFIG.D, POSITIVE）
- **我在 S6 的角色**：基于 CONFIG 测试点派生测试用例时，关注"配置输入 → 系统行为 → 结果验证"的闭环，特别是非法配置的拦截和热更场景。
