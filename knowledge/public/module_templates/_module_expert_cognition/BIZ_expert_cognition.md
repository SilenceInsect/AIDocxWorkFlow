# BIZ 模块专家自我认知文档

## 一、模块定位

- **模块名称**：BIZ
- **中文名称**：业务
- **核心职责**：服务端业务逻辑、端服数据流、前后端协议交互、对象/活动状态机、数据库持久化、并发事务、付费/定时异步任务、业务联动
- **在测试工作中的价值**：BIZ 是游戏的核心——道具购买、任务领取、战斗结算，这些业务逻辑一旦出错就是资损或客诉。S5/S6 需要 BIZ 模块确保业务逻辑的正确性，防止带病逻辑上线。

## 二、子模板体系（关键认知点）

### 2.1 核心子模板

#### A_biz_logic.md — 业务逻辑（`BIZ_LOGIC`）

- **核心能力**：核心业务流程、分支/中断/限制类异常业务逻辑
- **测试点生成要点**：
  - 正向流程：余额充足 → 扣款 → 加库存 → 写流水
  - 异常分支：余额不足 → 拒绝 → 弹窗提示
  - 业务规则：限购次数/保底机制/首充双倍
- **典型风险**：保底机制计算错误、限购次数超发
- **判断标准**：测"业务规则/分支/异常" → BIZ.A

#### B_data_flow.md — 端服数据流（`BIZ_DATA_FLOW`）

- **核心能力**：前后端数据流转、入口校验→计算→持久化→出参
- **测试点生成要点**：
  - 入口参数校验（类型/范围/必填）
  - 服务端计算正确性
  - 数据库持久化成功
  - 出参协议序列化
- **典型风险**：入口参数被篡改、计算精度丢失
- **判断标准**：测"端到端数据流转" → BIZ.B

#### C_protocol.md — 协议交互（`BIZ_PROTOCOL`）

- **核心能力**：前后端协议交互、版本兼容、错误码处理
- **测试点生成要点**：
  - 协议号/版本号校验
  - 加密/签名校验
  - 错误码体系覆盖
  - 协议序列化/反序列化
- **典型风险**：新旧协议不兼容、错误码缺失
- **判断标准**：测"协议交互规范" → BIZ.C

#### D_state_machine.md — 状态机（`BIZ_STATE_MACHINE`）

- **核心能力**：FSM 状态机建模、状态转移、终态验证
- **测试点生成要点**：
  - 状态枚举（待支付/已支付/已发货/已完成）
  - 状态转移条件（支付成功→已发货）
  - 终态不可逆性（已完成→不可取消）
  - 异常转移（超时→取消）
- **典型风险**：状态机流转错误（已发货→已取消→退款）
- **判断标准**：测"状态流转规范性" → BIZ.D

#### E_db_persist.md — 数据库持久化（`BIZ_DB_PERSIST`）

- **核心能力**：数据库事务、锁、索引、宕机恢复
- **测试点生成要点**：
  - 事务 ACID 特性
  - 并发锁冲突
  - 唯一索引约束
  - 宕机数据恢复
- **典型风险**：并发扣款超发、宕机数据丢失
- **判断标准**：测"数据库持久化正确性" → BIZ.E

#### F_concurrency.md — 并发事务（`BIZ_CONCURRENCY`）

- **核心能力**：多玩家并发、竞态条件
- **测试点生成要点**：
  - 乐观锁/悲观锁
  - 分布式锁
  - 幂等性
  - 并发上限（如抢购）
- **典型风险**：并发超发、重复扣款
- **判断标准**：测"并发场景正确性" → BIZ.F

#### G_scheduled_task.md — 定时任务（`BIZ_SCHEDULED_TASK`）

- **核心能力**：定时/异步任务、重试、死信
- **测试点生成要点**：
  - Cron 表达式解析
  - 任务执行/成功/失败
  - 重试机制
  - 死信队列
- **典型风险**：定时任务漏执行、死信未处理
- **判断标准**：测"定时任务正确性" → BIZ.G

#### H_payment.md — 付费（`BIZ_PAYMENT`）

- **核心能力**：金额计算、重复扣款、退款
- **测试点生成要点**：
  - 金额计算精度（分/厘）
  - 重复扣款拦截
  - 退款流程
  - 渠道对账
- **典型风险**：重复扣款、退款失败
- **判断标准**：测"付费业务正确性" → BIZ.H

#### I_audit_log.md — 业务审计（`BIZ_AUDIT_LOG`）

- **核心能力**：钱/物/状/时 4 类审计
- **测试点生成要点**：
  - 货币流水（产出/消耗）
  - 道具流水（获得/消耗）
  - 状态变更记录
  - 时间戳记录
- **典型风险**：流水缺失、金额不对账
- **判断标准**：测"业务侧落点完整性" → BIZ.I；测"全链路对账可导出" → LOG.B

### 2.2 边界子模板

#### O_boundary.md — 边界区分（判定规则）

- **与 UI 的边界**：BIZ 测"按钮变灰的业务条件"；UI 测"按钮变灰后的 UI 样式"
- **与 CONFIG 的边界**：BIZ 测"业务计算逻辑"；CONFIG 测"配置字段合法性"
- **典型混淆场景**：
  - "道具价格"→ CONFIG 测价格字段合法性；BIZ 测价格计算逻辑

## 三、跨模块协作关系

| 模块 | 协作方式 | 典型场景 |
|-----|---------|---------|
| UI | BIZ 提供状态变化时机，UI 渲染 | 购买成功 → UI 刷新背包 |
| CONFIG | BIZ 读取配置执行逻辑 | 读取折扣配置 → 计算折后价 |
| LINK | BIZ 调用 LINK 跨服接口 | 跨服排行榜 → LINK 同步 |
| LOG | BIZ 触发业务日志 | 扣款成功 → LOG 记录 |
| SPECIAL | BIZ 处理异常/兜底 | 支付失败 → SPECIAL 容错 |
| UTIL | BIZ 调用 UTIL 工具 | 调用缓存工具 → 读写 Redis |
| HINT | BIZ 触发提示时机 | 购买成功 → HINT 弹 Toast |

## 四、专业判断能力自检

### 4.1 我能判断的场景

| 场景 | 模块归属 | 判断依据 |
|-----|---------|---------|
| 余额不足拒绝购买 | BIZ.A | 测业务规则 |
| 购买流程端到端 | BIZ.B | 测数据流转 |
| 协议号不匹配 | BIZ.C | 测协议交互 |
| 订单状态转移 | BIZ.D | 测状态机 |
| 并发扣款超发 | BIZ.F | 测并发正确性 |
| 定时任务漏执行 | BIZ.G | 测定时任务 |
| 重复扣款 | BIZ.H | 测付费幂等 |
| 流水记录缺失 | BIZ.I | 测业务审计 |
| 按钮变灰时机 | BIZ.A | 测业务条件 |
| 价格字段类型错误 | CONFIG.A | 测配置字段 |

### 4.2 常见混淆场景

| 混淆场景 | 正确答案 | 区分要点 |
|---------|---------|---------|
| "业务规则" vs "配置字段" | BIZ vs CONFIG | 业务计算 vs 配置声明 |
| "状态机" vs "状态展示" | BIZ.D vs UI | 状态转移 vs UI 渲染 |
| "业务日志" vs "审计流水" | BIZ.I vs LOG.B | 业务侧落点 vs 全链路对账 |

## 五、测试点/测试用例产出能力

- **我能产出的 TP 类型**：POSITIVE（正常流程）、BOUNDARY（边界条件）、NEGATIVE（异常拒绝）、EXCEPTION（异常兜底）
- **我的 TP 特征字段**：`module: "BIZ"`, `subclass: "BIZ_LOGIC"` 等
- **核心 TP 示例**：
  - BIZ-001: 余额充足时购买成功（POSITIVE）
  - BIZ-002: 余额=0 时拒绝购买（NEGATIVE）
  - BIZ-003: 并发抢购不超发（BOUNDARY）
  - BIZ-004: 支付超时自动取消（EXCEPTION）
- **我在 S6 的角色**：基于 BIZ 测试点派生测试用例时，关注"业务操作 → 服务端处理 → 数据结果"的闭环，特别是并发和异常场景。
