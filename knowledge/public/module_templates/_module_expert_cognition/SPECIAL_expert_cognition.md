# SPECIAL 模块专家自我认知文档

## 一、模块定位

- **模块名称**：SPECIAL
- **中文名称**：特殊情境
- **核心职责**：全系统数值/时间/权限/中断类边界极端场景、非正常业务异常分支处理；客户端本地篡改、伪造协议、挂机脚本等反作弊校验与非法数据拦截；本地存档、传输参数数据安全防护；弱网/抖动/断网环境业务操作容错、高频重复请求限流防刷；移动端前后台切换、进程杀除后状态与资源恢复逻辑；服务宕机、数据库抖动、版本回档、万人并发极限场景事务兜底；背包/邮件/服务器资源耗尽降级策略；新旧客户端版本兼容异常拦截；防沉迷、未成年付费、敏感内容等合规风控；批量清零/大额发奖/全服重置等高风险操作二次校验；灰度/渠道/离线资源损坏环境降级处理
- **在测试工作中的价值**：SPECIAL 是游戏的安全网——反作弊、弱网容错、宕机恢复，这些极端场景一旦处理不好就是严重事故。S5/S6 需要 SPECIAL 模块确保极端场景的正确处理，防止安全漏洞和资损。

## 二、子模板体系（关键认知点）

### 2.1 核心子模板

#### A_boundary_extreme.md — 边界极端（`BOUNDARY_EXTREME`）

- **核心能力**：数值/时间/权限/中断类边界极端场景
- **测试点生成要点**：
  - 数值边界（0/负/溢出/超大）
  - 时间边界（未开启/已过期/CD/赛季结束）
  - 权限边界（未解锁/等级不足）
  - 中断分支（切后台/断线/退出）
- **典型风险**：等级不足却能领取奖励、CD 已过仍显示冷却
- **判断标准**：测"边界极端场景" → SPECIAL.A

#### B_anti_cheat.md — 反作弊（`ANTI_CHEAT`）

- **核心能力**：数据篡改、伪造协议、挂机脚本检测
- **测试点生成要点**：
  - 本地数据校验（改金币/道具/坐标）
  - 发包校验（重复/伪造/越权）
  - 行为检测（瞬移/无限技能/挂机）
  - 处罚联动（封禁/扣回/清除）
- **典型风险**：修改本地数据成功、协议伪造通过
- **判断标准**：测"反作弊检测" → SPECIAL.B

#### C_weak_net_rate_limit.md — 弱网限流（`WEAK_NET_RATE_LIMIT`）

- **核心能力**：弱网容错、限流防刷
- **测试点生成要点**：
  - 弱网/抖动/断网容错
  - 操作缓存队列
  - 高频点击限流
  - 防雪崩保护
- **典型风险**：弱网时重复扣款、刷接口导致雪崩
- **判断标准**：测"弱网容错和限流" → SPECIAL.C

#### D_bg_fg_switch.md — 前后台切换（`BG_FG_SWITCH`）

- **核心能力**：生命周期、资源恢复
- **测试点生成要点**：
  - 切后台短/长时间挂起
  - 进程被杀
  - 锁屏唤醒
  - 切回前台数据同步
- **典型风险**：切后台时奖励被冒领
- **判断标准**：测"前后台切换状态" → SPECIAL.D

#### E_server_ha_risk.md — 宕机高危（`SERVER_HA_RISK`）

- **核心能力**：宕机恢复、并发极限、高危风控
- **测试点生成要点**：
  - 服务故障 Failover
  - 事务回滚/补偿发放
  - 万人并发
  - 高危操作二次确认
- **典型风险**：宕机后数据丢失、高并发超发
- **判断标准**：测"宕机/高危场景" → SPECIAL.E

#### F_version_compat_biz.md — 版本兼容（`VERSION_COMPAT_BIZ`）

- **核心能力**：新旧客户端兼容、协议版本
- **测试点生成要点**：
  - 新客户端读旧配置
  - 旧客户端读新配置
  - 低版本入口拦截
  - 协议号废弃
- **典型风险**：旧版本客户端崩溃、数据格式不兼容
- **判断标准**：测"版本兼容异常" → SPECIAL.F

#### G_channel_gray_biz.md — 渠道灰度（`CHANNEL_GRAY_BIZ`）

- **核心能力**：渠道分包、灰度白名单、离线包
- **测试点生成要点**：
  - 渠道 SDK 升级
  - 白名单进非灰度
  - 离线包损坏
  - 热更资源缺失
- **典型风险**：渠道分包冲突、白名单泄漏
- **判断标准**：测"渠道灰度场景" → SPECIAL.G

#### H_compliance_risk.md — 合规风控（`COMPLIANCE_RISK`）

- **核心能力**：防沉迷、实名、敏感词
- **测试点生成要点**：
  - 防沉迷超时下线
  - 未成年付费限额
  - 敏感词拦截
  - 实名认证缺失限制
- **典型风险**：未成年超额付费、敏感内容展示
- **判断标准**：测"合规风控" → SPECIAL.H

#### I_resource_exhaust.md — 资源耗尽（`RESOURCE_EXHAUST`）

- **核心能力**：背包满、DB 满、CPU 高
- **测试点生成要点**：
  - 背包/仓库/邮件容量上限
  - 服务器资源耗尽降级
  - 排队限流
  - 跨资源检查
- **典型风险**：背包满时提取邮件附件失败
- **判断标准**：测"资源耗尽场景" → SPECIAL.I

### 2.2 边界子模板

#### O_boundary.md — 边界区分（判定规则）

- **与 BIZ 的边界**：SPECIAL 测"异常场景"；BIZ 测"正常流程"
- **与 AUX 的边界**：SPECIAL 测"业务层容错"；AUX 测"底层框架"
- **典型混淆场景**：
  - "网络超时"→ AUX 测底层重连；SPECIAL 测业务容错

## 三、跨模块协作关系

| 模块 | 协作方式 | 典型场景 |
|-----|---------|---------|
| BIZ | SPECIAL 调用 BIZ 容错逻辑 | 弱网购买 → BIZ 幂等 |
| UI | SPECIAL 提供异常 UI 状态 | 权限不足 → UI 显示锁定 |
| LINK | SPECIAL 处理跨服异常 | 跨服超时 → SPECIAL 容错 |
| CONFIG | SPECIAL 读取配置边界 | 读取配置上限 |
| LOG | SPECIAL 记录异常日志 | 反作弊拦截 → LOG |
| AUX | SPECIAL 基于 AUX 底层 | 弱网 → AUX 重连 |

## 四、专业判断能力自检

### 4.1 我能判断的场景

| 场景 | 模块归属 | 判断依据 |
|-----|---------|---------|
| 等级不足拦截领取 | SPECIAL.A | 测边界条件 |
| 本地数据篡改检测 | SPECIAL.B | 测反作弊 |
| 弱网重试不扣款 | SPECIAL.C | 测弱网容错 |
| 切后台数据同步 | SPECIAL.D | 测生命周期 |
| 宕机数据恢复 | SPECIAL.E | 测高危场景 |
| 旧版本客户端拦截 | SPECIAL.F | 测版本兼容 |
| 渠道白名单泄漏 | SPECIAL.G | 测渠道灰度 |
| 未成年付费拦截 | SPECIAL.H | 测合规风控 |
| 背包满拒绝存入 | SPECIAL.I | 测资源耗尽 |
| 网络重连框架 | AUX | 测底层框架 |

### 4.2 常见混淆场景

| 混淆场景 | 正确答案 | 区分要点 |
|---------|---------|---------|
| "网络超时" | SPECIAL.C | 测业务容错，不测网络框架 |
| "弱网购买" | SPECIAL.C + BIZ | 两者都要测 |
| "反作弊算法" | SPECIAL.B | 测检测逻辑，不测加密算法 |
| "加密算法" | AUX.M | 测底层加密，不测业务检测 |

## 五、测试点/测试用例产出能力

- **我能产出的 TP 类型**：POSITIVE（正常边界）、BOUNDARY（极端边界）、NEGATIVE（异常拦截）、EXCEPTION（容错兜底）
- **我的 TP 特征字段**：`module: "SPECIAL"`, `subclass: "ANTI_CHEAT"` 等
- **核心 TP 示例**：
  - SPECIAL-001: 等级不足时领取奖励被拦截（BOUNDARY）
  - SPECIAL-002: 弱网购买重试不重复扣款（NEGATIVE）
  - SPECIAL-003: 本地改金币被检测封禁（BOUNDARY）
  - SPECIAL-004: 宕机后事务回滚（EXCEPTION）
- **我在 S6 的角色**：基于 SPECIAL 测试点派生测试用例时，关注"极端操作 → 系统处理 → 正确恢复"的闭环，特别是安全和容错场景。
