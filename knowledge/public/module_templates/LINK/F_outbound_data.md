# F. 对外数据透出

> **子类代码**：`OUTBOUND_DATA`
> **归属模块**：`LINK`
> **来源**：用户细化定义 §三-5-④「对外数据透出关联」（原 v1.1 完全缺失，v1.2 新增大类）
>
> **测什么**：游戏数据**对外透出**到跨服/客服后台/对账平台/第三方平台的业务规则——排行榜数据、玩家基础数据、付费数据透出、脱敏、字段对齐、实时/定时双模。
> **不测什么**：单服务内部的数据存储（归 BIZ）、外部 API 的鉴权/限流（归 D）、数据脱敏的具体实现（归 UTIL）。
> **与其他子类的差异**：F 关注"**游戏→外部平台**"的数据透出——D 关注"**外部平台→游戏**"的对接、E 关注"**资源/数值**"在多模块间互通、A 关注"**业务系统**"联动。

---

## 1. 典型场景

### 场景 1：排行榜数据透出至跨服
- 业务背景：玩家战力排行榜透出至跨服（跨服玩家可查看其他服的 Top）
- 涉及系统：各逻辑服、聚合服、外部跨服
- 触发动作：玩家战力提升、聚合服拉取数据
- 验证点：跨服正确显示各服的 Top 玩家、数据延迟 < 5s

### 场景 2：玩家基础数据透出至客服后台
- 业务背景：客服在后台查询玩家数据（等级、VIP、充值总额、最后登录时间等）
- 涉及系统：游戏服务端、客服后台
- 触发动作：客服输入玩家 ID 查询
- 验证点：返回数据完整、字段对齐、脱敏生效

### 场景 3：付费数据透出对账平台
- 业务背景：每日付费数据透出至对账平台（与渠道对账）
- 涉及系统：游戏服务端、对账平台
- 触发动作：每日定时推送付费数据
- 验证点：数据完整、字段对齐、推送时间稳定

### 场景 4：对外透出数据脱敏
- 业务背景：对外数据脱敏（手机号中间 4 位 *、身份证号只显示省份、姓名只显示姓）
- 涉及系统：游戏服务端、脱敏逻辑
- 触发动作：对外查询
- 验证点：敏感字段脱敏、非敏感字段正常显示

### 场景 5：字段对齐规范
- 业务背景：对外透出字段需对齐外部平台规范（如字段名、数值类型、枚举值）
- 涉及系统：游戏服务端、外部平台
- 触发动作：对外推送数据
- 验证点：字段名/类型/枚举值与外部平台规范一致

### 场景 6：实时同步
- 业务背景：玩家战力变化实时推送到跨服排行榜
- 涉及系统：游戏服务端、跨服排行榜
- 触发动作：玩家战力变化
- 验证点：延迟 < 1s、跨服立即可见

### 场景 7：定时同步
- 业务背景：付费数据每日 0 点定时推送到对账平台
- 涉及系统：游戏服务端、对账平台
- 触发动作：每日 0 点定时任务
- 验证点：定时任务按时执行、推送数据完整

### 场景 8：数据格式转换
- 业务背景：游戏内数据格式（ProtoBuf/JSON）转换为外部平台格式（CSV/Excel）
- 涉及系统：游戏服务端、外部平台
- 触发动作：对外推送数据
- 验证点：数据转换正确、字段不丢失

### 场景 9：透出失败重试
- 业务背景：对外推送失败（网络问题、外部平台宕机）
- 涉及系统：游戏服务端、外部平台
- 触发动作：推送失败
- 验证点：自动重试（最多 N 次）、最终失败告警

### 场景 10：透出数据权限控制
- 业务背景：不同外部账号有不同数据权限（运营只能看汇总、客服能看明细、BI 能看全量）
- 涉及系统：游戏服务端、外部平台
- 触发动作：外部账号查询
- 验证点：权限校验正确、低权限账号看不到高权限数据

### 场景 11：跨服排行榜字段脱敏
- 业务背景：跨服排行榜只显示玩家昵称 + 战力，不显示手机号/邮箱
- 涉及系统：游戏服务端、跨服排行榜
- 触发动作：跨服查询
- 验证点：跨服只看到脱敏后的字段

### 场景 12：数据透出实时性 vs 一致性平衡
- 业务背景：实时排行榜 vs 每日 0 点定时的全量排行榜
- 涉及系统：游戏服务端、外部平台
- 触发动作：玩家战力变化
- 验证点：实时排行榜快速但可能略有延迟、定时排行榜慢但绝对一致

### 场景 13：数据透出带宽控制
- 业务背景：对外透出数据量过大时需带宽控制（限流/分批）
- 涉及系统：游戏服务端、外部平台
- 触发动作：大量数据推送
- 验证点：限流生效、不打爆带宽

## 2. 种子测试点（TP 模板）

### TP-001（OUTBOUND_DATA）：跨服排行榜数据透出
- **scenario**：场景 1
- **module**：`OUTBOUND_DATA`
- **precondition**：3 个逻辑服各有 Top 100 战力玩家
- **test_data**：玩家 A 战力提升到跨服第 1
- **expected**：跨服排行榜 1s 内显示 A、脱敏后只显示昵称 + 战力
- **notes**：注意"跨服"≠ "多端"——跨服是 B（跨服务），多端是 C

### TP-002（OUTBOUND_DATA）：客服后台查询玩家数据
- **scenario**：场景 2
- **module**：`OUTBOUND_DATA`
- **precondition**：玩家 X 等级 30、VIP 5、充值 1000 元
- **test_data**：客服输入 X 的 ID 查询
- **expected**：返回数据完整（等级/VIP/充值/最后登录）、敏感字段脱敏（手机号中间 4 位 *）
- **notes**：注意"客服后台"是 D（外部平台）业务，但"游戏数据如何透出"是 F

### TP-003（OUTBOUND_DATA）：付费数据定时推送对账平台
- **scenario**：场景 3
- **module**：`OUTBOUND_DATA`
- **precondition**：每日 0 点定时推送付费数据
- **test_data**：当日总付费 100 万元、1000 笔
- **expected**：0 点准时推送、数据完整（每笔订单号/金额/玩家/渠道）、字段对齐规范
- **notes**：注意"推送时机"——每日/每月？是否补推（漏推时）

### TP-004（OUTBOUND_DATA）：敏感字段脱敏
- **scenario**：场景 4
- **module**：`OUTBOUND_DATA`
- **precondition**：玩家手机号 13800001234、身份证 110101199001011234、姓名"张三"
- **test_data**：对外查询
- **expected**：手机号"138****1234"、身份证"110101********1234"、姓名"张*"
- **notes**：注意"脱敏规则"——不同字段不同规则（手机号/身份证/姓名/邮箱）

### TP-005（OUTBOUND_DATA）：字段对齐外部规范
- **scenario**：场景 5
- **module**：`OUTBOUND_DATA`
- **precondition**：外部平台要求 `player_id`（int）、`nickname`（string）、`vip_level`（int 0-20）
- **test_data**：对外推送 1 个玩家
- **expected**：字段名/类型/范围完全对齐（player_id 是 int 不是 string、vip_level 0-20 不超限）
- **notes**：注意"字段对齐"——游戏内可能用 string ID，外部要求 int ID

### TP-006（OUTBOUND_DATA）：实时同步延迟
- **scenario**：场景 6
- **module**：`OUTBOUND_DATA`
- **precondition**：玩家战力 10000
- **test_data**：玩家穿戴装备战力提升到 11000
- **expected**：跨服排行榜 1s 内显示 11000
- **notes**：注意"实时延迟"——网络抖动时可接受 1-3s

### TP-007（OUTBOUND_DATA）：定时同步准时执行
- **scenario**：场景 7
- **module**：`OUTBOUND_DATA`
- **precondition**：每日 0 点定时推送付费数据
- **test_data**：当日新增 1000 笔付费订单
- **expected**：0 点准时推送完整数据、推送时间稳定（误差 < 5 分钟）
- **notes**：注意"定时任务稳定性"——服务重启后是否补推

### TP-008（OUTBOUND_DATA）：数据格式转换正确
- **scenario**：场景 8
- **module**：`OUTBOUND_DATA`
- **precondition**：游戏内数据格式 ProtoBuf
- **test_data**：对外推送 1000 个玩家
- **expected**：转换为外部平台要求的 CSV/Excel 格式、字段不丢失、类型正确
- **notes**：注意"格式转换"——可能涉及 ProtoBuf→JSON→CSV 多级转换

### TP-009（OUTBOUND_DATA）：透出失败重试
- **scenario**：场景 9
- **module**：`OUTBOUND_DATA`
- **precondition**：对外推送数据时网络中断
- **test_data**：推送失败 3 次
- **expected**：自动重试 3 次、最终失败告警、不丢失数据
- **notes**：注意"重试 vs 幂等"——同一数据多次推送需幂等

### TP-010（OUTBOUND_DATA）：数据透出权限控制
- **scenario**：场景 10
- **module**：`OUTBOUND_DATA`
- **precondition**：运营账号（汇总权限）、客服账号（明细权限）、BI 账号（全量权限）
- **test_data**：3 个账号分别查询
- **expected**：运营只能看汇总、客服能看明细、BI 能看全量、低权限看不到高权限
- **notes**：注意"权限"——是 F 的设计要求，具体鉴权实现是 D

### TP-011（OUTBOUND_DATA）：跨服排行榜脱敏
- **scenario**：场景 11
- **module**：`OUTBOUND_DATA`
- **precondition**：玩家 A 手机号、邮箱、身份证完整
- **test_data**：跨服查询 A 的数据
- **expected**：跨服只看到脱敏后数据（昵称 + 战力）、不显示手机/邮箱/身份证
- **notes**：注意"脱敏粒度"——按字段/按数据等级

### TP-012（OUTBOUND_DATA）：实时 vs 定时一致性平衡
- **scenario**：场景 12
- **module**：`OUTBOUND_DATA`
- **precondition**：实时排行榜（快速但略有延迟）、每日 0 点定时排行榜（慢但绝对一致）
- **test_data**：玩家战力变化
- **expected**：实时排行榜 1s 内更新、定时排行榜每日 0 点更新、两者最终一致
- **notes**：注意"实时 vs 一致性"——实时用最终一致性，定时用强一致性

### TP-013（OUTBOUND_DATA）：数据透出带宽控制
- **scenario**：场景 13
- **module**：`OUTBOUND_DATA`
- **precondition**：100 万玩家数据需对外推送
- **test_data**：一次性推送 vs 分批推送
- **expected**：分批推送（如 1 万/批）、限流生效、不打爆带宽、最终完成推送
- **notes**：注意"带宽"——大量数据推送需分批/限流

### TP-014（OUTBOUND_DATA）：数据透出字段缺失告警
- **scenario**：场景 5 变体
- **module**：`OUTBOUND_DATA`
- **precondition**：游戏内新增字段 X、但外部平台未要求
- **test_data**：对外推送
- **expected**：可选字段 X 可推送或不推送、必填字段缺失时告警、不发送不完整数据
- **notes**：注意"必填 vs 选填"——对接双方需对齐字段表

### TP-015（OUTBOUND_DATA）：数据透出完整性校验
- **scenario**：场景 8 变体
- **module**：`OUTBOUND_DATA`
- **precondition**：1000 笔订单需推送
- **test_data**：推送后对账
- **expected**：推送数 = 服务端订单数、无丢失、无重复、字段值一致
- **notes**：注意"完整性"——推送后需校验（可与对账系统对账）

## 3. 边界陷阱

### 边界 1：vs D 外部第三方
- **混淆点**："客服后台"听起来像 D（外部平台），但"数据如何透出"是 F
- **判定规则**：测"外部平台对接规则（鉴权/限流/回调）" → D；测"游戏数据对外透出业务规则" → F
- **实例**：客服后台 API 鉴权 → D；玩家数据如何脱敏/字段对齐后推送给客服 → F

### 边界 2：vs BIZ 业务
- **混淆点**："排行榜"听起来像 BIZ，但"跨服排行榜"是 F
- **判定规则**：测"单系统内部业务" → BIZ；测"跨平台数据透出" → F
- **实例**：本服排行榜计算逻辑 → BIZ；本服排行榜透出至跨服 → F

### 边界 3：vs UTIL 底层
- **混淆点**："数据脱敏"听起来像 F，但具体脱敏实现是 UTIL
- **判定规则**：测"业务透出规则" → F；测"底层能力（脱敏算法/数据库/缓存）" → UTIL
- **实例**：敏感字段脱敏的业务规则（哪些字段脱敏/如何脱敏）→ F；脱敏的具体算法（手机号中间 4 位 *）→ UTIL

### 边界 4：vs LOG 日志
- **混淆点**："数据上报"听起来像 F，但"日志埋点"是 LOG
- **判定规则**：测"业务数据对外透出" → F；测"日志/埋点规范" → LOG
- **实例**：付费数据推送给对账平台 → F；付费操作的日志埋点规范 → LOG

## 4. 验证证据

### 视觉证据
- 跨服排行榜截图（脱敏后）
- 客服后台查询截图（脱敏生效）
- 对账平台接收数据截图

### 日志证据
- `<LOG_OUTBOUND_DATA_PUSH>`：对外数据推送，target_platform、record_count、field_count
- `<LOG_OUTBOUND_DATA_RETRY>`：推送重试
- `<LOG_DATA_DESENSITIZE>`：脱敏生效
- `<LOG_FIELD_ALIGN_CHECK>`：字段对齐校验

### 数据证据
- 数据库 `outbound_push_log` 表：对外推送记录
- 数据库 `data_desensitize_rule` 表：脱敏规则
- 数据库 `field_align_spec` 表：字段对齐规范
- 数据库 `outbound_permission` 表：对外透出权限

### 性能证据
- 实时同步延迟 < 1s
- 定时同步时间误差 < 5 分钟
- 大批量数据推送带宽控制（10MB/s）
- 字段对齐校验耗时 < 100ms（1000 字段）
