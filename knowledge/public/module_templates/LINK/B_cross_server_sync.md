# B. 跨服务/分布式关联

> **子类代码**：`CROSS_SERVER_SYNC`
> **归属模块**：`LINK`
> **来源**：用户细化定义 §三-2「跨服务/分布式关联」（原"跨服务协议、数据同步"扩展）
>
> **测什么**：游戏内**多个分布式服务**（逻辑服/活动服/充值服/聊天服/网关服/跨服）之间的数据同步、时序校验、断点续同步、灰度/多服环境隔离。
> **不测什么**：单服务内部的业务流程（归 BIZ）、底层网络收发包实现（归 UTIL）、多端客户端之间的同步（归 C）。
> **与其他子类的差异**：B 关注"**服务之间**"的数据同步——A 关注"**业务系统之间**"的联动规则、C 关注"**多端之间**"的状态对齐、D 关注"**外部平台之间**"的对接。

---

## 1. 典型场景

### 场景 1：跨服组队副本
- 业务背景：玩家 A 在"逻辑服 1"、玩家 B 在"逻辑服 2"，跨服组队进入"修罗场"副本
- 涉及服务：逻辑服 1、逻辑服 2、跨服副本服
- 触发动作：双方同时进入跨服副本
- 验证点：双方实时看到对方、操作响应延迟 < 200ms、副本结算数据正确回写双方逻辑服

### 场景 2：跨服拍卖行
- 业务背景：跨服拍卖行允许多个逻辑服的玩家竞拍同一件道具
- 涉及服务：多个逻辑服、跨服拍卖服、跨服撮合服
- 触发动作：A 服玩家出价竞拍 B 服玩家上架的道具
- 验证点：出价实时同步至所有服的玩家、最终成交时卖家和买家所在服都能正确发放/扣除

### 场景 3：跨服竞技场
- 业务背景：跨服竞技场匹配全服玩家对战
- 涉及服务：竞技匹配服、对战服、积分服、多个逻辑服
- 触发动作：玩家匹配 + 进入对战
- 验证点：跨服积分实时同步、胜负结果回写逻辑服、跨赛季积分归档

### 场景 4：全服排行榜数据双向同步
- 业务背景：全服排行榜数据来源于多个逻辑服
- 涉及服务：多个逻辑服、排行榜聚合服
- 触发动作：玩家 A 在逻辑服 1 战力提升、玩家 B 在逻辑服 2 战力提升
- 验证点：排行榜聚合服实时拉取各服 Top 数据、双向同步延迟 < 1s、跨服排序正确

### 场景 5：跨服指令透传
- 业务背景：网关服接收客户端指令后透传到指定逻辑服
- 涉及服务：客户端、网关服、逻辑服
- 触发动作：客户端发起"购买道具"指令
- 验证点：网关正确识别目标逻辑服、指令透传无丢失、响应回传路径正确

### 场景 6：跨服数据读写一致性
- 业务背景：玩家在逻辑服 A 充值，跨服读取账户信息
- 涉及服务：充值服、账户中心服、逻辑服
- 触发动作：充值完成 + 跨服查询余额
- 验证点：跨服立即看到更新后的余额、不出现"旧数据"、读写一致性延迟 < 500ms

### 场景 7：跨服操作回滚
- 业务背景：跨服撮合交易时某一步骤失败
- 涉及服务：撮合服、双方逻辑服、支付服
- 触发动作：A 服扣款 + B 服发货，中途撮合服宕机
- 验证点：A 服扣款回滚、B 服未发货、双方数据状态最终一致

### 场景 8：多服并发同玩家数据
- 业务背景：玩家在逻辑服 A 完成"日常任务"，同时在逻辑服 B 完成"竞技场"
- 涉及服务：逻辑服 A、逻辑服 B、任务聚合服
- 触发动作：两服几乎同时上报玩家进度
- 验证点：时序正确（按上报顺序处理）、不重复发放奖励、不丢失进度

### 场景 9：服务重启后跨服断点续同步
- 业务背景：聚合服重启期间有玩家产生了新数据
- 涉及服务：聚合服、多个逻辑服
- 触发动作：聚合服重启恢复
- 验证点：聚合服启动后自动从各逻辑服拉取重启期间未同步的数据、断点续同步不重复不丢失

### 场景 10：测试服/正式服/灰度服数据隔离
- 业务背景：3 个环境完全隔离，不能跨环境数据互通
- 涉及服务：测试服、正式服、灰度服
- 触发动作：在测试服充值 100 元
- **expected**：测试服 +100、正式服不变、灰度服不变、跨环境数据不互通
- **notes**：注意"数据隔离"是 B（跨服）子项，但"灰度配置同步"也是 B 子项

### 场景 11：白名单灰度跨服配置同步
- 业务背景：白名单玩家在灰度服体验新功能，灰度配置需同步到该服
- 涉及服务：配置中心、灰度服、正式服
- 触发动作：运营更新灰度配置
- 验证点：白名单玩家进入灰度服立即看到新功能、非白名单玩家仍在正式服、配置按白名单维度生效

### 场景 12：跨服时序错乱
- 业务背景：玩家跨服操作产生的事件时序错乱
- 涉及服务：事件总线服、目标服
- 触发动作：事件 A（10:00:01 上报）后到达事件 B（10:00:00 上报）
- 验证点：服务端按"业务时序"而非"到达时序"处理、时序错乱时按业务规则修正

## 2. 种子测试点（TP 模板）

### TP-001（CROSS_SERVER_SYNC）：跨服组队实时同步
- **scenario**：场景 1
- **module**：`CROSS_SERVER_SYNC`
- **precondition**：玩家 A（逻辑服 1）+ 玩家 B（逻辑服 2），双方组队进入"修罗场"
- **test_data**：A 移动到坐标 (10, 10)
- **expected**：B 端在 200ms 内看到 A 移动到 (10, 10)、双方操作响应同步
- **notes**：注意"实时同步延迟"——跨服网络抖动时可接受 200-500ms，但需测试峰值

### TP-002（CROSS_SERVER_SYNC）：跨服拍卖出价同步
- **scenario**：场景 2
- **module**：`CROSS_SERVER_SYNC`
- **precondition**：B 服玩家上架道具"屠龙刀"、A 服玩家在线
- **test_data**：A 服玩家出价 1000 金币
- **expected**：A 服出价后 B 服卖家立即看到出价信息、跨服撮合服正确记录出价、最终成交时双方服都更新
- **notes**：注意"跨服撮合"——撮合服宕机时数据不能丢

### TP-003（CROSS_SERVER_SYNC）：跨服竞技积分同步
- **scenario**：场景 3
- **module**：`CROSS_SERVER_SYNC`
- **precondition**：玩家 A（逻辑服 1，积分 1500）匹配到玩家 B（逻辑服 2，积分 1480）
- **test_data**：A 战胜 B
- **expected**：双方积分变化（+15 / -15）实时同步到各自逻辑服、跨赛季积分正确归档
- **notes**：注意"跨赛季归档"——避免新赛季清掉旧赛季数据

### TP-004（CROSS_SERVER_SYNC）：全服排行榜双向同步
- **scenario**：场景 4
- **module**：`CROSS_SERVER_SYNC`
- **precondition**：排行榜聚合服从 3 个逻辑服拉数据
- **test_data**：A 服玩家战力 +1000、B 服玩家战力 +2000
- **expected**：聚合服在 1s 内拉取最新数据、排行榜排序正确、双向同步不出现旧数据
- **notes**：注意"双向"——不只是聚合服拉数据，下发到各服也要正确

### TP-005（CROSS_SERVER_SYNC）：跨服指令透传
- **scenario**：场景 5
- **module**：`CROSS_SERVER_SYNC`
- **precondition**：客户端连接到网关服
- **test_data**：客户端发起"购买道具 ID=100"指令
- **expected**：网关服正确识别目标逻辑服、指令透传成功、响应回传路径完整
- **notes**：注意"指令无丢失"——网络抖动时需重试机制（归 B，不是 UTIL）

### TP-006（CROSS_SERVER_SYNC）：跨服读写一致性
- **scenario**：场景 6
- **module**：`CROSS_SERVER_SYNC`
- **precondition**：玩家在逻辑服 A 充值服完成充值
- **test_data**：充值后立即在跨服场景查询余额
- **expected**：500ms 内跨服立即看到新余额、不出现"读旧数据"、读写一致性延迟 < 500ms
- **notes**：注意"读写一致性"是 LINK B，但具体缓存实现是 UTIL

### TP-007（CROSS_SERVER_SYNC）：跨服交易回滚
- **scenario**：场景 7
- **module**：`CROSS_SERVER_SYNC`
- **precondition**：A 服扣款 100 金 + 撮合服通知 B 服发货
- **test_data**：撮合服在 A 扣款后、B 发货前宕机
- **expected**：A 服扣款回滚、B 服未发货、双方数据最终一致、无"钱扣了但货没发"
- **notes**：注意"回滚 vs 重试"——需评估是回滚还是重试，跨服操作推荐"幂等 + 重试"

### TP-008（CROSS_SERVER_SYNC）：多服并发同玩家数据
- **scenario**：场景 8
- **module**：`CROSS_SERVER_SYNC`
- **precondition**：玩家在逻辑服 A 完成日常任务、逻辑服 B 完成竞技场
- **test_data**：两服几乎同时上报玩家进度
- **expected**：服务端按业务时序处理（任务先、竞技场后）、不重复发放奖励、不丢失进度
- **notes**：注意"业务时序"——需要在事件上加业务时间戳

### TP-009（CROSS_SERVER_SYNC）：服务重启断点续同步
- **scenario**：场景 9
- **module**：`CROSS_SERVER_SYNC`
- **precondition**：聚合服宕机 5 分钟，期间 100 个玩家有新数据
- **test_data**：聚合服重启
- **expected**：聚合服启动后自动从各逻辑服拉取宕机期间未同步的数据、不重复不丢失
- **notes**：注意"断点"——以"上次成功同步时间戳"为断点

### TP-010（CROSS_SERVER_SYNC）：测试服/正式服/灰度服数据隔离
- **scenario**：场景 10
- **module**：`CROSS_SERVER_SYNC`
- **precondition**：3 个环境完全隔离
- **test_data**：在测试服充值 100 元
- **expected**：测试服 +100、正式服 +0、灰度服 +0、跨环境数据 0 互通
- **notes**：注意"完全隔离"——即使是同一玩家账号也不能跨环境同步

### TP-011（CROSS_SERVER_SYNC）：白名单灰度配置同步
- **scenario**：场景 11
- **module**：`CROSS_SERVER_SYNC`
- **precondition**：白名单玩家 A 在灰度服、非白名单玩家 B 在正式服
- **test_data**：运营更新灰度配置（新功能开关=1）
- **expected**：玩家 A 立即看到新功能、玩家 B 仍看不到、按白名单维度生效
- **notes**：注意"白名单生效"——不是"环境生效"

### TP-012（CROSS_SERVER_SYNC）：跨服时序错乱处理
- **scenario**：场景 12
- **module**：`CROSS_SERVER_SYNC`
- **precondition**：事件 A（业务时间 10:00:01）后到达事件 B（业务时间 10:00:00）
- **test_data**：乱序事件流
- **expected**：服务端按业务时序（非到达时序）处理、错乱时按业务规则修正
- **notes**：注意"业务时序优先"——网络抖动导致到达时序≠业务时序时优先业务

### TP-013（CROSS_SERVER_SYNC）：跨服操作超时重试
- **scenario**：场景 7 扩展
- **module**：`CROSS_SERVER_SYNC`
- **precondition**：跨服 RPC 调用超时
- **test_data**：调用超时（> 3s）
- **expected**：自动重试（最多 3 次）、幂等处理、最终成功或回滚
- **notes**：注意"幂等"——是 B 的设计要求，否则重试会重复扣款

### TP-014（CROSS_SERVER_SYNC）：全服通用配置与单服独有配置联动
- **scenario**：场景 10/11
- **module**：`CROSS_SERVER_SYNC`
- **precondition**：全服通用配置（基础功能）+ 单服独有配置（特色活动）
- **test_data**：运营同时更新两类配置
- **expected**：全服配置对所有服生效、单服配置只对目标服生效、不冲突
- **notes**：注意"配置优先级"——单服配置可覆盖全服通用配置

### TP-015（CROSS_SERVER_SYNC）：跨服数据双向同步延迟
- **scenario**：场景 4 变体
- **module**：`CROSS_SERVER_SYNC`
- **precondition**：100 个逻辑服，聚合服每 1s 拉取一次
- **test_data**：单服 1 万玩家同时更新战力
- **expected**：聚合服在 5s 内完成全部拉取、排序正确、无旧数据混入
- **notes**：注意"性能压测"——大服场景下需测峰值

## 3. 边界陷阱

### 边界 1：vs A 内部业务关联
- **混淆点**："跨服组队副本"听起来像 B（跨服），但"组队内部的玩法"是 A
- **判定规则**：测"跨服务/分布式数据同步" → B；测"业务系统之间的联动规则" → A
- **实例**：跨服组队的数据同步 → B；组队副本内部的玩法规则 → A

### 边界 2：vs C 多端一致性
- **混淆点**："全服排行榜"听起来像 B（跨服），但"排行榜在多端显示"是 C
- **判定规则**：测"服务端跨服数据" → B；测"多端客户端同步显示" → C
- **实例**：聚合服拉取各服战力 → B；玩家在 PC 和移动端看到一致的战力 → C

### 边界 3：vs UTIL 底层
- **混淆点**："跨服网络"听起来像 B，但具体 TCP/HTTP 实现是 UTIL
- **判定规则**：测"业务互通规则" → B；测"底层能力（网络/缓存）" → UTIL
- **实例**：跨服指令透传的业务规则 → B；TCP 长连接封装 → UTIL

### 边界 4：vs SPECIAL 异常
- **混淆点**："跨服数据错乱"是异常，但"跨服时序校验"是 B
- **判定规则**：测"正常跨服同步规则" → B；测"异常场景下的安全拦截" → SPECIAL
- **实例**：跨服时序按业务时间排序 → B；利用跨服时序漏洞刷奖励 → SPECIAL

## 4. 验证证据

### 视觉证据
- 跨服组队副本中双方玩家位置同步截图
- 跨服拍卖行跨服玩家出价实时显示截图
- 跨服排行榜跨服排序正确截图

### 日志证据
- `<LOG_CROSS_SERVER_RPC>`：跨服 RPC 调用，耗时、状态
- `<LOG_SYNC_PULL>` / `<LOG_SYNC_PUSH>`：跨服拉取/推送
- `<LOG_SYNC_BREAKPOINT>`：断点续同步时间戳
- `<LOG_ENV_ISOLATION>`：跨环境数据隔离校验
- `<LOG_TIMELINE_REORDER>`：事件时序修正

### 数据证据
- 数据库 `cross_server_player_data` 表：跨服玩家数据同步状态
- 数据库 `auction_cross_server` 表：跨服拍卖订单
- 数据库 `leaderboard_aggregate` 表：聚合服排行榜
- 数据库 `env_isolation_check` 表：环境隔离校验记录

### 性能证据
- 跨服 RPC 延迟 < 200ms（同区）、< 500ms（跨区）
- 跨服读写一致性延迟 < 500ms
- 聚合服拉取 100 服数据 < 5s
- 服务重启后断点续同步 100 玩家 < 10s
