# O. LINK 模块边界区分（判定规则）

> **非测试类型**——本文件是 S5 生成测试点时的**判定参考**，
> 避免误标 LINK 标签（实际是 **UTIL 底层 / BIZ 单系统 / SPECIAL 异常 / LOG 日志 / UI 控件**）。
>
> **来源**：用户最终版 §四「区分边界」+ §6.1「UTIL vs LINK 核心区分」（v1.8）
>
> **核心规则**（用户强调）：**UTIL 是"水管"，LINK 是"水管里流通什么业务数据、两端业务怎么对齐"**。二者**无重叠**。

---

## 1. 判定原则

> **LINK = 多系统/多服务/多端/外部第三方之间的业务互通规则 + 数据同步一致性校验**
>
> **不归 LINK**：
> - 底层能力（网络/缓存/SDK 封装/数据库/资源加载）→ **UTIL**
> - 单系统独立业务流程 → **BIZ**
> - 异常场景下的安全拦截（反作弊/限流/防刷）→ **SPECIAL**
> - 日志/埋点/审计追踪规范 → **LOG**
> - UI 控件渲染/布局/动效 → **UI**
> - 提示/红点/飘字 → **HINT**
> - 配置表字段/范围/版本 → **CONFIG**

### 判定 3 步法

1. **是"底层能力/工具/SDK"吗？** 是 → 归 **UTIL**
2. **是"单系统独立业务"吗？** 是 → 归 **BIZ**（或其他模块按业务归类）
3. **都不属于 + 涉及多系统/多服务/多端/外部互通？** 是 → 归 **LINK**（按 A-F 子类选择）

> 任何重叠项归 **UTIL（基础设施）/ LINK（业务规则）**，**无第三选项**。

## 2. 边界对照表（vs 其他 7 模块）

### 2.1 vs UTIL（辅助）— **核心边界**（用户最强调）

| 场景 | 归 LINK | 归 UTIL |
|------|---------|--------|
| 支付 | 微信支付下单 + 回调解析 + 补发 + 退款回滚业务规则 | 微信 SDK 加密、TCP 重连、HTTP 请求工具 |
| 网络 | 跨服 RPC 业务规则、跨服时序校验 | TCP 长连接封装、消息队列底层、HTTP 工具 |
| 缓存 | 跨服数据同步、客户端本地缓存差值刷新业务 | 本地缓存的 LRU 淘汰、Redis 缓存命中率 |
| 资源 | 跨系统通用资源/限购共享池业务规则 | 资源下载、断点续传、引用计数、卸载 |
| 第三方 | 第三方登录态与游戏内角色绑定、外部 API 业务对接 | 第三方 SDK 底层封装（SDK 加密/解密/初始化）|
| 配置 | 配置跨环境隔离、白名单灰度同步 | 配置表加载/解析/导出（归 CONFIG）|
| 安全 | 跨系统数据联动约束、一致性校验 | 加密算法、底层异常兜底（崩溃捕获）|

**判定口诀**：
- **UTIL** = 能力底座（"水管"）
- **LINK** = 业务规则（"水管里流通什么业务数据、两端业务怎么对齐"）
- **同一功能**：测 SDK 怎么用 → UTIL；测业务怎么用 SDK 实现互通 → LINK

### 2.2 vs BIZ（业务）

| 场景 | 归 LINK | 归 BIZ |
|------|---------|--------|
| 充值 | 微信支付回调业务处理、补发、游戏内兜底 | 商城扣款/发货的内部业务逻辑 |
| 任务 | 任务完成解锁商城（多系统联动规则）| 任务内部流程（接取/进度/完成/奖励）|
| 商城 | 跨系统限购共享池业务 | 商城内部购买流程 |
| 抽卡 | 跨活动积分共享 | 抽卡概率计算、保底 |
| 战斗 | 跨服组队数据同步 | 战斗内部伤害计算、暴击 |
| 活动 | 活动复开/重置业务规则、多活动并行相交 | 单活动的开启/进行/结束流程 |
| 邮件 | 邮件跨端同步（见 C）、邮件附件资源发放业务 | 邮件内部状态机、附件发放 |
| 资源 | 跨系统通用资源/限购互通 | 资源内部产出/消耗逻辑 |

**判定口诀**：
- **BIZ** = 单系统独立业务（"1 个系统内的事"）
- **LINK A** = 业务系统间联动规则（"多个系统间的事"）

### 2.3 vs SPECIAL（特殊情境）

| 场景 | 归 LINK | 归 SPECIAL |
|------|---------|------------|
| 支付 | 外部接口超时游戏内兜底、回调幂等 | 第三方接口恶意篡改、刷单拦截 |
| 跨服 | 跨服时序校验、断点续同步 | 跨服数据作弊拦截、利用漏洞刷奖励 |
| 活动 | 活动重置数据隔离 | 活动重置时刷奖励漏洞拦截 |
| 接口 | 外部 API 限流场景下的兜底 | 外部接口限流防刷、风控检测 |
| 灰度 | 灰度/多服环境数据隔离 | 灰度流量调度、刷邀请码 |
| 第三方 | 第三方回调正常业务处理 | 第三方接口异常安全拦截 |

**判定口诀**：
- **LINK** = 正常互通规则（"按规则办事"）
- **SPECIAL** = 异常安全拦截（"按异常办事 + 防坏人"）

### 2.4 vs C 多端一致性（LINK 内部子类）

| 场景 | 归 LINK A | 归 LINK C |
|------|-----------|-----------|
| 任务 | 任务完成解锁商城（多系统联动）| 任务跨端同步显示（A 端完成 B 端可见）|
| 充值 | 充值触发累充发奖（多系统联动）| 充值记录跨端同步 |
| 装备 | 换装备战力刷新（多系统联动）| 战力跨端显示 |
| 活动 | 活动状态多系统同步 | 活动状态跨端同步 |
| 邮件 | 邮件发奖（系统联动）| 邮件未读/已读跨端同步 |

**判定口诀**：
- **A** = 业务系统间（"server-side 多系统"）
- **B** = 服务间（"server-side 分布式"）
- **C** = 端到端（"client-side 多端"）
- **D** = 外部平台（"server ↔ 外部"）
- **E** = 资源/数值（"资源/数据"）
- **F** = 对外数据（"server → 外部"）

### 2.5 vs LOG（日志）

| 场景 | 归 LINK | 归 LOG |
|------|---------|--------|
| 支付 | 支付回调业务处理 | 支付操作的日志埋点规范 |
| 资源 | 跨系统资源数值互通 | 资源变动的日志埋点规范 |
| 上报 | 第三方数据上报业务规则 | 日志/埋点采集规范 |
| 同步 | 跨服/跨端同步业务规则 | 同步事件的日志埋点规范 |
| 透出 | 对外数据透出业务规则 | 透出操作的日志埋点规范 |

**判定口诀**：
- **LINK** = 业务规则本身（"做什么"）
- **LOG** = 业务日志规范（"记录什么字段、什么时机"）

### 2.6 vs UI（界面）

| 场景 | 归 LINK | 归 UI |
|------|---------|-------|
| 多端 | 跨端数据同步 | 多端 UI 控件适配（不同分辨率）|
| 排行榜 | 跨服排行榜数据透出 | 排行榜 UI 控件渲染 |
| 提示 | 跨系统解锁的入口出现 | 解锁动效、UI 弹窗 |
| 邮件 | 邮件跨端同步 | 邮件 UI 渲染 |
| 充值 | 充值业务规则 | 充值弹窗 UI |

**判定口诀**：
- **LINK** = 业务数据/规则
- **UI** = 控件渲染/布局/动效

### 2.7 vs HINT（提示）

| 场景 | 归 LINK | 归 HINT |
|------|---------|---------|
| 跨系统 | 跨系统联动业务规则 | 跨系统触发的飘字/红点 |
| 充值 | 充值业务规则 | 充值成功的飘字 |
| 错误 | 业务错误（如充值失败补偿）| 错误提示文案 |
| 邮件 | 邮件发奖业务 | 邮件未读红点 |
| 活动 | 活动重置业务 | 活动入口红点 |

**判定口诀**：
- **LINK** = 业务动作/规则
- **HINT** = 玩家可见的反馈（红点/飘字/Toast）

### 2.8 vs CONFIG（配置）

| 场景 | 归 LINK | 归 CONFIG |
|------|---------|-----------|
| 配置 | 配置跨环境隔离业务规则、白名单灰度同步 | 配置表字段合法性、范围、跨表依赖 |
| 资源 | 资源跨系统互通业务 | 资源 ID 字段、类型、范围 |
| 概率 | 跨活动共享积分业务 | 概率字段范围、公式 |

**判定口诀**：
- **LINK** = 配置的业务规则（"配置怎么用"）
- **CONFIG** = 配置本身（"配置怎么定义/校验"）

## 3. 常见误判案例

### 误判 1：微信支付 → 错标 UTIL
- **错误**：把"微信支付"全部归 UTIL（认为"SDK 都是 UTIL"）
- **正确**：
  - **UTIL** 测："微信 SDK 怎么调用"（SDK 初始化、加密、HTTP 请求）→ B.网络层 / M.加密安全
  - **LINK D** 测："微信支付怎么对接业务"（下单参数、回调处理、补发/退款、对账）→ D.外部第三方
  - **同一 Story 应有 2 个 TP**：1 个 UTIL（SDK 联调）+ 1 个 LINK D（业务对接）

### 误判 2：跨服组队 → 错标 BIZ
- **错误**：把"跨服组队"归 BIZ（认为"组队是业务"）
- **正确**：
  - **BIZ** 测："组队副本内部的玩法规则"（队伍组建、匹配、队长权限、战斗结算）→ BIZ A.核心业务逻辑
  - **LINK B** 测："跨服数据同步"（跨服 RPC、时序校验、断点续同步）→ B.跨服务/分布式
  - **同一 Story 应有 2 个 TP**：1 个 BIZ（玩法）+ 1 个 LINK B（跨服同步）

### 误判 3：充值跨端同步 → 错标 C 还是 D
- **错误**：把"充值跨端同步"全部归 C 或全部归 D
- **正确**：
  - **LINK D** 测："第三方支付业务对接"（微信/支付宝回调、订单状态、补发/退款）→ D.外部第三方
  - **LINK C** 测："充值记录跨端同步"（A 端充值后 B 端看到）→ C.多端一致性
  - **同一 Story 应有 2 个 TP**：1 个 D（支付对接）+ 1 个 C（跨端同步）

### 误判 4：排行榜 → 错标 F 还是 B
- **错误**：把"排行榜"全部归 B（跨服）或全部归 F（透出）
- **正确**：
  - **BIZ** 测："本服排行榜计算逻辑"（战力/等级/积分排序）→ BIZ A.核心业务逻辑
  - **LINK B** 测："跨服排行榜聚合"（聚合服拉取各服数据）→ B.跨服务/分布式
  - **LINK F** 测："排行榜透出至跨服/外部"（跨服玩家查看、脱敏、字段对齐）→ F.对外数据透出
  - **同一 Story 应有 3 个 TP**：BIZ（计算）+ B（聚合）+ F（透出）

### 误判 5：多活动共享积分 → 错标 E 还是 A
- **错误**：把"多活动共享积分"全部归 E（资源互通）或全部归 A（业务联动）
- **正确**：
  - **LINK A** 测："多活动并行相交业务规则"（共享/互斥、限购、进度重置）→ A.内部业务关联
  - **LINK E** 测："跨活动共享积分池数值互通"（总积分上限、数值计算、资源发放）→ E.跨模块资源互通
  - **同一 Story 应有 2 个 TP**：1 个 A（业务规则）+ 1 个 E（数值互通）

### 误判 6：本地缓存差值刷新 → 错标 UTIL 还是 C
- **错误**：把"本地缓存"全部归 UTIL（认为"缓存都是底层"）
- **正确**：
  - **UTIL C** 测："本地缓存实现"（LRU 淘汰、内存占用、读写性能）→ C.缓存层
  - **LINK C** 测："本地缓存与服务端差值刷新业务规则"（什么时机刷新、字段粒度、兜底校正）→ C.多端一致性
  - **同一 Story 应有 2 个 TP**：1 个 UTIL（缓存实现）+ 1 个 LINK C（差值刷新业务）

### 误判 7：灰度/多服环境数据隔离 → 错标 CONFIG 还是 B
- **错误**：把"环境隔离"全部归 CONFIG（认为"环境是配置"）
- **正确**：
  - **CONFIG** 测："环境配置字段"（测试服/正式服/灰度服的配置定义）→ CONFIG F.版本兼容
  - **LINK B** 测："环境数据隔离业务规则"（跨环境数据 0 互通、白名单灰度同步）→ B.跨服务/分布式
  - **同一 Story 应有 2 个 TP**：1 个 CONFIG（环境配置）+ 1 个 LINK B（隔离业务）

## 4. 判定流程图

```
Story 涉及某个功能点
    │
    ├── 是 "底层能力/工具/SDK/网络/缓存"？
    │     是 → 归 UTIL（按 B/C/D/E/F/J/K 子类选）
    │
    ├── 是 "单系统独立业务/玩法/流程"？
    │     是 → 归 BIZ（按 A/B/C/... 子类选）
    │
    ├── 是 "异常场景下的安全拦截"？
    │     是 → 归 SPECIAL（反作弊/限流/防刷）
    │
    ├── 是 "日志/埋点/审计规范"？
    │     是 → 归 LOG
    │
    ├── 是 "UI 控件/布局/动效"？
    │     是 → 归 UI
    │
    ├── 是 "提示/红点/飘字/Toast"？
    │     是 → 归 HINT
    │
    ├── 是 "配置表字段/范围/跨表依赖"？
    │     是 → 归 CONFIG
    │
    └── 涉及多系统/多服务/多端/外部互通？
          是 → 归 LINK（按 A/B/C/D/E/F 子类选）：
                ├─ 业务系统间联动 → A
                ├─ 服务间数据同步 → B
                ├─ 端到端同步 → C
                ├─ 外部平台对接 → D
                ├─ 资源/数值互通 → E
                └─ 对外数据透出 → F
```

## 6. 总结：LINK 子类判定口诀

| 子类 | 判定口诀 |
|------|---------|
| **A 内部业务关联** | "多个业务系统间联动"（任务→商城、活动→邮件）|
| **B 跨服务/分布式** | "多个服务端同步"（跨服 RPC、聚合服拉取、灰度隔离）|
| **C 多端一致性** | "多个客户端同步"（PC↔移动端、本地缓存差值刷新）|
| **D 外部第三方** | "游戏↔外部平台对接"（SDK 登录/支付/上报、API 回调、消息队列）|
| **E 跨模块资源互通** | "资源/数值在多模块间"（通用货币/道具/限购共享池）|
| **F 对外数据透出** | "游戏→外部平台数据"（排行榜→跨服、玩家→客服、付费→对账）|

> **UTIL vs LINK 终极口诀**：
> - **UTIL = "水管"**——底层通用框架/工具/SDK 封装/传输基础设施
> - **LINK = "水管里流通什么业务数据 + 两端业务怎么对齐"**——业务互通规则
> - **二者无重叠**
