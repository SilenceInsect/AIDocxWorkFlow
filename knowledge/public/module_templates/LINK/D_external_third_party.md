# D. 外部第三方关联

> **子类代码**：`EXTERNAL_THIRD_PARTY`
> **归属模块**：`LINK`
> **来源**：用户细化定义 §三-4「外部第三方关联」（原"外部 API、第三方支付/登录集成"扩展 + 业务子项：异步消息队列透传）
>
> **测什么**：游戏与**外部第三方平台**（渠道 SDK / 平台开放 API / 业务消息队列）之间的业务对接规则——登录态绑定、支付下单/回调/补发/退款/对账、数据上报、外部 API 调用与兜底、消息队列上下游透传。
> **不测什么**：第三方 SDK 底层加密/封装（归 UTIL）、单笔充值业务规则（归 BIZ）、异常安全拦截（归 SPECIAL）。
> **与其他子类的差异**：D 关注"**外部平台之间**"的对接——A 关注"**业务系统之间**"的联动、B 关注"**服务之间**"的同步、C 关注"**多端之间**"的对齐。

> **UTIL 隔离（核心）**：本子类的"业务对接规则"是 LINK；"SDK 底层封装（微信 SDK、支付宝 SDK、HTTP 工具）"是 UTIL——本子模板不写底层封装内容。

## 1. 典型场景

### 场景 1：游客转正式账号
- 业务背景：玩家以"游客"身份登录后绑定为正式账号
- 涉及外部：游客 SDK、正式账号 SDK、平台账号体系
- 触发动作：游客登录 → 引导绑定
- 验证点：游客数据迁移到正式账号、不丢失进度、绑定后游客入口消失

### 场景 2：账号绑定/解绑
- 业务背景：玩家绑定手机号、邮箱、微信、QQ 等
- 涉及外部：第三方登录 SDK（微信/QQ）、平台账号体系
- 触发动作：玩家绑定/解绑
- 验证点：绑定流程完整、解绑后数据保留、可重新绑定

### 场景 3：渠道账号互通
- 业务背景：玩家用 A 渠道账号登录后想用 B 渠道账号登录
- 涉及外部：A 渠道 SDK、B 渠道 SDK
- 触发动作：B 渠道登录
- 验证点：提示"该手机号已绑定其他渠道，请先解绑"、互通规则生效

### 场景 4：登录状态同步游戏内角色
- 业务背景：玩家通过微信登录后游戏内创建/选择角色
- 涉及外部：微信登录 SDK、游戏内角色系统
- 触发动作：微信登录成功
- 验证点：登录态与游戏内角色正确关联、token 校验通过、过期正确处理

### 场景 5：第三方支付下单
- 业务背景：玩家点击"充值 30 元"
- 涉及外部：支付 SDK（微信/支付宝/银联/苹果内购）
- 触发动作：选择支付方式
- 验证点：下单参数正确（商品 ID/金额/订单号）、签名校验通过、跳转支付成功

### 场景 6：渠道回调解析
- 业务背景：玩家支付完成后第三方渠道回调通知游戏服务端
- 涉及外部：支付渠道回调
- 触发动作：玩家完成支付
- 验证点：回调签名校验、订单号匹配、金额核对、回调幂等（防重复通知）

### 场景 7：订单状态同步
- 业务背景：玩家支付完成后游戏内订单状态从"待支付"→"已支付"
- 涉及外部：支付渠道、订单系统
- 触发动作：支付成功回调
- 验证点：订单状态正确流转、游戏内发货、跨端可见（见 C 多端一致性）

### 场景 8：补发资源回滚
- 业务背景：玩家已发货但退款时需要回滚
- 涉及外部：支付渠道、游戏内资源系统
- 触发动作：玩家申请退款
- 验证点：补发资源正确回滚、玩家余额/道具同步、跨系统联动约束生效

### 场景 9：渠道对账
- 业务背景：每日/每月与渠道对账（我方订单 vs 渠道订单）
- 涉及外部：支付渠道、对账系统
- 触发动作：定时对账
- 验证点：金额一致、订单号匹配、差异订单可追溯、对账不平生成工单

### 场景 10：第三方上报
- 业务背景：向渠道上报玩家在线时长、付费、角色数据
- 涉及外部：渠道数据上报接口
- 触发动作：玩家登录、付费、升级
- 验证点：上报字段对齐渠道规范、上报时机正确、不重复上报

### 场景 11：运营后台开放 API
- 业务背景：运营后台通过 API 拉取游戏数据
- 涉及外部：运营后台 API
- 触发动作：运营查询玩家数据
- 验证点：API 鉴权正确、参数校验、返回数据完整、限流生效

### 场景 12：客服工单接口
- 业务背景：客服系统通过 API 提交工单
- 涉及外部：客服工单 API
- 触发动作：客服创建工单
- 验证点：工单数据正确传递、附件上传成功、状态回调生效

### 场景 13：防沉迷实名校验
- 业务背景：玩家实名认证
- 涉及外部：防沉迷实名校验接口
- 触发动作：玩家提交身份信息
- 验证点：接口调用成功、身份证号校验、未成年人游戏时长限制

### 场景 14：外部接口超时
- 业务背景：支付回调接口超时
- 涉及外部：支付渠道
- 触发动作：回调请求 5s 未响应
- 验证点：服务端重试（最多 N 次）、超时日志、超时后游戏内兜底（玩家可在订单列表查）

### 场景 15：外部接口重试幂等
- 业务背景：支付回调失败重试
- 涉及外部：支付渠道
- 触发动作：服务端重试回调
- 验证点：幂等处理（同一订单号只发一次货）、重试不导致重复扣款

### 场景 16：业务消息队列上下游透传
- 业务背景：充值完成推送活动服（"充值 X 元 + 累充 Y"）
- 涉及外部：消息队列（Kafka/RabbitMQ）
- 触发动作：充值成功
- 验证点：消息发布到队列、活动服消费消息、消息顺序正确、不重复消费

### 场景 17：消息丢失/重复幂等
- 业务背景：消息队列传输中丢失/重复
- 涉及外部：消息队列
- 触发动作：网络抖动/服务端宕机
- 验证点：丢失消息可重发、重复消息幂等、最终一致性

### 场景 18：异步消息驱动业务状态对齐
- 业务背景：副本结算推送排行榜服 + 邮件服
- 涉及外部：消息队列、排行榜服、邮件服
- 触发动作：副本结算
- 验证点：排行榜服更新、邮件服发奖、两服最终状态对齐

## 3. 边界陷阱

### 边界 1：vs UTIL 底层
- **混淆点**："微信支付"听起来像 D，但"微信 SDK 底层封装"是 UTIL
- **判定规则**：测"业务对接规则（下单/回调/补发/对账）" → D；测"SDK 底层封装（加密/HTTP/签名）" → UTIL
- **实例**：微信支付回调业务处理 → D；微信 SDK 的加密算法 → UTIL

### 边界 2：vs BIZ 业务
- **混淆点**："充值"听起来像 BIZ，但"充值跨外部平台"是 D
- **判定规则**：测"单系统独立业务流程" → BIZ；测"跨外部平台的对接" → D
- **实例**：商城购买流程（扣款/发货）→ BIZ；微信支付回调处理 → D

### 边界 3：vs SPECIAL 异常
- **混淆点**："支付回调超时"是异常，但"回调超时游戏内兜底"是 D
- **判定规则**：测"正常外部对接规则" → D；测"异常场景下的安全拦截" → SPECIAL
- **实例**：支付回调超时后玩家可在订单列表查 → D；利用支付漏洞刷单 → SPECIAL

### 边界 4：vs LOG 日志
- **混淆点**："上报渠道"听起来像 D，但"上报字段的日志埋点规范"是 LOG
- **判定规则**：测"业务对接规则" → D；测"日志/埋点规范" → LOG
- **实例**：支付回调时上报渠道数据 → D；上报字段的埋点规范（什么字段、什么时机）→ LOG

- `<LOG_THIRD_PARTY_LOGIN>`：第三方登录，渠道、user_id、token
- `<LOG_THIRD_PARTY_PAY>`：支付下单/回调，order_id、amount、status
- `<LOG_THIRD_PARTY_CALLBACK_RETRY>`：回调重试
- `<LOG_THIRD_PARTY_RECONCILE>`：渠道对账
- `<LOG_THIRD_PARTY_REPORT>`：数据上报
- `<LOG_MSG_QUEUE_PUBLISH>` / `<LOG_MSG_QUEUE_CONSUME>`：消息队列发布/消费

### 数据证据
- 数据库 `third_party_user` 表：第三方账号绑定
- 数据库 `payment_order` 表：支付订单
- 数据库 `reconcile_diff` 表：对账差异
- 数据库 `msg_queue_log` 表：消息队列消费记录
- 数据库 `callback_idempotent` 表：回调幂等记录

### 性能证据
- 第三方登录响应 < 2s
- 支付回调处理 < 1s
- 渠道对账 10000 单 < 30s
- 消息队列消费延迟 < 500ms
- 退款回滚 < 3s
