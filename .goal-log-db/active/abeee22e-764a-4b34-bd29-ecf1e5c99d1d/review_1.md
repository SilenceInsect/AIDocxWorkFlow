## 2. 根因定位

### 根因 1（机制）：底座设计是"骨架 + 个性化"分离

- **机制**：`governance/design_iter/plans/v36/base_template.md` §2 明确"8 节通用骨架 + N 节专家特色节"
- **好处**：避免模板污染，避免伪专家
- **风险**：如果 Round 2/3 验证时发现 8 节也不够用，需重新设计

### 根因 2（规范）：ui-expert 的 14 节不可全盘复制

- **机制**：14 节中"🌱 对话前提" / "🎤 实际场景示例" / "🔗 协作清单" / "📝 不喜欢的事" / "🎯 提问方式" / "🏗️ 项目级规则管理" / "🚨 反假设污染" / "🛠️ 主动识别" 都是真实对话沉淀，不可作为通用骨架
- **做法**：只把 8 节"必备骨架"提炼，剩余的留给每个专家自己长出来

### 根因 3（习惯）：commit 前缀的命名一致性

- **现状**：`[UI-专家直写]` / `[BIZ-专家直写]` / `[AUX-专家直写]` / `[HINT-专家直写]`
- **隐患**：流程专家（aidocx-s5-...）目前没有"专家 commit"传统——Round 3 启动时要立约定
