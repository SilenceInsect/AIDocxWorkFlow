# AIDocxWorkFlow

AI 驱动的端到端需求全流程工作流，从原始需求输入到测试用例审核，全链路资产沉淀，支持自我迭代。

> **适用场景**：游戏 QA 团队的游戏功能测试用例自动生成。
> **前置要求**：Cursor IDE（含 Composer 模型）。
> **网络要求**：仅 AI 产出阶段（S1-S7）需要网络；数据管理、生成报告、版本管理全部离线可用。

## 快速开始

### 1. 打开项目

在 Cursor 中打开 `AIDocxWorkFlow` 项目目录。

### 2. 输入需求

在 Cursor 对话中描述需求（粘贴需求文本或描述功能），AI 将引导完成后续流程。

### 3. 等待产出

7 个阶段依次执行，每个阶段完成后自动进入下一阶段。最终产出：
- `backlog.md` — Epic/Story 拆解
- `test_points.json` — 测试点
- `test_cases.md` / `test_cases.json` / `test_cases.xlsx` — 测试用例
- `review_report.md` — 用例审查报告

### .docx 文件自动解析

提到 `.docx` 文件路径时，Cursor 自动触发钩子，将其转换为 Markdown，每个文档在 `process_assets/<文档名>/` 下独立打包（包含 source/、parsed/、requirements/、test_cases/、feedback/ 子目录）。

## 流程架构

```
需求输入
    ↓
┌─────────────────────────────────────────────┐
│ S1 需求评审 — AI 评分、缺失检测、冲突检测   │
└─────────────────────────────────────────────┘
    ↓ [L1 自审通过]
┌─────────────────────────────────────────────┐
│ S2 需求拆解 — Epic → Story → Task           │
└─────────────────────────────────────────────┘
    ↓ [L1 自审通过]
┌─────────────────────────────────────────────┐
│ S3 迭代规划 — 7步执行规划                   │
└─────────────────────────────────────────────┘
    ↓ [L1 自审通过]
┌─────────────────────────────────────────────┐
│ S4 测试点生成 — 7 模块覆盖、结构化 JSON      │
└─────────────────────────────────────────────┘
    ↓ [L1 自审通过]
┌─────────────────────────────────────────────┐
│ S5 测试用例生成 + 评审                      │
│     — Markdown + JSON + Excel                │
│     — 结构验证 + 覆盖率计算                  │
└─────────────────────────────────────────────┘
    ↓ [人工验收]
┌─────────────────────────────────────────────┐
│ S6 全流程报告 — 进度 + 质量 + 效率报告      │
└─────────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────────┐
│ S7 质量门禁 — 产出门禁检查 + 迭代复盘       │
└─────────────────────────────────────────────┘
    ↓
最终交付物 → 人工终验 → 归档
```

**关键设计**：
- 每阶段产出自动持久化，后续阶段可复用
- 流程中断时输出暂停报告，人工决策后继续

## 质量保障体系

### L1 自审（AI 自动校验）

每个阶段产出后，AI 自动进行 4 层校验：

| 门禁 | 内容 |
|---|---|
| 结构门禁 | JSON 是否可解析、必填字段是否存在 |
| 质量门禁 | 评分是否达到阈值 |
| 完整性门禁 | 7 模块是否全部覆盖 |
| 边界门禁 | 正/负/边界用例是否均衡 |

**重试策略**：初始最大 5 次，连续成功 10 次后自动收敛至 2 次。收敛参数可在 `config.py` 中调整。

### L2 用例审查（S5 阶段执行）

基于 7 模块标准对测试点和用例进行评分：

| 模块 | 权重 | 必检项 |
|---|---|---|
| SPECIAL | 2.0 | 发包安全（重复/改参/高频）、弱网、中断 |
| BIZ | 1.5 | 核心业务逻辑、实体缓存、数据库持久化 |
| CONFIG | 1.0 | 四种配置重载方式、字段关联校验、沙箱隔离 |
| UI | 1.0 | 控件覆盖、跨分辨率/平台适配 |
| HINT | 1.0 | 红点、道具/货币飘字、系统消息 |
| LOG | 1.0 | 资产变更日志、进度触发日志、异常日志 |
| LINK | 0.8 | 关联模块逻辑、回归测试范围 |

**审查结果**：`PASS`（可交付）/ `FAIL`（需修正）/ `REJECT`（拒绝）

### 流程中断机制

以下情况触发流程中断，输出暂停报告供人工决策：
- L1 自审达到最大重试次数仍不通过
- L2 审查发现 BLOCKER 级问题

人工决策选项：`CONTINUE`（修复后继续）/ `SKIP`（跳过）/ `RETRY`（重试）/ `TERMINATE`（终止）

## 七模块测试框架

所有测试点必须覆盖以下 7 个模块：

| 模块 | 前缀 | 核心内容 |
|---|---|---|
| CONFIG | 配置 | 配置表重载（4种方式）、字段关联校验、前后端一致性 |
| UI | 界面 | 控件覆盖、布局、分辨率/画质/平台适配 |
| BIZ | 业务 | 业务逻辑、协议交互、实体缓存/入库 |
| HINT | 奖励提示 | 红点、道具飘字、货币飘字、系统消息、活动推送 |
| LINK | 关联&回归 | 需求外模块关联逻辑测试 + 回归验证 |
| SPECIAL | 特殊情境 | 反作弊（重复/改参/高频发包）、弱网、中断、跨服、边界值 |
| LOG | 日志 | 关键业务资产变更和进度触发的日志记录（格式不强制，关键信息齐全） |

详见 `workflow_assets/example_test_points/README.md`。

## 内网离线能力说明

### Python 自动化模块

以下 Python 模块可在内网（无需网络）运行：

```
ai_workflow/
  requirement_reviewer_auto.py   # S1 自动评分（规则引擎）
  test_case_formatter.py        # S4/S5 骨架生成 + ID分配 + 格式化
  auto_reviewer.py              # S5 自动审查（结构验证 + 覆盖率）
  iteration_aggregator.py        # S7 聚合（反馈日志 + 审核统计）
  feedback_logger.py             # 反馈日志读写
  excel/                        # Excel 生成（xlsxwriter）
```

### 需要网络的阶段

仅 S2（需求拆解）、S3（迭代规划）需要 AI 模型参与，其余阶段均可离线自动执行。

## 团队角色

| 角色 | 职责 |
|---|---|
| 需求提出者 | 输入原始需求，确认评审结论 |
| AI 评审员 | 执行自动评审（S1） |
| 产品经理 | 审核拆解结果，确认 Story 粒度（S2） |
| 开发负责人 | 审核原型与流程图的技术可行性（S3） |
| 测试负责人 | 审核测试点，7 模块覆盖率把控（S4） |
| 测试工程师 | 执行用例审核，确认用例可执行性（S5） |
| 测试经理 | 最终审批用例，版本发布（S5 人工验收） |
